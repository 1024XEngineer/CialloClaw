# 08-扩展接口规范

> 本文档定义 CialloClaw 的扩展接入规范。目标是让新功能可以在既有架构上，通过实现接口、注册组件、声明事件与状态读写规则的方式接入，而不是通过修改主干逻辑、跨层直连或堆积临时代码。

---

## 1. 文档目标

本文档解决以下问题：

1. 新功能应该以什么形式接入系统。
2. 哪些能力允许作为扩展实现，哪些必须放在核心层。
3. 扩展需要实现哪些接口。
4. 扩展如何注册到运行时。
5. 扩展如何声明权限、风险、配置、事件与状态依赖。
6. 扩展如何被启用、禁用、升级和卸载。

本文档与以下文档配套使用：

- `02-事件总线与内部接口协议.md`
- `04-事件字典.md`
- `05-模块职责与依赖边界.md`
- `06-Repository与持久化设计.md`
- `07-启动与运行时装配.md`

---

## 2. 扩展设计原则

### 2.1 接口优先

扩展接入必须优先通过接口和注册机制完成，不允许通过修改核心模块内部逻辑来接入。

### 2.2 事件驱动优先

扩展之间、扩展与核心模块之间，优先通过事件交互。只有在文档明确允许的边界内，才允许直接接口调用。

### 2.3 能力最小化

每个扩展只负责一种明确能力：

- 一个感知源
- 一个工具
- 一个 Agent 角色
- 一个中间件
- 一个状态投影器
- 一个 UI 组件

避免一个扩展同时承担“感知 + 规划 + 执行 + 持久化”。

### 2.4 核心对象不被绕过

扩展不能绕过以下核心对象：

- Session
- Task
- Loop
- Approval
- Memory
- Log

扩展可以读取或更新这些对象，但必须通过规定的 Repository、Service 或事件流程完成。

### 2.5 可禁用、可观测、可恢复

任何扩展都必须满足：

- 可以单独启用/禁用
- 可以记录日志和审计信息
- 可以在重启后恢复到可解释状态
- 失败不会破坏核心运行链路

### 2.6 配置与代码解耦

扩展是否启用、优先级、参数、权限、风险等级应由配置决定，而不应写死在代码中。

## 3. 扩展分类

CialloClaw 中的扩展分为以下几类。

### 3.1 感知扩展（Perception Extension）

用于采集外部输入并发布事件，例如：

- 用户输入监听
- 剪贴板监听
- 选中文本监听
- 屏幕 OCR
- 文件系统监听
- 系统状态监听

产出通常是 perception 类事件。

### 3.2 认知扩展（Cognition Extension）

用于理解、推理、规划、路由，例如：

- 意图识别器
- 上下文增强器
- 任务分解器
- 收敛判定器
- Agent 路由器

产出通常是 cognition 类事件或 Task/Loop 状态更新。

### 3.3 执行扩展（Action Extension）

用于真正执行操作，例如：

- 浏览器工具
- 终端工具
- 文件系统工具
- 系统操作工具
- 外部命令适配器

产出通常是 action 类事件、Approval 请求、执行结果与 Artifact。

### 3.4 状态扩展（State Extension）

用于维护状态投影、索引、记忆沉淀、日志导出，例如：

- Episodic Memory 摘要器
- Semantic Memory 提炼器
- Task 读模型投影器
- 搜索索引更新器

### 3.5 UI 扩展（UI Extension）

用于向用户展示状态或收集输入，例如：

- 新增 TUI 视图
- 调试面板
- 批量审批页
- 事件查看器

### 3.6 中间件扩展（Middleware Extension）

用于拦截、变换、审计、限流、去重、追踪事件，例如：

- Trace 注入
- 去重
- 审批前置校验
- 指标采集
- 事件持久化

### 3.7 恢复扩展（Recovery Extension）

用于系统启动时恢复某类状态，例如：

- 恢复 Pending Approval
- 恢复运行中 Loop
- 恢复未完成 Task
- 重建某些索引

## 4. 核心扩展模型

推荐所有扩展都实现一个最小元数据接口。

```go
package extension

type ExtensionKind string

const (
    ExtensionKindPerception ExtensionKind = "perception"
    ExtensionKindCognition  ExtensionKind = "cognition"
    ExtensionKindAction     ExtensionKind = "action"
    ExtensionKindState      ExtensionKind = "state"
    ExtensionKindUI         ExtensionKind = "ui"
    ExtensionKindMiddleware ExtensionKind = "middleware"
    ExtensionKindRecovery   ExtensionKind = "recovery"
)

type ExtensionMetadata struct {
    ID          string
    Name        string
    Version     string
    Kind        ExtensionKind
    Description string
    Author      string

    EnabledByDefault bool
    Experimental     bool

    RequiredCapabilities []string
    RequiredConfigKeys   []string
    EmitsEvents          []string
    ConsumesEvents       []string
    DependsOn            []string
    ConflictsWith        []string
}

type Extension interface {
    Metadata() ExtensionMetadata
}
```

### 4.1 元数据必须回答的问题

每个扩展都应能清楚回答：

- 它是谁
- 它属于哪一类
- 它依赖什么
- 它消费什么事件
- 它产出什么事件
- 它是否默认启用
- 它是否仍处于实验状态

---

## 5. 扩展运行时上下文

推荐为扩展统一提供运行时上下文，而不是让扩展直接到处找依赖。

```go
package extension

type RuntimeContext struct {
    Config        ConfigProvider
    EventBus      EventBus
    Logger        Logger
    Clock         Clock
    IDGenerator   IDGenerator

    Sessions      SessionRepository
    Tasks         TaskRepository
    Loops         LoopRepository
    Approvals     ApprovalRepository
    Memory        MemoryRepository
    Logs          LogRepository

    Tools         ToolRegistry
    Agents        AgentRegistry
    Services      ServiceLocator
}
```

### 5.1 原则

- 扩展只通过 `RuntimeContext` 获取所需依赖。
- 扩展不得自己 new 一个新的 EventBus、Repository、Logger。
- 扩展不得绕开 Repository 直接改底层数据库。

---

## 6. 生命周期接口

建议定义统一生命周期接口。

```go
package extension

type Initializable interface {
    Init(ctx RuntimeContext) error
}

type Startable interface {
    Start(ctx context.Context) error
}

type Stoppable interface {
    Stop(ctx context.Context) error
}

type HealthCheckable interface {
    HealthCheck(ctx context.Context) error
}
```

### 6.1 生命周期约束

- `Init`：只做依赖校验、配置解析、内部准备，不做耗时监听。
- `Start`：启动监听、订阅、后台任务。
- `Stop`：释放资源、停止 goroutine、刷写缓冲。
- `HealthCheck`：返回扩展当前是否健康。

## 7. 分类接口规范

---

### 7.1 感知扩展接口

```go
package perception

type Source interface {
    extension.Extension
    extension.Initializable
    extension.Startable
    extension.Stoppable

    SourceName() string
}
```

#### 感知扩展要求

1. 只能发布 perception 或少量 system 事件。
2. 不直接创建 Task，除非通过明确的入口服务。
3. 不能直接调用工具执行操作。
4. 需要具备去重、节流和错误隔离能力。

#### 推荐事件流

`perception.xxx` -> `context.updated` -> `intent.recognized` -> `task.created`

---

### 7.2 中间件扩展接口

```go
package eventbus

type Middleware interface {
    Name() string
    Handle(ctx context.Context, event Event, next func(context.Context, Event) error) error
}
```

#### 中间件要求

1. 不应长时间阻塞主链路。
2. 不应私自吞掉关键事件，除非文档明确允许。
3. 若会修改事件，应保持可追踪性。
4. 应明确说明自己是只读、中断型还是变换型中间件。

---

### 7.3 认知扩展接口

```go
package cognition

type Processor interface {
    extension.Extension
    eventbus.Subscriber
}
```

#### 认知扩展要求

1. 主要消费 perception、state、action result 事件。
2. 主要产出意图、计划、任务、路由、loop 相关事件。
3. 不直接执行系统副作用操作。
4. 若需发起真实操作，应通过 Task/Step + Action 事件链进入执行层。

---

### 7.4 工具扩展接口

```go
package tool

type RiskLevel string

const (
    RiskNone     RiskLevel = "none"
    RiskLow      RiskLevel = "low"
    RiskMedium   RiskLevel = "medium"
    RiskHigh     RiskLevel = "high"
    RiskCritical RiskLevel = "critical"
)

type Metadata struct {
    Name            string
    Description     string
    Version         string
    Risk            RiskLevel
    Idempotent      bool
    RequiresApproval bool
    Timeout         time.Duration
    Tags            []string
}

type Tool interface {
    Metadata() Metadata
    Schema() JSONSchema
    Execute(ctx context.Context, call ToolCall) (ToolResult, error)
}
```

#### 工具扩展要求

1. 必须声明风险等级。
2. 必须声明是否需要审批。
3. 必须声明超时时间。
4. 必须定义输入输出 schema。
5. 必须将副作用和结果写入日志。
6. 高风险工具不能绕过 Approval。

---

### 7.5 Agent 扩展接口

```go
package agent

type Capability string

type Agent interface {
    extension.Extension

    Name() string
    Capabilities() []Capability
    CanHandle(task Task) bool
    Execute(ctx context.Context, task Task) (AgentResult, error)
}
```

#### Agent 扩展要求

1. Agent 的输入单位是 Task 或 TaskStep，不是任意 Event。
2. Agent 可以消费事件，但不直接订阅全部系统事件。
3. Agent 不直接修改底层 Repository，优先通过服务或返回结构由调度器落库。
4. Agent 若需要调用工具，应通过 ToolRegistry 和审批策略完成。

---

### 7.6 UI 扩展接口

```go
package ui

type View interface {
    extension.Extension
    Render(state ViewModel) error
}

type InteractionHandler interface {
    extension.Extension
    HandleUserAction(action UIAction) error
}
```

#### UI 扩展要求

1. UI 只能展示状态或收集用户输入。
2. UI 不直接做业务决策。
3. UI 收到用户动作后应发布事件或调用明确的应用服务。

---

### 7.7 恢复扩展接口

```go
package recovery

type Recoverable interface {
    extension.Extension
    Recover(ctx context.Context) error
    RecoveryPriority() int
}
```

#### 恢复扩展要求

1. 必须幂等。
2. 可以重复执行，不得造成重复副作用。
3. 必须有明确的恢复边界和停止条件。

---

## 8. 注册规范

### 8.1 注册器

系统应提供统一注册器。

```go
package registry

type ExtensionRegistry interface {
    Register(ext extension.Extension) error
    Get(id string) (extension.Extension, bool)
    List() []extension.Extension
    ListByKind(kind extension.ExtensionKind) []extension.Extension
}
```

### 8.2 注册时机

建议顺序如下：

1. 加载配置
2. 初始化 Repository
3. 初始化 EventBus
4. 注册 Tool
5. 注册 Agent
6. 注册 Middleware
7. 注册 State Processor / Projection
8. 注册 UI
9. 注册 Perception Source
10. 执行 Recovery
11. 启动运行时组件

### 8.3 注册校验

注册时必须校验：

- ID 是否冲突
- 依赖扩展是否存在
- 冲突扩展是否同时启用
- 配置项是否齐全
- 所消费/生产的事件是否在事件字典中存在

---

## 9. 配置规范

推荐为扩展统一配置命名空间。

```yaml
extensions:
  clipboard_listener:
    enabled: true
    dedup_window_ms: 1000
    max_payload_bytes: 65536

  browser_tool:
    enabled: true
    timeout_seconds: 20
    requires_approval_for_external_nav: true

  writer_agent:
    enabled: true
    model: "gpt-4.1"
    temperature: 0.3
```

### 9.1 配置约束

- 每个扩展必须有独立配置段。
- 扩展不得从多个不一致来源读配置。
- 可热更新的配置项必须明确标注。
- 风险相关配置变更必须记录审计日志。

---

## 10. 事件接入规范

扩展若消费或生产事件，必须满足：

1. 事件名必须存在于 `04-事件字典.md`。
2. 事件头字段必须完整。
3. 新增事件必须补充到事件字典。
4. 关键事件必须写明：
   - 生产者
   - 消费者
   - 幂等要求
   - 投递语义
   - 是否持久化
   - 是否审计

### 10.1 扩展新增事件流程

1. 先更新事件字典。
2. 再定义事件 payload struct。
3. 再实现 Subscriber / Publisher。
4. 再补测试和日志。

---

## 11. 状态读写规范

扩展允许访问状态，但必须遵守边界。

### 11.1 允许的方式

- 通过 Repository 读写核心对象
- 通过应用服务更新复杂业务状态
- 通过事件触发异步投影或记忆沉淀

### 11.2 禁止的方式

- 绕过 Repository 直接改数据库文件
- 在多个扩展中同时持有同一对象的可变副本
- 直接把临时状态塞进 WorkingMemory 当作长期存储
- 用 Event 代替稳定状态对象

---

## 12. 审批与权限规范

扩展只要涉及副作用，就必须声明权限边界。

### 12.1 必须声明的内容

- 操作类型
- 风险等级
- 是否需要审批
- 默认审批策略
- 是否允许批量审批
- 是否支持 dry-run

### 12.2 典型高风险操作

- 删除、覆盖、移动文件
- 执行 shell 命令
- 打开外部链接
- 写入系统剪贴板
- 模拟快捷键和窗口控制
- 修改配置或长期记忆

---

## 13. 版本化与兼容性

### 13.1 版本号

每个扩展都应有自己的版本号。

### 13.2 兼容性要求

- 不兼容的 schema 变更必须升级主版本。
- 删除事件字段前应至少保留一个兼容期。
- 旧扩展在新运行时中失败时，不应拖垮系统。

### 13.3 扩展能力协商

可以定义可选能力标记，例如：

- `supports_recovery`
- `supports_hot_reload`
- `supports_dry_run`
- `supports_audit_export`

---

## 14. 测试规范

每类扩展都应至少包含以下测试：

### 14.1 单元测试

- 配置解析
- 元数据正确性
- 核心逻辑
- 失败分支

### 14.2 集成测试

- 注册成功
- 事件链路正确
- 状态写入正确
- 与审批策略协同正确

### 14.3 恢复测试

- 重启后恢复是否重复执行
- Pending 状态是否正确恢复
- 幂等性是否满足

---

## 15. 新增扩展标准步骤

### 15.1 新增一个工具扩展

1. 定义 Tool 接口实现。
2. 定义输入输出 schema。
3. 声明风险等级与审批规则。
4. 注册到 ToolRegistry。
5. 定义其消费/产出的事件。
6. 补日志、审计、测试。

### 15.2 新增一个感知扩展

1. 实现 Source 接口。
2. 配置监听参数与节流参数。
3. 定义感知事件 payload。
4. 注册到 Perception Manager。
5. 补错误隔离与停止逻辑。

### 15.3 新增一个 Agent 扩展

1. 定义 Agent 的能力边界。
2. 定义其可处理的 Task Kind。
3. 声明允许调用的工具集。
4. 注册到 AgentRegistry。
5. 补执行、失败、重试与降级逻辑。

---

## 16. 禁止事项

以下做法明确禁止：

1. 扩展直接跨层调用未公开接口。
2. 扩展自己创建新的 EventBus、Logger、Repository 实例。
3. 扩展绕过 Approval 执行高风险操作。
4. 扩展直接写 UI 状态来驱动业务结果。
5. 扩展发布未登记事件。
6. 扩展把长时间阻塞逻辑放进中间件主链。
7. 扩展修改核心对象结构但不更新文档。

  perception/
    sources/
      clipboard/

      screen/

    processors/
      intent/

  action/
    tools/
      browser/
      terminal/
      filesystem/

  recovery/
    approvals/
    tasks/

对于用户可下载安装的能力，可以进一步抽象到：

```text
skills/
  clipboard-helper/
  article-summarizer/
  code-reviewer/
```

但无论是内置扩展还是 skills，都应服从同一套接口与注册规则。

## 18. 建议的首批可扩展点

建议优先把以下能力做成标准扩展点：

1. Perception Source
2. Tool
3. Agent
4. Middleware
5. Recovery Handler
6. UI View

因为这六类最能支撑后续功能扩展，而不会让主干代码不断膨胀。

---

## 19. 最终结论

CialloClaw 的扩展不应该是“往目录里多塞几个包”，而应该是：

- 有元数据
- 有生命周期
- 有配置
- 有事件协议
- 有状态边界
- 有审批边界
- 有恢复能力
- 有日志与测试

后续所有新增功能，都优先按“实现接口 + 注册扩展 + 声明事件 + 补状态流转”的模式接入。

这套规范一旦执行，系统才能长期保持可维护、可扩展、可恢复。
