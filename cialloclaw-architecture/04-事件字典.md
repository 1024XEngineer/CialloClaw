# 事件字典

> 目标：给 CialloClaw 建立一份可以直接指导开发、联调、扩展和排障的统一事件字典。
>
> 本文档约束：事件命名、事件头、主干事件、载荷约定、同步语义、持久化规则、审计规则。

---

## 1. 事件字典的作用

在 CialloClaw 中，事件不是“随手发一个字符串”，而是系统内部的**运行协议**。

事件字典要解决的问题：

- 什么事件允许存在
- 每个事件由谁发布
- 每个事件由谁消费
- 事件最少要带哪些上下文标识
- 事件是否允许异步
- 事件是否必须入日志/审计
- 事件的后续链路通常是什么

一句话：

**新功能优先通过“新增事件 + 新增订阅者”接入，而不是跨层直接调用。**

---

## 2. 统一命名规则

### 2.1 推荐事件值命名

推荐事件值统一使用**领域点号风格**：

```text
<domain>.<object>.<action>
```

例如：

- `user.input.received`
- `intent.recognized`
- `task.created`
- `task.execution.started`
- `approval.requested`
- `ui.message.display`

### 2.2 Go 常量命名

代码中可以继续保留 Go 风格常量名：

```go
const (
    EventTypeUserInputReceived   = "user.input.received"
    EventTypeIntentRecognized    = "intent.recognized"
    EventTypeTaskCreated         = "task.created"
    EventTypeApprovalRequested   = "approval.requested"
)
```

### 2.3 命名禁忌

不要使用：

- `do_task`
- `something_happened`
- `process_data`
- `agent_work`

原因：

- 语义不稳定
- 不利于检索
- 不利于事件治理
- 不利于版本化

---

## 3. 统一事件头

所有正式事件都应至少包含以下字段。

```go
type Event struct {
    ID           string
    Type         string
    Version      string

    SessionID    string
    TaskID       *string
    TaskStepID   *string
    LoopID       *string
    ApprovalID   *string

    Source       string
    TraceID      string
    SpanID       string
    ParentSpanID *string

    Priority     int
    Timestamp    time.Time

    Headers      map[string]string
    Payload      any
}
```

### 3.1 必填字段

以下字段必须存在：

- `ID`
- `Type`
- `Version`
- `SessionID`
- `Source`
- `TraceID`
- `SpanID`
- `Timestamp`

### 3.2 条件必填字段

以下字段在特定场景下必须存在：

- `TaskID`：与任务有关的事件必须带
- `TaskStepID`：步骤级执行事件必须带
- `LoopID`：循环相关事件必须带
- `ApprovalID`：审批相关事件必须带
- `ParentSpanID`：事件由上游事件衍生时建议带

### 3.3 Source 规范

`Source` 应表示“谁发出的这个事件”，而不是“谁最初触发了整个链路”。

推荐值示例：

- `perception.user_input`
- `perception.selection`
- `cognition.intent_classifier`
- `cognition.task_planner`
- `cognition.director`
- `action.tool_executor`
- `action.approval_service`
- `ui.tui`
- `runtime.eventbus`
- `state.memory_manager`

---

## 4. 优先级与投递语义

### 4.1 优先级建议

```go
const (
    PriorityLow = iota
    PriorityNormal
    PriorityHigh
    PriorityCritical
)
```

默认规则：

- 用户显式输入：`PriorityCritical`
- 审批响应：`PriorityCritical`
- 任务执行结果：`PriorityHigh`
- 屏幕/剪贴板变化：`PriorityNormal`
- 监控、指标、整理类事件：`PriorityLow`

### 4.2 投递语义

每类事件都需要明确其投递方式：

- **同步顺序**：必须按顺序处理，不能乱序
- **异步并发**：允许并发消费
- **可丢弃**：高频低价值，可被采样或去重
- **必须持久化**：必须进入日志/审计/状态变更

---

## 5. 领域事件总表

以下事件为当前推荐主干事件集。后续扩展应优先在现有领域下补充，而不是随意开新域。

---

## 6. Perception 事件

### 6.1 `user.input.received`

**含义**：用户在终端/TUI/热键触发输入后，系统收到输入内容。

- 生产者：`perception.user_input`
- 消费者：`cognition.context_aggregator`、`cognition.intent_classifier`、`state.log`
- 是否必须带 `TaskID`：否
- 是否必须带 `TraceID`：是
- 投递语义：同步优先，不能被低优先级事件饿死
- 审计：记录普通日志即可

Payload：

```go
type UserInputPayload struct {
    Text        string
    Raw         string
    InputSource string // terminal / tui / hotkey / voice
}
```

常见后续事件：

- `context.updated`
- `intent.recognized`
- `ui.message.display`（澄清问题场景）

---

### 6.2 `selection.changed`

**含义**：用户选中文本发生变化。

- 生产者：`perception.selection`
- 消费者：`cognition.context_aggregator`、`cognition.intent_classifier`
- `TaskID`：否
- 投递语义：异步，可去重
- 审计：通常不需要审计

Payload：

```go
type SelectionChangedPayload struct {
    Text        string
    ContentType string // text / code / url / file / image
    App         string
    WindowTitle string
}
```

---

### 6.3 `clipboard.changed`

**含义**：粘贴板内容变化。

- 生产者：`perception.clipboard`
- 消费者：`cognition.context_aggregator`
- 投递语义：异步，可去重，可采样
- 审计：默认不审计

---

### 6.4 `screen.captured`

**含义**：屏幕抓取完成。

- 生产者：`perception.screen`
- 消费者：`cognition.context_aggregator`、`cognition.screen_analyzer`
- 投递语义：异步，通常可采样
- 审计：默认不审计

---

### 6.5 `window.focus.changed`

**含义**：焦点窗口切换。

- 生产者：`perception.window`
- 消费者：`cognition.context_aggregator`、`state.working_memory`
- 投递语义：异步，可采样

---

### 6.6 `file.changed`

**含义**：被监听的文件或目录发生变化。

- 生产者：`perception.filewatch`
- 消费者：`cognition.context_aggregator`、`task` 相关订阅者
- 投递语义：异步，可去重

---

## 7. Cognition 事件

### 7.1 `context.updated`

**含义**：上下文聚合器已把多源输入整理为可供后续使用的上下文。

- 生产者：`cognition.context_aggregator`
- 消费者：`cognition.intent_classifier`、`cognition.planner`
- `TaskID`：否
- 投递语义：同步或单线程顺序，避免覆盖竞争

Payload：

```go
type ContextUpdatedPayload struct {
    Summary         string
    WindowContext   map[string]any
    Selection       *string
    RecentEventIDs  []string
}
```

---

### 7.2 `intent.recognized`

**含义**：完成意图识别。

- 生产者：`cognition.intent_classifier`
- 消费者：`cognition.task_planner`、`state.working_memory`
- 投递语义：同步优先
- 审计：建议记录日志

Payload：

```go
type IntentRecognizedPayload struct {
    Intent      string
    Confidence  float64
    Entities    map[string]any
    RawInput    string
}
```

---

### 7.3 `task.created`

**含义**：创建新的任务对象。

- 生产者：`cognition.task_planner` 或 `cognition.director`
- 消费者：`state.task_repository`、`ui.task_view`
- `TaskID`：是
- 投递语义：必须顺序、必须持久化
- 审计：记录日志

Payload：

```go
type TaskCreatedPayload struct {
    Kind      string
    Title     string
    Goal      string
    ParentTaskID *string
}
```

---

### 7.4 `task.planned`

**含义**：任务已生成执行计划。

- 生产者：`cognition.task_planner`
- 消费者：`cognition.director`、`state.task_repository`
- `TaskID`：是
- 投递语义：必须持久化

Payload：

```go
type TaskPlannedPayload struct {
    PlanID string
    Steps  []map[string]any
}
```

---

### 7.5 `task.assigned`

**含义**：任务或步骤被分配给某个 Agent / Tool。

- 生产者：`cognition.director`
- 消费者：`action.executor`、`ui.task_view`
- `TaskID`：是
- 投递语义：必须持久化

---

### 7.6 `loop.started`

**含义**：任务进入循环执行模式。

- 生产者：`cognition.loop_controller`
- 消费者：`state.loop_repository`、`ui.loop_view`
- `TaskID`：是
- `LoopID`：是
- 投递语义：必须持久化

---

### 7.7 `loop.iterated`

**含义**：循环完成一轮迭代。

- 生产者：`cognition.loop_controller`
- 消费者：`state.loop_repository`、`ui.loop_view`、`memory.extractor`
- `LoopID`：是
- 投递语义：必须持久化

Payload：

```go
type LoopIteratedPayload struct {
    Iteration     int
    Decision      string // continue / stop / fallback / ask_user
    Score         *float64
    Summary       string
}
```

---

### 7.8 `loop.converged`

**含义**：循环达到收敛条件。

- 生产者：`cognition.loop_controller`
- 消费者：`state.loop_repository`、`cognition.director`
- `LoopID`：是
- 投递语义：必须持久化

---

## 8. Action 事件

### 8.1 `task.execution.started`

**含义**：任务执行已开始。

- 生产者：`action.executor`
- 消费者：`state.task_repository`、`ui.task_view`
- `TaskID`：是
- 投递语义：必须持久化

---

### 8.2 `tool.call.requested`

**含义**：准备调用某个工具。

- 生产者：`action.executor` 或某 Agent
- 消费者：`action.tool_executor`、`action.approval_service`
- `TaskID`：是
- 投递语义：必须持久化
- 审计：高风险工具应审计

Payload：

```go
type ToolCallRequestedPayload struct {
    ToolName     string
    Args         map[string]any
    RiskLevel    string
    RequireApproval bool
}
```

---

### 8.3 `approval.requested`

**含义**：高风险动作进入待确认队列。

- 生产者：`action.approval_service`
- 消费者：`ui.confirmation`、`state.approval_repository`
- `ApprovalID`：是
- `TaskID`：建议带
- 投递语义：必须持久化、必须审计

Payload：

```go
type ApprovalRequestedPayload struct {
    ActionType    string
    ActionSummary string
    ProposedArgs  map[string]any
    RiskLevel     string
    TimeoutAt     *time.Time
}
```

---

### 8.4 `approval.responded`

**含义**：用户对审批做出批准/拒绝/取消响应。

- 生产者：`ui.confirmation`
- 消费者：`action.approval_service`、`state.approval_repository`
- `ApprovalID`：是
- 投递语义：同步优先、必须持久化、必须审计

Payload：

```go
type ApprovalRespondedPayload struct {
    Decision string // approved / rejected / canceled / expired
    Reason   *string
}
```

---

### 8.5 `tool.call.completed`

**含义**：工具调用完成。

- 生产者：`action.tool_executor`
- 消费者：`state.task_repository`、`cognition.loop_controller`、`ui.task_view`
- `TaskID`：是
- 投递语义：必须持久化

Payload：

```go
type ToolCallCompletedPayload struct {
    ToolName    string
    Success     bool
    Result      map[string]any
    Error       *string
    DurationMs  int64
}
```

---

### 8.6 `task.execution.completed`

**含义**：任务执行完成。

- 生产者：`action.executor`
- 消费者：`state.task_repository`、`memory.extractor`、`ui.task_view`
- `TaskID`：是
- 投递语义：必须持久化

---

### 8.7 `task.execution.failed`

**含义**：任务执行失败。

- 生产者：`action.executor`
- 消费者：`state.task_repository`、`cognition.loop_controller`、`ui.error_view`
- `TaskID`：是
- 投递语义：必须持久化、建议审计高风险失败

---

## 9. State 事件

### 9.1 `memory.working.updated`

**含义**：工作记忆更新。

- 生产者：`state.working_memory`
- 消费者：一般不建议大量订阅，仅供调试/观察
- 投递语义：可选持久化

---

### 9.2 `memory.episodic.created`

**含义**：生成一条情景记忆。

- 生产者：`memory.extractor`
- 消费者：`state.memory_repository`
- 投递语义：必须持久化

---

### 9.3 `memory.semantic.extracted`

**含义**：从情景中提炼出稳定事实。

- 生产者：`memory.extractor`
- 消费者：`state.memory_repository`
- 投递语义：必须持久化

---

### 9.4 `memory.procedural.extracted`

**含义**：提炼出工作流/操作套路。

- 生产者：`memory.extractor`
- 消费者：`state.memory_repository`
- 投递语义：必须持久化

---

### 9.5 `log.entry.created`

**含义**：生成一条结构化日志。

- 生产者：`state.logger`
- 消费者：观察类组件、导出器
- 投递语义：可异步，但日志存储必须可靠

---

## 10. UI 事件

### 10.1 `ui.message.display`

**含义**：向用户展示一条消息。

- 生产者：任意业务模块
- 消费者：`ui.tui`
- 投递语义：异步即可

---

### 10.2 `ui.notification.push`

**含义**：发送系统通知。

- 生产者：`action.executor`、`scheduler`、`approval_service`
- 消费者：`ui.notification`
- 投递语义：异步即可，可合并

---

### 10.3 `ui.confirmation.requested`

**含义**：要求 UI 向用户展示确认请求。

- 生产者：`action.approval_service`
- 消费者：`ui.interactive`
- `ApprovalID`：是
- 投递语义：必须顺序

---

### 10.4 `ui.confirmation.responded`

**含义**：UI 收到用户确认结果并发回系统。

- 生产者：`ui.interactive`
- 消费者：`action.approval_service`
- `ApprovalID`：是
- 投递语义：同步优先

---

## 11. System 事件

### 11.1 `service.started`

**含义**：服务启动完成。

- 生产者：`runtime.bootstrap`
- 消费者：日志、UI、监控
- 投递语义：必须持久化到日志

---

### 11.2 `service.stopped`

**含义**：服务正常停止。

- 生产者：`runtime.shutdown`
- 消费者：日志、状态保存器

---

### 11.3 `config.updated`

**含义**：配置热更新。

- 生产者：`config.loader`
- 消费者：依赖配置的模块
- 投递语义：同步或串行，避免并发配置漂移

---

### 11.4 `system.error`

**含义**：发生系统级错误。

- 生产者：任意模块
- 消费者：日志、UI、恢复策略
- 投递语义：必须审计严重错误

---

## 12. 事件治理规则

### 12.1 哪些事件必须持久化

以下事件应至少进入结构化日志：

- `task.created`
- `task.planned`
- `task.assigned`
- `loop.started`
- `loop.iterated`
- `loop.converged`
- `task.execution.started`
- `task.execution.completed`
- `task.execution.failed`
- `approval.requested`
- `approval.responded`
- `system.error`
- `service.started`
- `service.stopped`

### 12.2 哪些事件必须审计

以下事件必须进入审计日志：

- 涉及文件删除/覆盖/移动的工具调用
- 终端命令执行
- 配置修改
- 系统快捷键模拟
- 外部打开文件/应用
- 审批请求与审批响应

### 12.3 哪些事件允许丢弃

以下事件允许采样或去重：

- `clipboard.changed`
- `window.focus.changed`
- `screen.captured`
- 高频 `file.changed`
- 指标类事件

---

## 13. 新增事件的 Checklist

新增事件前，必须回答清楚：

1. 它属于哪个领域？
2. 它是事实事件，还是命令伪装成事件？
3. 它是否已有同义事件可复用？
4. 它的生产者是谁？
5. 它的消费者是谁？
6. 它要不要带 `TaskID / LoopID / ApprovalID`？
7. 它是否必须持久化？
8. 它是否必须审计？
9. 它是同步还是异步？
10. 失败时的补偿路径是什么？

如果这 10 条说不清，就不应该立刻新增事件。

---

## 14. 推荐最小事件集

如果你要先做 MVP，建议先实现以下最小集：

- `user.input.received`
- `context.updated`
- `intent.recognized`
- `task.created`
- `task.planned`
- `task.assigned`
- `task.execution.started`
- `tool.call.requested`
- `tool.call.completed`
- `approval.requested`
- `approval.responded`
- `task.execution.completed`
- `task.execution.failed`
- `ui.message.display`
- `service.started`
- `system.error`

这套事件足够支撑一条完整主链路。
