# 09-错误处理与恢复策略

> 本文档定义 CialloClaw 的错误处理、失败隔离、重试、补偿、恢复与重启策略。目标不是“让系统永不失败”，而是让系统在失败时可解释、可回退、可恢复、可继续运行。

---

## 1. 文档目标

本文档回答以下问题：

1. 系统中有哪些错误类型。
2. 哪些错误可以重试，哪些必须中断。
3. 失败时应更新哪些状态对象。
4. 重启后哪些对象要恢复，按什么顺序恢复。
5. 如何避免重复执行和脏状态。
6. 用户应该看到什么，日志应该记录什么。

## 2. 总体原则

### 2.1 错误是系统状态的一部分

错误不应只打印日志或返回给调用者，而应反映到系统状态中。例如：

- Task 失败 -> `task.status = failed`
- Loop 超时 -> `loop.status = timeout`
- Approval 超时 -> `approval.status = expired`
- Tool 执行失败 -> 记录到 TaskStep / Log / Event

### 2.2 失败必须可追踪

每个错误至少应关联：

- SessionID
- TaskID（如适用）
- LoopID（如适用）
- ApprovalID（如适用）
- EventID / TraceID
- 时间戳
- 错误分类
- 是否可重试

### 2.3 失败隔离优先

单个扩展、单个工具、单个感知源失败，不应拖垮整个系统。

### 2.4 重试必须有限且可解释

重试不是默认万能方案。每次重试都必须满足：

- 有明确上限
- 有退避策略
- 有去重或幂等保证
- 有日志记录

### 2.5 恢复必须幂等

恢复流程允许重复执行，不应导致：

- 同一 Task 被重复完成
- 同一文件被重复移动
- 同一审批被重复通过
- 同一 Loop 被重复推进

## 3. 错误分类

推荐使用统一错误分类，而不是到处返回裸字符串。

```go
package errors

type ErrorKind string

const (
    ErrorKindValidation    ErrorKind = "validation"
    ErrorKindConfiguration ErrorKind = "configuration"
    ErrorKindDependency    ErrorKind = "dependency"
    ErrorKindTimeout       ErrorKind = "timeout"
    ErrorKindCanceled      ErrorKind = "canceled"
    ErrorKindPermission    ErrorKind = "permission"
    ErrorKindApproval      ErrorKind = "approval"
    ErrorKindNotFound      ErrorKind = "not_found"
    ErrorKindConflict      ErrorKind = "conflict"
    ErrorKindRateLimit     ErrorKind = "rate_limit"
    ErrorKindTransient     ErrorKind = "transient"
    ErrorKindPermanent     ErrorKind = "permanent"
    ErrorKindExternal      ErrorKind = "external"
    ErrorKindInternal      ErrorKind = "internal"
)

type AppError struct {
    Kind       ErrorKind
    Code       string
    Message    string
    Cause      error
    Retryable  bool
    UserVisible bool
    Metadata   map[string]any
}
```

## 4. 按层定义错误边界

### 4.1 Config / Bootstrap 层

#### 常见错误

- 配置缺失
- 配置格式错误
- API Key 缺失或无效
- 数据目录权限不足
- 依赖初始化失败

#### 处理策略

- 启动前尽早失败
- 输出明确错误信息
- 写启动日志
- 不进入主运行阶段

#### 恢复策略

- 无自动恢复
- 需要人工修复配置后重启

---

### 4.2 EventBus / Middleware 层

#### 常见错误

- 事件格式不合法
- 必填头字段缺失
- 中间件 panic
- 订阅者执行失败
- 事件投递阻塞

#### 处理策略

- 非法事件直接拒绝并记录日志
- 中间件 panic 要 recover，不能让总线崩溃
- 关键订阅者失败要发出 `system.error` 或 `task.failed` 类事件
- 对不可恢复的坏事件做丢弃并审计

#### 恢复策略

- 不恢复单个已损坏事件
- 恢复订阅者健康状态
- 若有持久化事件队列，则仅恢复未确认部分

---

### 4.3 Perception 层

#### 常见错误

- 系统监听权限不足
- OCR 组件不可用
- 剪贴板读取失败
- 文件监听器断开
- 感知事件重复风暴

#### 处理策略

- 单独隔离感知源
- 重试监听器初始化
- 加节流与去重
- 连续失败过多时降级或熔断该 source

#### 恢复策略

- Source 可单独重启
- 重启系统时重新注册并启动 source

---

### 4.4 Cognition 层

#### 常见错误

- 上下文构建失败
- 意图识别失败
- 计划生成失败
- Loop 收敛判断异常
- Agent 路由失败

#### 处理策略

- 标记对应 Task 为 `blocked` 或 `failed`
- 允许 fallback 到更简单的处理链路
- 必要时转为用户确认或用户追问

#### 恢复策略

- 可从 Task 当前状态继续
- 若 Loop 中断，可从最近一次 iteration 恢复

---

### 4.5 Action / Tool 层

#### 常见错误

- 工具超时
- 工具执行失败
- 外部程序返回非零
- 文件冲突
- 网络不可达
- 高风险操作未获审批

#### 处理策略

- 记录 ToolCall 与 ToolResult
- 根据错误分类决定重试、补偿或终止
- 高风险操作未审批直接拒绝执行
- 尽量支持 dry-run

#### 恢复策略

- 幂等工具可安全重试
- 非幂等工具要依赖补偿或人工确认

---

### 4.6 State / Repository 层

#### 常见错误

- 数据写入失败
- 事务冲突
- 锁冲突
- 索引损坏
- 快照读取失败

#### 处理策略

- 失败必须回滚事务
- 标记对象为不确定状态时要显式记录
- 非关键投影失败不应影响主业务对象提交

#### 恢复策略

- 优先恢复核心对象，再恢复派生索引
- 索引类数据可重建
- 快照类数据可丢弃并重新生成

---

### 4.7 UI 层

#### 常见错误

- 视图渲染失败
- 用户交互超时
- 输入解析失败
- 通知系统失败

#### 处理策略

- UI 错误不应改变核心业务状态
- 给用户可理解的提示
- 关键交互失败应保留原 Task 状态

#### 恢复策略

- UI 重启后可从 Session/Task/Approval 重新渲染

---

## 5. 错误严重性分级

建议定义严重性分级：

```text
DEBUG    仅供调试
INFO     正常失败分支或可预期拒绝
WARN     局部异常，可降级处理
ERROR    影响当前任务，但不影响系统整体
FATAL    无法继续启动或无法继续核心运行
AUDIT    涉及权限、审批、安全的关键记录
```

## 6. 错误与状态对象的映射

### 6.1 Session

#### 更新规则

- 致命启动错误：Session 不创建或创建后立即标记 `crashed`
- 运行时非致命错误：Session 仍可 `active`
- 无法恢复的全局崩溃：Session -> `crashed`

### 6.2 Task

#### 更新规则

- 可等待外部条件：`waiting`
- 需用户介入：`blocked`
- 不可恢复：`failed`
- 被用户取消：`canceled`
- 重试排队：`queued`

### 6.3 Loop

#### 更新规则

- 超过最大次数：`stopped`
- 达到收敛条件：`converged`
- 超时：`timeout`
- 内部异常无法继续：`failed`
- 用户暂停：`paused`

### 6.4 Approval

#### 更新规则

- 等待用户输入：`pending`
- 用户同意：`approved`
- 用户拒绝：`rejected`
- 超时未处理：`expired`
- 上游取消：`canceled`

## 7. 重试策略

### 7.1 允许重试的错误

通常仅以下错误允许自动重试：

- `timeout`
- `rate_limit`
- `transient`
- 某些 `external`
- 某些资源争用类 `conflict`

### 7.2 不应自动重试的错误

- `validation`
- `configuration`
- `permission`
- `approval`
- 明确的 `permanent`

### 7.3 推荐退避策略

```text
第 1 次失败：立即记录并等待 500ms
第 2 次失败：等待 1s
第 3 次失败：等待 2s
第 4 次失败：等待 5s
超过上限：标记 failed 或 blocked
```

### 7.4 重试记录

每次重试都必须记录：

- 对象 ID
- 当前重试次数
- 原因
- 下一次重试时间
- 是否仍可自动重试

## 8. 补偿策略

并不是所有失败都能重试。对于已产生副作用但未完整完成的操作，应考虑补偿。

### 8.1 典型补偿场景

- 文件已移动一半
- 多步写入只成功部分
- 一组工具调用中前半成功后半失败
- Approval 已通过但执行中断

### 8.2 补偿原则

- 优先采用业务补偿，而不是低层回滚幻觉
- 补偿动作本身也要记录日志和审批
- 无法自动补偿时，标记人工介入

### 8.3 典型策略

- 移动文件失败：记录已移动清单，提供恢复脚本或回滚步骤
- 计划执行失败：保留已完成 step，重新从未完成 step 开始
- 写入类操作失败：使用临时文件 + 原子替换降低半成状态概率

---

## 9. 幂等策略

恢复和重试的前提是幂等。

### 9.1 幂等对象

以下动作应优先设计为幂等：

- Task 状态推进
- Approval 状态更新
- Log 追加
- 某些 Tool 的查询操作
- Recovery Handler

### 9.2 幂等手段

- 使用 EventID / CommandID / OperationID
- 使用唯一索引防重
- 执行前检查对象当前状态
- 对已完成对象拒绝重复推进

### 9.3 示例

若某个 Approval 已 `approved`，恢复流程再次执行时，不应再次写入“已批准”副作用，而应只确认状态一致。

## 10. Panic 处理策略

### 10.1 可 recover 的区域

以下位置必须加 panic recover：

- EventBus 分发边界
- Middleware 执行边界
- Subscriber 执行边界
- 后台 worker goroutine
- Perception source goroutine
- UI 异步刷新 goroutine

### 10.2 Panic 后的处理

- 记录 FATAL/ERROR 日志
- 记录堆栈
- 更新相关 Task/Loop 状态
- 视严重性决定是否禁用该扩展或关闭系统

### 10.3 禁止做法

- 恢复 panic 后静默吞掉
- 让 goroutine 直接退出且无日志

## 11. 用户可见错误与内部错误

### 11.1 用户可见错误

应尽量短、明确、可操作，例如：

- “未获得删除文件的确认，本次操作已取消。”
- “终端命令执行超时，已停止重试。”
- “剪贴板监听暂时不可用，稍后会自动重试。”

### 11.2 内部错误

内部日志应保留：

- 原始错误
- 堆栈
- 关联对象 ID
- 输入摘要
- 环境信息

### 11.3 原则

- 用户看到的是可操作信息
- 日志里保留技术细节
- 不向用户暴露内部密钥或敏感路径

---

## 12. 恢复范围定义

### 12.1 必须恢复的对象

- Pending Approval
- Queued / Running / Waiting / Blocked Task
- Running / Paused Loop
- Session 快照或 WorkingMemory 基础信息
- 未刷写的审计/日志缓冲

### 12.2 可以重建的对象

- 搜索索引
- 部分缓存
- Context 压缩结果
- UI ViewModel
- 派生读模型

### 12.3 不恢复的对象

- 已明确失败且不可重试的瞬时动作
- 已过期的临时上下文缓存
- 已完成且无后续依赖的临时事件

---

## 13. 启动恢复顺序

建议恢复顺序如下：

1. Repository 可用性检查
2. 恢复 Session 基础信息
3. 恢复 Pending Approval
4. 恢复 Active Task
5. 恢复 Loop
6. 恢复 Worker 队列中的可恢复任务
7. 重建必要投影与索引
8. 启动 UI 并渲染恢复结果
9. 启动感知源

### 13.1 为什么 Approval 优先于 Task

因为某些 Task 处于 `blocked` 正是由于 Approval 未决。先恢复 Approval，Task 才能解释当前状态。

---

## 14. 对象级恢复策略

### 14.1 Session 恢复

- 若上次正常关闭：可创建新 Session，也可延续最近 Session
- 若上次异常退出：将旧 Session 标记为 `crashed_recovered` 或记录恢复标签
- 恢复其 WorkingMemory 摘要与活动对象引用

### 14.2 Task 恢复

按状态处理：

- `queued`：重新入队
- `planned`：可重新入队或等待调度
- `running`：恢复为 `waiting_recovery` 后再决定是否重跑
- `waiting`：恢复等待条件
- `blocked`：检查 Approval 或用户输入
- `failed/succeeded/canceled`：仅保留历史，不重跑

### 14.3 Loop 恢复

- 恢复 `current_iteration`
- 恢复 stop condition 与 convergence state
- 若上次 iteration 未完成，标记该 iteration 中断
- 再决定是继续当前轮还是新开一轮

### 14.4 Approval 恢复

- 仅恢复 `pending` 且未过期的 Approval
- 恢复后重新显示给用户
- 已 `approved/rejected/expired/canceled` 的 Approval 不再进入待处理队列

### 14.5 Memory 恢复

- WorkingMemory：恢复基础上下文
- Episodic/Semantic/Procedural：作为持久知识直接可用
- 派生摘要可重建

### 14.6 Log 恢复

- 未落盘缓冲尽量刷写
- 若是 append-only 存储，恢复后不修改历史日志

## 15. 失败后事件规范

发生错误后，建议发布标准事件，而不是仅在本地处理。

### 15.1 推荐事件

- `system.error.raised`
- `task.failed`
- `task.blocked`
- `task.retry.scheduled`
- `loop.timeout`
- `loop.failed`
- `approval.expired`
- `tool.execution.failed`
- `recovery.started`
- `recovery.completed`
- `recovery.failed`

### 15.2 事件要求

错误事件必须带：

- 错误分类
- 是否可重试
- 关联对象 ID
- 用户可见摘要
- 技术日志引用

## 16. 熔断与降级策略

### 16.1 熔断对象

适合熔断的对象：

- 某个持续失败的感知源
- 某个外部依赖工具
- 某个经常 panic 的扩展
- 某个高频失败的 Agent

### 16.2 熔断条件

例如：

- 30 秒内连续失败 5 次
- 5 分钟内超时 10 次
- 单次致命 panic

### 16.3 降级方式

- 关闭该扩展
- 切换到更简单的替代实现
- 转为用户手动确认模式
- 只保留只读能力，禁用副作用能力

---

## 17. 审批相关错误策略

### 17.1 未审批直接执行

视为严重错误，必须：

- 拒绝执行
- 记录 AUDIT 日志
- 发布安全事件

### 17.2 审批超时

- Approval -> `expired`
- Task -> `blocked` 或 `canceled`
- 给用户可见提示

### 17.3 用户拒绝

- Approval -> `rejected`
- Task 根据业务决定 `blocked` 或 `canceled`
- 不应再自动重试同一危险动作

---

## 18. 测试要求

### 18.1 必测场景

1. Tool 超时
2. EventBus 中间件 panic
3. Subscriber 执行失败
4. Approval 超时
5. Loop 达到最大迭代次数
6. Repository 写入失败回滚
7. 崩溃后恢复 Pending Approval
8. 崩溃后恢复 Running Task
9. 重试上限触发
10. 熔断后降级生效

### 18.2 恢复测试原则

- 至少做一次“运行中直接 kill 进程”的集成测试
- 验证恢复后无重复副作用
- 验证状态对象与日志一致

## 20. 禁止事项

以下做法明确禁止：

1. 失败后只打日志、不更新状态。
2. 无上限自动重试。
3. 对不可幂等操作盲目重试。
4. panic 后静默恢复且不记录。
5. 未审批操作通过异常路径绕过执行。
6. 恢复流程直接重放所有历史事件。
7. 用 WorkingMemory 替代真实恢复依据。

---

## 21. 最终结论

CialloClaw 的错误处理与恢复策略应建立在以下主线上：

- 错误先分类
- 失败要落状态
- 重试必须有限
- 补偿优于幻想式回滚
- 恢复必须幂等
- 用户提示与内部日志分离
- 单点失败不拖垮全局

系统的目标不是“没有失败”，而是“失败后仍然可解释、可恢复、可继续演进”。
