# 事件总线与内部接口协议

> 本文档定义 CialloClaw 的内部协议：事件结构、EventBus 行为、中间件、订阅者、Tool、Agent、Repository 等。
>
> 目标：让系统的扩展通过接口实现，而不是通过修改主链路实现。

---

## 1. 总体设计目标

事件总线在本项目中不只是“传消息”，而是系统运行组织方式的中心。它的价值包括：

- 解耦模块
- 支持异步与并发
- 提供中间件插入点
- 支撑非线性流程（分支、反馈、迭代、中断、恢复）
- 让新增功能主要通过“新增订阅者 / 新增事件类型”接入

但同时必须牢记：

- EventBus 负责传递与分发
- **Task / Session / Loop / Approval / Memory / Log 才负责承载系统状态**

## 2. Event 设计建议

### 2.1 不推荐长期使用的简化写法

```go
type Event struct {
    Type      string
    Data      map[string]interface{}
    Source    string
    Timestamp time.Time
    SessionID string
    LoopID    string
}
```

这个结构足够原型验证，但到了中后期会出现问题：

- 类型不安全
- 缺少链路追踪信息
- 难以做版本管理
- 业务语义都被塞进 `map[string]interface{}`

---

### 2.2 推荐的正式 Event 模型

```go
type EventPriority int

const (
    PriorityLow EventPriority = iota
    PriorityNormal
    PriorityHigh
    PriorityCritical
)

type Event struct {
    ID           string
    Type         string
    Version      string

    SessionID    string
    TaskID       *string
    TaskStepID   *string
    LoopID       *string

    Source       string
    TraceID      string
    SpanID       string
    ParentSpanID *string

    Priority     EventPriority
    Timestamp    time.Time

    Headers      map[string]string
    Payload      any
}
```

### 2.3 设计原则

- Header 存放通用元数据
- Payload 存放具体业务数据
- Event 必须有 `ID / Type / Source / Timestamp / TraceID`
- 一类事件对应一种明确 Payload 结构
- 避免无边界地在 `map[string]any` 中堆字段

## 5. EventBus 接口设计

### 5.1 总线接口

```go
type Bus interface {
    Publish(ctx context.Context, event Event) error
    Subscribe(sub Subscriber) error
    Unsubscribe(sub Subscriber) error
    Use(mw Middleware)
}
```

### 5.2 Subscriber 接口

```go
type Subscriber interface {
    ID() string
    SubscribeTypes() []string
    Handle(ctx context.Context, event Event) error
}
```

### 5.3 Middleware 接口

```go
type Middleware interface {
    ID() string
    Handle(ctx context.Context, event Event, next func(context.Context, Event) error) error
}
```

### 5.4 为什么不建议在 Subscriber 上放 `GetSessionID()`

因为会话过滤最好收敛到以下两层：

1. 中间件层：统一做 Session/Trace 合法性检查
2. 业务层：Handler 根据事件内容决定是否处理

如果把会话过滤塞进 Subscriber 接口，会导致：

- 订阅者职责变重
- 动态多 Session 路由复杂
- 不利于广播类订阅者（日志、监控、审计）

更推荐：

- 在事件头部明确 `SessionID`
- 在中间件中先做通用检查
- 在订阅者内部做业务相关过滤

## 6. 分发与并发策略

### 6.1 原则

不是所有事件都应该无脑 `go Handle()`。

建议为事件处理定义策略：

- **SyncOrdered**：同步有序执行
- **AsyncParallel**：异步并行执行
- **AsyncWorkerPool**：异步进入 Worker 池

### 6.2 推荐策略示例

| 事件类型 | 建议策略 | 原因 |
|---|---|---|
| `user.input.received` | SyncOrdered | 保证输入顺序 |
| `intent.recognized` | SyncOrdered / WorkerPool | 避免状态竞争 |
| `log.entry.created` | AsyncParallel | 旁路逻辑 |
| `metrics.updated` | AsyncParallel | 非关键路径 |
| `tool.call.completed` | SyncOrdered | 需要精确驱动后续状态 |
| `approval.requested` | SyncOrdered | 必须稳定进入等待态 |
| `memory.episodic.created` | WorkerPool | 可异步沉淀 |

### 6.3 建议接口

```go
type DeliveryMode string

const (
    DeliverySyncOrdered DeliveryMode = "sync_ordered"
    DeliveryAsyncParallel DeliveryMode = "async_parallel"
    DeliveryAsyncWorkerPool DeliveryMode = "async_worker_pool"
)
```

可通过事件类型注册表决定分发模式。

## 7. 中间件建议

### 7.1 推荐中间件列表

- `logging`：统一日志
- `tracing`：TraceID / SpanID 注入与传播
- `deduplication`：去重
- `rate_limit`：限流
- `security_guard`：安全检查
- `session_guard`：Session 合法性检查
- `priority_scheduler`：优先级处理
- `loop_guard`：循环检测与熔断
- `metrics`：统计指标

### 7.2 中间件职责边界

中间件适合处理：

- 横切逻辑
- 通用元数据增强
- 安全与稳定性控制

中间件不适合处理：

- 业务决策
- 任务规划
- 工具调用主逻辑
- 状态对象核心变更

## 8. Tool 接口协议

### 8.1 统一 Tool 接口

```go
type Tool interface {
    Name() string
    Description() string
    Schema() any
    Metadata() ToolMetadata
    Execute(ctx context.Context, input ToolInput) (ToolResult, error)
}

type ToolInput struct {
    SessionID string
    TaskID    *string
    StepID    *string
    Args      map[string]any
}

type ToolResult struct {
    Success   bool
    Data      map[string]any
    Error     *string
    Artifacts []ArtifactRef
}

type ToolMetadata struct {
    Category       string
    RequiresApproval bool
    CostHint       string
    Timeout        time.Duration
    Idempotent     bool
}
```

### 8.2 ToolRegistry 接口

```go
type ToolRegistry interface {
    Register(tool Tool) error
    Get(name string) (Tool, bool)
    List() []Tool
}
```

### 8.3 设计规则

- Tool 不直接操作 UI
- Tool 不直接操作 Session 全局对象
- Tool 只通过输入参数和返回值工作
- 高风险 Tool 通过 Approval Policy 控制

## 9. Agent 接口协议

### 9.1 统一 Agent 接口

```go
type Agent interface {
    Name() string
    Capabilities() []string
    CanHandle(task Task) bool
    Execute(ctx context.Context, req AgentRequest) (AgentResponse, error)
}

type AgentRequest struct {
    SessionID     string
    Task          Task
    Step          *TaskStep
    Context       map[string]any
    AvailableTools []ToolDescriptor
}

type AgentResponse struct {
    Summary       string
    NextAction    string
    ToolCalls     []PlannedToolCall
    Output        map[string]any
}
```

### 9.2 AgentRegistry 接口

```go
type AgentRegistry interface {
    Register(agent Agent) error
    Match(task Task) []Agent
    List() []Agent
}
```

### 9.3 设计规则

- Agent 面向任务，不面向事件本身
- Event 只是触发 Agent 参与的方式
- Agent 的输出应尽量结构化，便于 Director 决策

## 10. Repository / Store 接口协议

### 10.1 为什么必须抽象存储

为了支持：

- 本地文件存储
- SQLite / BoltDB / BadgerDB
- 内存实现（测试）
- 后续替换实现不影响业务层

### 10.2 示例接口

```go
type SessionRepository interface {
    Save(ctx context.Context, s Session) error
    GetByID(ctx context.Context, id string) (Session, error)
    Update(ctx context.Context, s Session) error
}

type TaskRepository interface {
    Save(ctx context.Context, t Task) error
    GetByID(ctx context.Context, id string) (Task, error)
    ListBySession(ctx context.Context, sessionID string) ([]Task, error)
    Update(ctx context.Context, t Task) error
}

type ApprovalRepository interface {
    Save(ctx context.Context, a Approval) error
    GetByID(ctx context.Context, id string) (Approval, error)
    ListPendingBySession(ctx context.Context, sessionID string) ([]Approval, error)
    Update(ctx context.Context, a Approval) error
}
```

## 11. 状态更新规则

建议采用如下统一规则：

1. 事件进入后，先在 Handler 中解析 Payload
2. 加载相关状态对象（Session / Task / Loop / Approval）
3. 执行业务逻辑
4. 保存状态对象
5. 追加 Log
6. 再发布后续事件

顺序上尽量遵循：

```text
读状态 -> 算逻辑 -> 写状态 -> 记日志 -> 发新事件
```

这样比“先发事件再补状态”更稳。

## 12. 常见坏味道与修正建议

### 坏味道 1：所有事件都用 `map[string]any`

修正：为核心事件定义正式 Payload struct。

### 坏味道 2：总线里塞业务逻辑

修正：EventBus 只做分发、调度、中间件处理。

### 坏味道 3：订阅者随意并发改状态

修正：关键事件采用同步有序处理；状态更新通过 Repository 串起来。

### 坏味道 4：事件没有 TraceID

修正：统一注入 TraceID / SpanID / ParentSpanID。

### 坏味道 5：审批只是 UI 询问

修正：审批必须落为 Approval 状态实体。

## 13. 协议版本管理建议

### 13.1 事件版本化

每个事件带 `Version` 字段，如：

- `v1`
- `v2`

当 Payload 结构发生不兼容变化时：

- 新旧事件并行支持一段时间
- 明确废弃窗口
- 不建议直接无版本覆盖旧结构

### 13.2 Tool / Agent 元数据版本化

对重要 Tool 和 Agent 也建议保留：

- 版本号
- 能力标签
- 输入 Schema 版本
- 输出 Schema 版本
