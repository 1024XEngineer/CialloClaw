# 扩展开发指南

> 本文档定义如何在不破坏主架构的前提下，为 CialloClaw 新增功能。
>
> 核心目标：未来新增功能应主要通过“实现接口 + 注册组件 + 定义事件”完成，而不是大量改动原有主链路。

---

## 1. 扩展总原则

### 1.1 新功能优先走扩展点

扩展优先顺序建议如下：

1. 新增事件类型
2. 新增 Payload 结构
3. 新增 Subscriber / Handler
4. 新增 Agent / Tool / Middleware
5. 新增 Repository 实现
6. 最后才考虑修改主流程

### 1.2 新功能接入前必须回答 6 个问题

1. 它会发布什么事件？
2. 它会订阅什么事件？
3. 它会读取哪些状态对象？
4. 它会更新哪些状态对象？
5. 它是否需要审批？
6. 它是否会产生长期记忆？

## 2. 标准扩展点清单

### 2.1 感知扩展点

适用场景：

- 新增剪贴板监听
- 新增屏幕 OCR
- 新增语音输入
- 新增文件系统监控

推荐接口：

```go
type Sensor interface {
    Name() string
    Start(ctx context.Context) error
    Stop(ctx context.Context) error
}
```

工作方式：

- Sensor 内部监听现实世界输入
- 采集后组装 Event
- 发布到 EventBus

### 2.2 认知扩展点

适用场景：

- 新增意图分类能力
- 新增上下文增强器
- 新增任务规划器
- 新增任务路由策略

推荐接口：

```go
type IntentRecognizer interface {
    Recognize(ctx context.Context, input string, wm WorkingMemory) (IntentRecognizedPayload, error)
}

type Planner interface {
    Plan(ctx context.Context, task Task, ctxData map[string]any) (TaskPlan, error)
}

type Router interface {
    Route(ctx context.Context, task Task) (string, error) // agent name
}
```

### 2.3 执行扩展点

适用场景：

- 新增浏览器自动化
- 新增文件操作工具
- 新增系统控制工具
- 新增写作助手 Agent

推荐接口：

- `Tool`
- `ToolRegistry`
- `Agent`
- `AgentRegistry`
- `ApprovalPolicy`

### 2.4 状态扩展点

适用场景：

- 新增本地数据库实现
- 新增快照机制
- 新增记忆抽取器

推荐接口：

```go
type MemoryExtractor interface {
    Name() string
    Extract(ctx context.Context, task Task, logs []LogEntry) ([]EpisodicMemory, error)
}

type SnapshotStore interface {
    SaveSnapshot(ctx context.Context, sessionID string, data []byte) error
    LoadSnapshot(ctx context.Context, sessionID string) ([]byte, error)
}
```

### 2.5 UI 扩展点

适用场景：

- 新增 TUI 面板
- 新增通知策略
- 新增确认对话框样式
- 新增任务历史视图

推荐接口：

```go
type View interface {
    ID() string
    Render(ctx context.Context, model any) error
}

type InteractionHandler interface {
    HandleUserAction(ctx context.Context, input any) error
}
```

## 3. 新增功能的标准流程

### 步骤 1：定义目标

例：新增“剪贴板代码自动解释”功能。

目标拆解：

- 感知剪贴板变化
- 判断是否是代码
- 识别意图：需要解释或建议优化
- 生成 Task
- 调用 coder Agent
- 用 UI 展示解释结果

### 步骤 2：定义事件

- `clipboard.changed`
- `intent.recognized`
- `task.created`
- `task.assigned`
- `task.execution.completed`
- `ui.message.display`

### 步骤 3：定义状态影响

- 更新 WorkingMemory 的 `SelectedContent`
- 创建新的 Task
- 追加 Log
- 生成 EpisodicMemory（如有价值）

### 步骤 4：确定扩展点

- 新增 Sensor：clipboard watcher
- 新增 IntentRecognizer 或分类规则
- 复用 coder Agent
- 复用 llm Tool
- 新增 UI 订阅者

### 步骤 5：注册组件

在启动流程中完成：

- 注册 Sensor
- 注册 Subscriber
- 注册 Tool / Agent
- 注册 Middleware

## 4. 具体扩展示例

---

### 4.1 示例：新增一个 Tool

目标：新增 `browser.search` 工具。

#### 第一步：实现接口

```go
type BrowserSearchTool struct{}

func (t *BrowserSearchTool) Name() string { return "browser.search" }
func (t *BrowserSearchTool) Description() string { return "搜索网页并返回结构化结果" }
func (t *BrowserSearchTool) Schema() any { return SearchSchema{} }
func (t *BrowserSearchTool) Metadata() ToolMetadata {
    return ToolMetadata{
        Category: "browser",
        RequiresApproval: false,
        Timeout: 10 * time.Second,
        Idempotent: true,
    }
}
func (t *BrowserSearchTool) Execute(ctx context.Context, input ToolInput) (ToolResult, error) {
    // 实现具体搜索逻辑
    return ToolResult{Success: true, Data: map[string]any{"items": []any{}}}, nil
}
```

#### 第二步：注册

```go
registry.Register(&BrowserSearchTool{})
```

#### 第三步：让 Agent 可发现

- 在 ToolRegistry 暴露元数据
- Agent 根据能力标签或工具名选择调用

---

### 4.2 示例：新增一个 Agent

目标：新增 `reviewer` Agent。

#### 第一步：实现接口

```go
type ReviewerAgent struct{}

func (a *ReviewerAgent) Name() string { return "reviewer" }
func (a *ReviewerAgent) Capabilities() []string { return []string{"review", "critique", "quality_check"} }
func (a *ReviewerAgent) CanHandle(task Task) bool {
    return task.Kind == TaskKindWriting || task.Kind == TaskKindCoding
}
func (a *ReviewerAgent) Execute(ctx context.Context, req AgentRequest) (AgentResponse, error) {
    return AgentResponse{
        Summary: "review complete",
        NextAction: "tool_call",
        ToolCalls: []PlannedToolCall{},
        Output: map[string]any{"verdict": "pass"},
    }, nil
}
```

#### 第二步：注册

```go
agentRegistry.Register(&ReviewerAgent{})
```

#### 第三步：更新路由规则

让 Router 在适当任务类型下返回 `reviewer`。

---

### 4.3 示例：新增一个 Middleware

目标：新增去重中间件。

```go
type DedupMiddleware struct{}

func (m *DedupMiddleware) ID() string { return "dedup" }
func (m *DedupMiddleware) Handle(ctx context.Context, event Event, next func(context.Context, Event) error) error {
    // 判断事件是否重复
    return next(ctx, event)
}
```

注册：

```go
bus.Use(&DedupMiddleware{})
```

---

### 4.4 示例：新增一个感知模块

目标：新增 `selection` 选中文本监听。

#### 接入步骤

1. 实现 `Sensor`
2. 监听 OS 选中文本变化
3. 识别内容类型
4. 发布 `selection.changed`
5. 由认知层决定是否生成 Task

#### 设计要求

- 感知层只负责采集与发布，不直接决策
- 去重、节流可由中间件或感知层本地辅助处理

---

### 4.5 示例：新增一个记忆提炼器

目标：从成功任务中提炼 ProceduralMemory。

#### 接入点

- 订阅 `task.execution.completed`
- 获取 Task 与关联日志
- 判断是否属于高复用流程
- 写入 `ProceduralMemory`

#### 注意事项

- 不要在主执行链路上阻塞
- 推荐走 WorkerPool 异步执行

## 5. 推荐的插件式注册方式

### 5.1 启动注册器

建议在启动流程中用统一注册器初始化：

```go
type Bootstrapper struct {
    Bus           Bus
    ToolRegistry  ToolRegistry
    AgentRegistry AgentRegistry
}

func (b *Bootstrapper) RegisterAll() error {
    // 注册中间件
    // 注册 Tool
    // 注册 Agent
    // 注册 Subscriber
    // 注册 Sensor
    return nil
}
```

### 5.2 每类组件独立注册包

例如：

```text
internal/bootstrap/
├── middleware.go
├── tools.go
├── agents.go
├── subscribers.go
└── sensors.go
```

这样新增功能时，主要是：

- 写新组件
- 在 bootstrap 中注册

## 6. 对外暴露的“能力发现”机制

为了让未来技能包、插件、外部扩展都能接入，建议保留能力描述对象：

```go
type CapabilityDescriptor struct {
    Name        string
    Category    string
    Description string
    InputSchema any
    OutputSchema any
    Version     string
    Tags        []string
}
```

Tool、Agent、Skill 都可暴露类似 descriptor，用于：

- 动态发现
- UI 列表展示
- 自动路由
- 权限与审批判断

## 7. 审批扩展规则

### 7.1 ApprovalPolicy 接口

```go
type ApprovalPolicy interface {
    ShouldRequireApproval(ctx context.Context, actionType string, args map[string]any) (bool, string)
}
```

### 7.2 建议策略

以下操作默认要求审批：

- 删除、覆盖、移动大量文件
- 执行 shell 命令
- 访问外部网络
- 修改系统配置
- 控制窗口或输入设备
- 读取敏感路径

### 7.3 扩展方式

新增高风险动作时，不改主逻辑，只需要：

- 更新 ApprovalPolicy
- 定义 ActionType
- 在执行前统一检查

## 8. 扩展时的边界纪律

### 应该做

- 新增正式对象
- 新增接口实现
- 新增事件类型与 Payload
- 新增订阅者
- 新增注册逻辑

### 不应该做

- 在总线里塞特定业务判断
- 在 UI 层直接改状态对象
- 在 Tool 中直接访问全局 Session map
- 为了图快，把所有东西塞进 `map[string]any`
- 新功能直接串接到多个具体模块内部

---

## 9. 推荐的开发清单模板

以后每新增一个功能，都建议写一份简短设计单，格式如下：

```markdown
# 功能名

## 目标

## 用户触发方式

## 感知事件

## 认知处理

## Task/Loop 变化

## 是否需要 Approval

## Tool / Agent 依赖

## UI 输出

## Memory 沉淀

## 失败与恢复策略
```

---

## 10. 最终扩展哲学

未来的功能开发应该遵循这条路线：

> 新能力 = 新接口实现 + 新事件 + 新注册，而不是重写主架构。

只要状态模型稳定、事件协议稳定、扩展接口稳定，CialloClaw 就可以在当前架构上持续生长，而不会越改越乱。
