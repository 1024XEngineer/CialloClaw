# CialloClaw Chat Panel And Avatar Floating Ball Design

## 1. Goal

Extend the current desktop-shell MVP with two focused UI upgrades:

- replace the floating ball letter button with an avatar image based on the provided character art
- add a persistent assistant chat area inside the main panel

The target is still a minimal prototype. The goal is to make the desktop shell feel more like an assistant product without introducing real model integration or complex conversation systems.

## 2. Scope

### In scope

- Replace the floating ball text button with a local image asset
- Style the floating avatar so it looks clean and intentional in the desktop floating window
- Restructure the main panel into two columns:
  - left work area
  - right chat area
- Keep the left work area responsible for:
  - task inspection
  - summary and draft
  - execution log
- Add a right-side chat area with:
  - message list
  - text input
  - send button
  - lightweight quick actions
- Add local mock chat behavior so the panel shows real message exchange when the user types or clicks quick actions

### Out of scope

- Real model or API integration
- multi-turn semantic memory
- file upload in chat
- rich-text or markdown message rendering
- animated character behavior
- speech or audio
- complex deep linkage where every chat action rewrites the left workflow automatically

## 3. Product Shape

### Floating avatar

- The floating renderer remains a small separate Electron window
- The button is visually replaced with the provided avatar image
- The avatar remains the desktop entry point for opening and hiding the main panel
- The click behavior stays unchanged from the existing desktop-shell MVP

### Main panel

- The main panel becomes a two-column assistant workspace
- Left side remains the current operational workflow
- Right side becomes a persistent chat assistant surface

This keeps the existing working prototype intact while making the panel feel more conversational.

## 4. Layout

### Left work area

- Task Inspection
- Summary and Draft
- Execution Log

These sections remain functionally unchanged and continue to display the current prototype workflow.

### Right chat area

- A header that identifies the assistant panel
- A scrollable message history region
- A small group of quick-action buttons such as:
  - summarize
  - explain
  - extract todos
- A text input field
- A send button

The right chat area should remain visible at all times while the main panel is open.

## 5. User Flow

1. User clicks the floating avatar
2. Main panel opens
3. User sees the existing workflow on the left and the chat assistant on the right
4. Chat area starts with a welcome assistant message
5. User either:
   - types a message and clicks send
   - clicks a quick-action button
6. The panel appends a user message and then an assistant response
7. The left-side workflow continues to work as before

## 6. Chat Behavior

The chat area uses simple local response rules only.

### Message model

Each message should minimally include:

- role (`user` or `assistant`)
- text

### Initial state

The chat area starts with one assistant welcome message, for example:

- “I can help summarize the current task state, explain the workflow, or extract next steps.”

Chat history may remain in memory while the panel stays alive during the current app session. Persistent storage across app restarts is out of scope.

### Send behavior

- When the user sends text, append a user message immediately
- Then append a mock assistant reply generated by local rules

### Quick actions

- Clicking a quick action behaves like sending a predefined prompt
- Each quick action should append both a user-style prompt message and an assistant response message
- Quick actions should produce visibly different responses so the feature feels real

### Suggested local rules

- If the prompt contains “summarize”, return a summary-oriented response that may reference the current inspection summary if available
- If the prompt contains “todo” or “task”, return a task-extraction style response based on the demo task list
- If the prompt contains “explain”, return a short explanation of the current prototype workflow
- Otherwise return a generic helper response

## 7. Avatar Asset Handling

- Store the provided character image as a local asset in the project, for example under `assets/`
- Use a concrete local filename during implementation, such as `assets/floating-avatar.png`, so the renderer can reference a stable path
- Load it from `floating.html`
- Style it to avoid the current plain text button look
- The floating window background should remain transparent
- The visible avatar button should feel like a character token rather than a browser button

For MVP, if the source image includes a background, the implementation may still use it as-is; background cleanup is optional and should not block the feature.

## 8. File Impact

Expected touched files:

- `assets/...` - local avatar image file
- `floating.html` - replace text button content with avatar image markup
- `floating.css` - avatar styling
- `floating.js` - keep click-to-toggle behavior, adjust only if needed for updated markup
- `index.html` - add right-side chat structure
- `style.css` - add two-column workspace and chat styles
- `app.js` - add chat state and local message-response behavior

## 9. Behavior Constraints

- The avatar floating button must still toggle the main panel reliably
- The chat area must not break the existing left-side workflow
- The existing inspection, draft, log, and undo flow must continue to work
- The chat behavior must remain local and deterministic enough for testing

## 10. Verification Criteria

The feature is successful if the following are true:

1. The floating renderer shows the avatar image instead of a letter button
2. Clicking the avatar still opens and hides the main panel
3. The main panel shows a right-side chat area
4. The chat area starts with a welcome assistant message
5. Sending a typed message appends both a user message and an assistant message
6. Clicking a quick-action button appends both a user-style prompt message and an assistant message
7. The existing left-side workflow still works:
   - inspection
   - draft confirmation
   - draft generation
   - log updates
   - undo

## 11. Why This Design

This is the smallest useful upgrade from the current desktop-shell MVP:

- the avatar makes the floating entry point feel character-driven instead of placeholder-like
- the right-side chat area gives the panel a real assistant surface
- the existing workflow remains intact, avoiding unnecessary rework
- local mock responses keep the feature demonstrable without introducing model dependencies
