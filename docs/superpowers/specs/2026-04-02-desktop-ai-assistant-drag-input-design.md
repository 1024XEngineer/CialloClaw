# 桌面端 AI 助手前端原型设计

## 1. 项目概述

本原型用于展示一款桌面端 AI 助手的核心交互：用户将桌面对象直接拖拽到常驻悬浮球上，系统将对象视为输入源，并在识别对象类型后给出对应的快捷动作与结果展示。

原型仅覆盖前端高保真交互与界面表达，不接入后端逻辑、不依赖真实解析能力，所有内容均使用 mock 数据。

## 2. 设计目标

- 突出“对象即输入”，而不是聊天框输入
- 清晰展示桌面端拖拽到悬浮球的接收过程
- 完整覆盖从投递、识别、动作选择、处理中到结果查看的状态闭环
- 体现专业、克制、接近系统级桌面工具的视觉与动效气质
- 适用于产品与设计评审，便于快速理解核心价值与状态覆盖

## 3. 非目标

- 不实现真实文件解析、OCR、PDF 摘要或联网能力
- 不实现复杂对话流、历史会话管理、账户体系或设置系统
- 不将原型重点放在聊天窗口或大文本输入框上
- 不覆盖移动端布局与交互

## 4. 目标用户与评审语境

- 主要受众：产品/设计评审
- 文案语言：全中文
- 风格取向：中性系统风，不明显偏 Windows 或 macOS
- 评审重点：核心交互是否成立、状态定义是否清晰、视觉语言是否统一

## 5. 原型形式

采用“主桌面交互 + 右侧上下文栏”的单页高保真原型。

- 左侧 65%：桌面交互主舞台，演示真实对象被拖向悬浮球并触发处理
- 右侧 35%：上下文栏，默认显示状态画廊；打开详情时切换为详情视图

该结构同时满足两类需求：

- 主舞台负责讲清楚“怎么输入、怎么被接收、怎么得到结果”
- 上下文栏负责讲清楚“系统一共有哪些状态、每一态如何定义”，并在需要时承接详情阅读

为避免布局冲突，`结果卡片` 始终出现在主舞台内；`侧边详情态` 不与完整状态画廊并排共存，而是在点击 `展开详情` 后由右侧上下文栏从“状态画廊模式”切换到“详情模式”。

## 6. 核心设计原则

### 6.1 对象优先

界面不以聊天输入框作为中心入口。桌面中的文本、文件、图片、PDF、链接等对象本身就是输入载体，悬浮球是唯一主入口。

### 6.2 反馈分层明确

每一个状态都只回答一个问题：

- 能不能投
- 是否正在接收
- 系统识别成什么
- 可以做什么
- 正在处理什么
- 结果在哪里看

### 6.3 系统工具气质

视觉与动效都要表现为可靠、克制、专业，避免娱乐化 AI 产品常见的夸张发光、强人格化文案与聊天式叙事。

### 6.4 结果结构化

处理结果优先以结果卡片与侧边详情承载，而不是长聊天记录。

## 7. 页面信息架构

### 7.1 桌面背景区

模拟系统桌面工作区，使用轻渐变与轻模糊层构建真实环境。背景存在感要弱，目的是承托对象与悬浮球。

### 7.2 拖拽对象层

桌面上放置 5 类样本对象：

- PDF 文档卡片
- 图片缩略图
- 文本便签
- 链接卡片
- 通用文件卡片（默认作为不支持格式演示对象，例如 ZIP）

这些对象均可拖拽，用于演示“对象即输入”。主舞台左上角设置 `示例对象托盘` 作为固定切换入口，用户可从托盘直接拖出对象，或点击对象标签切换当前演示样本。

点击对象标签仅切换“当前待拖拽样本”，不自动播放识别、处理与结果动画；完整闭环仅在实际拖拽或点击 `一键演示` 控件时触发。

### 7.2.1 原型中的桌面拖拽模拟方式

原型不接入真实 OS 级拖放事件，而是通过页内拖拽 + 系统风格拖拽预览进行模拟：

- 从 `示例对象托盘` 按下对象后，生成跟随鼠标的拖拽代理卡片
- 代理卡片包含对象缩略信息、文件名/类型标签与轻阴影
- 鼠标指针与拖拽代理的组合视觉尽量接近桌面拖拽感受
- 当对象进入悬浮球接收范围时，触发接收高亮、吸附与状态同步

这样既能在前端原型内稳定实现，也能明确传达“未来目标是接收桌面对象，而非网页内部表单输入”。

### 7.3 悬浮球组件

悬浮球固定在桌面右下偏中位置，常驻桌面，是全局唯一主入口。默认尺寸建议为 56-64px。

### 7.4 动作选择面板

对象释放后从悬浮球右侧展开。该面板承载对象识别结果、建议动作与当前上下文。

### 7.5 结果卡片区

对象处理完成后，优先在悬浮球上方偏左的主舞台区域出现结构化结果卡片，用于快速查看结果。结果卡片不占用右侧上下文栏，以避免与状态画廊或详情视图发生冲突。

### 7.6 侧边详情栏

用户点开结果卡片后，右侧上下文栏从“状态画廊模式”切换到“详情模式”，展示完整结果、原对象信息与后续动作。详情关闭后恢复到状态画廊模式，并保留当前高亮状态。

### 7.7 状态画廊

右侧在默认状态下纵向排列状态缩略卡，按流程顺序展示完整状态链路，每张卡片附 1 句说明。当前主舞台所处状态需在画廊中同步高亮。

## 8. 主舞台推荐演示路径

默认演示路径为：

1. 用户从桌面拖拽 `产品方案.pdf`
2. 对象靠近悬浮球，触发接收高亮
3. 悬停时悬浮球放大并提示“松手处理”
4. 松手后进入对象识别态
5. 展开动作选择面板，默认显示 `总结 PDF`、`提取重点`、`生成问答`、`整理内容`
6. 用户选择 `总结 PDF`
7. 系统进入处理中态
8. 输出 `PDF 总结` 结果卡片
9. 点击 `展开详情`，打开右侧详情栏

其它对象类型通过 `示例对象托盘` 切换补充演示。切换对象时，主舞台重置到默认态，右侧状态画廊回到第一张状态卡，并更新对应动作配置。

## 9. 关键状态定义

### 9.1 悬浮球默认态

- 球体为深灰金属感底色，带极轻冷白内发光
- 存在轻微呼吸感，但不主动打扰
- 不出现聊天暗示，不显示“问我任何问题”类文案

### 9.2 对象靠近时的接收高亮态

- 当拖拽对象进入悬浮球附近接收范围，触发接收场
- 反馈顺序为：外环点亮 -> 球体轻微放大 -> 对象轻微吸附
- 桌面其余区域轻微降噪，帮助用户识别接收目标

### 9.3 拖拽悬停态

- 球体放大至约 1.18x-1.24x
- 中心文案显示“松手处理”或“释放以识别”
- 被拖拽对象自动居中并获得轻微高亮，形成“被接住”的感受

### 9.4 放手后的对象识别态

- 球体短暂收束，进入 600-900ms 的识别过渡
- 展示对象类型标签，如 `PDF`、`图片`、`文本`、`链接`、`文件`
- 展示对象摘要，如文件名、页数、图片尺寸或文本前两行
- 核心目标是让用户确认系统理解正确

### 9.5 动作选择面板

- 面板从悬浮球右侧展开，保持紧凑与系统感
- 顶部显示已识别对象信息，如 `已识别：产品方案.pdf`、`12 页 PDF`
- 下方展示 4-6 个与对象类型强相关的动作
- 每个动作命名采用“动词 + 对象/结果”结构

### 9.6 处理中态

- 悬浮球中心切换为环形扫描或脉冲进度反馈
- 面板保留当前对象与所选动作，明确系统正在做什么
- 文案示例：`正在总结 PDF`、`正在识别图片文字`
- 可展示 2-3 档骨架屏，模拟逐步产出感

### 9.7 结果卡片态

- 结果第一落点为结构化卡片，而非聊天记录
- 标题直接对应动作结果，如 `PDF 总结`、`图片文字提取`
- 卡片内容根据动作类型切换模板，而不是所有对象共用单一结构
- 卡片底部保留 `展开详情`、`复制结果`、`继续处理` 等轻操作
- `继续处理` 的行为定义为重新展开动作面板，并将当前对象与结果上下文带回面板中，不进入自由聊天流

结果卡片模板建议如下：

- 摘要/提炼类：3 条重点 + 1 段摘要 + 2 个后续动作
- OCR 类：识别文本预览 + 关键信息块 + 2 个后续动作
- 翻译类：原文/译文对照 + 语言标签 + 2 个后续动作
- 生成问答类：3 组示例问答 + 2 个后续动作

### 9.8 侧边详情态

- 右侧上下文栏切换为固定宽度详情视图
- 上部展示原对象信息
- 中部展示完整处理结果
- 下部展示继续动作，如二次整理、生成问答、复制导出
- 所有继续动作仅回到既定动作面板或切换到下一个 mock 结果态，不扩展为开放式聊天窗口

### 9.9 错误与不支持格式提示

- 不支持对象时，应尽量在靠近或悬停阶段就给出预警
- 悬浮球切换为克制的警示态，不进入成功接收高亮
- `格式不支持` 与 `识别失败` 分开定义
- `格式不支持` 提供可用格式提示
- `识别失败` 提供 `重试`、`查看原因` 等恢复动作

## 10. 对象类型与快捷动作映射

### 10.1 PDF

- 总结 PDF
- 提取重点
- 生成问答
- 整理内容

### 10.2 图片

- OCR 图片
- 提取重点
- 整理内容

### 10.3 文本

- 翻译文本
- 提取重点
- 整理内容

### 10.4 链接

- 提炼网页要点
- 生成摘要
- 整理内容

### 10.5 不支持对象

- 暂不支持该格式
- `通用文件卡片` 默认映射到此状态，用于演示不支持格式提示
- 提示用户尝试拖入 PDF、图片、文本或链接

## 11. 视觉规范

### 11.1 色彩

- 主色：石墨灰、雾白、冷灰蓝
- 强调色：低饱和冷白高亮
- 警示色：低饱和琥珀或灰红
- 避免高纯度品牌色与过强霓虹发光

### 11.2 组件质感

- 悬浮球：深灰金属核 + 轻内发光 + 细外环
- 面板与卡片：中等圆角、细描边、轻阴影
- 背景：弱存在感渐变，不使用纯平单色背景

### 11.3 排版

- 中文无衬线系统风排版
- 标题克制，正文紧凑
- 文案短句优先，避免长段解释

## 12. 动效规范

- 靠近反馈：120-160ms
- 吸附与放大：180-220ms
- 识别过渡：600-900ms
- 结果展开：200-260ms

动效应以平稳、清晰为主，避免夸张 bounce、旋转或娱乐化轨迹。

## 13. 文案原则

- 所有文案围绕对象与动作本身
- 避免过强人格化表达
- 推荐使用：`识别为 PDF`、`松手处理`、`正在提取重点`、`展开详情`
- 避免使用：`我来帮你看看`、`你可以问我任何问题`

## 14. 原型实现范围

### 14.1 需要实现

- 单页高保真桌面原型
- 主舞台拖拽演示流程
- `示例对象托盘` 与对象切换
- 右侧上下文栏的两种模式：状态画廊、详情视图
- 对象类型切换
- 动作面板、处理中态、结果卡片、详情栏、错误提示的 UI 与状态切换

### 14.2 不需要实现

- 真正的文件拖放系统接入，仅模拟 OS 风格拖拽体验
- 真正的 OCR、PDF 解析、联网抓取与翻译
- 后端接口、权限、用户系统、持久化

## 15. 评审验收标准

- 评审者在第一眼能理解“对象拖到悬浮球就是输入”
- 评审者能在主舞台中完整看到一次闭环演示
- 评审者能在状态画廊中快速检查所有关键状态
- 整体视觉符合系统级桌面工具气质，而非聊天机器人界面
- 对象识别、动作选择、结果展示三层结构清晰，不相互混淆

## 16. 后续实现建议

- 优先实现 PDF 演示主链路，作为默认场景
- 再补充图片、文本、链接的快捷动作差异
- 最后补充错误与不支持格式提示，完成闭环
