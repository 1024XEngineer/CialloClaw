# 产品路演大纲

## 定位

- 这份 PPT 只讲产品。
- 团队分工和系统架构放在另一份 PPT 里。
- 目标观众：非专业但有一定技术理解的观众。
- 建议时长：8-10 分钟。

## 讲述主线

`背景 -> 目标用户 -> 用户问题 -> 产品定位 -> 产品形态 -> 核心能力 -> 主场景展开 -> 信任机制 -> 产品完整度 -> 产品价值 -> 收尾`

## 第 1 页：封面

- 标题建议：`CialloClaw：让 AI 从回答问题走向完成任务`
- 内容：
  - 产品名
  - 一句话定义
  - 一张主视觉图
- 目标：
  - 让观众一开始就明白，这是一个帮助用户完成任务的产品。
- 建议时长：30 秒

## 第 2 页：产品背景

- 标题建议：`AI 很强，但“完成任务”仍然是空白地带`
- 内容：
  - 现在大家已经习惯用 AI 提问、找资料、生成内容。
  - 但很多真实工作并不会在一个回答之后结束。
  - 用户真正需要的是，从提出需求，到执行过程，再到拿到结果的完整闭环。
- 核心信息：
  - 今天的大多数 AI 产品更擅长回答，对任务完成的承接还不够完整。
- 建议时长：45-60 秒

## 第 3 页：目标用户

- 标题建议：`CialloClaw 的目标用户`
- 内容：
  - 我们的目标用户：
    - 经常在电脑上处理网页、文件、信息和临时任务的人。
  - 他们通常有这几个特点：
    - 手头一直有具体事情要处理。
    - 希望 Agent 能直接接住当前工作。
    - 希望任务能继续推进，不停在一轮回答里。
    - 希望结果能保存、打开、继续使用。
    - 希望高风险操作先确认。
- 核心信息：
  - 我们服务的是正在电脑前处理事情、并希望 Agent 真正接住工作的用户。
- 建议时长：45-60 秒

## 第 4 页：用户问题

- 标题建议：`从“得到回答”到“真的做完”，中间还有很多断点`
- 内容：
  - 发起断点：用户看到问题时，往往还要切到聊天框，重新整理上下文，才能把需求交给 AI。
  - 推进断点：很多请求停在一轮回答里，缺少继续推进、持续追踪和查看进度的能力。
  - 交付断点：结果出来之后，用户还要自己复制、保存、查找和整理，正式承接不够顺。
  - 信任断点：一旦涉及文件、命令、网页等操作，用户会担心执行边界，也缺少清晰确认。
- 呈现形式：
  - 四张卡片，分别对应后面要展开的四个可演示能力。
- 核心信息：
  - 真正的问题不在于 AI 会不会回答，而在于从需求出现到结果落地，中间存在很多没有被接住的断点。
- 过渡句：
  - CialloClaw 想解决的，就是这四类断点。
- 建议时长：50-60 秒

## 第 5 页：产品定位

- 标题建议：`CialloClaw：面向任务完成的桌面 Agent 产品`
- 内容：
  - 一句话定位：
    - `CialloClaw 是一个直接在桌面现场协作、并能形成正式交付结果的本地 Agent 产品。`
  - 三个关键词：
    - 桌面协作
    - 任务推进
    - 正式交付
  - 对四类断点的回答：
    - 发起断点：入口回到桌面现场，用户可以就地发起需求。
    - 推进断点：请求会进入任务链路，并持续推进。
    - 交付断点：结果会进入正式结果与产物承接链路。
    - 信任断点：风险动作会进入确认、授权和审计链路。
  - 对比口径建议：
    - 和常见 AI 产品相比，CialloClaw 把重点从“回答问题”推进到了“交付结果”。
    - 和常见 Agent 产品相比，CialloClaw 更强调直接在桌面现场协作，把执行过程和交付结果承接进产品体验里。
    - `OpenClaw` 更偏开放能力和执行空间，`CialloClaw` 更偏桌面协作、任务推进和结果交付。
- 核心信息：
  - CialloClaw 的核心定位，是把 Agent 放进桌面现场，围绕任务推进，并最终交付结果。
- 建议时长：40-50 秒

## 第 6 页：产品形态

- 标题建议：`一个已经具备完整使用路径的桌面产品体验`
- 内容：
  - 从用户视角展示产品的主要界面：
    - 悬浮球入口
    - 近场输入与气泡承接
    - 工作台中的任务与结果页
  - 如果可以，优先用一张整体界面图，减少零散截图。
- 核心信息：
  - 这个产品把桌面现场、任务过程和结果交付组织成一个清晰可用的体验。
- 建议时长：50-60 秒

## 第 7 页：核心能力总览

- 标题建议：`当前产品已经具备的 4 个核心能力`
- 内容：
  - 就地发起
    - 用户不用切到聊天框重新组织上下文，可以在当前工作现场直接发起需求。
  - 围绕任务推进
    - 系统不会把请求留在一轮对话里，而是把它变成可追踪任务。
  - 结果正式交付
    - 结果不会停留在一句回答，而会进入正式结果与产物承接链路。
  - 风险可见、可控
    - 涉及高风险动作时，系统会进入授权与确认链路，确保过程可见可控。
- 说明：
  - 这四个能力都能在后面几页里用真实场景演示出来。
- 核心信息：
  - 这四个能力已经足以支撑一条完整的产品主链路，也足以支撑一场完整路演。
- 建议时长：45-60 秒

## 第 8 页：主场景一，就地发起

- 标题建议：`用户不用先打开 Agent，再重新描述问题`
- 内容：
  - 用户痛点：
    - 我不想切到聊天框里，再重新解释我看到的问题。
  - 典型场景：
    - 在网页上看到一段内容，想快速总结、解释或翻译。
    - 在本地文件里看到一段内容，想让 Agent 直接围绕这段内容工作。
    - 手上有一份文件，想直接拖给 Agent 处理。
    - 想直接说一句话，不想打一大段提示词。
  - 当前已有能力：
    - 悬浮球作为常驻入口。
    - 悬停后可直接输入一句需求。
    - 长按可语音输入。
    - 选中文本后可直接围绕选中内容发起任务。
    - 文件拖拽到悬浮球后可进入附件队列，再正式发送。
    - 双击可打开工作台，不必走传统聊天页。
  - 现场演示说法：
    - `你不用先打开 Agent，再复制上下文给它；你就在当前工作现场，把问题扔给它就行。`
  - 适合演示的动作：
    - 选中网页或文档中的一段文字。
    - 点击或悬停悬浮球。
    - 发起“帮我总结 / 翻译 / 解释”。
    - 展示气泡或任务页返回结果。
- 核心信息：
  - 产品的第一步，是让 Agent 进入用户正在工作的现场。
- 建议时长：60 秒

## 第 9 页：主场景二，围绕任务推进

- 标题建议：`用户不只要一轮回答，而是希望它真的变成任务`
- 内容：
  - 用户痛点：
    - 我不想问一句就结束，我希望系统把这件事接过去，继续追踪。
  - 典型场景：
    - 我想知道它现在做到哪一步了，有没有卡住。
    - 我希望这件事能从近场输入一路进入正式任务链，持续推进下去。
  - 当前已有能力：
    - 系统对外主对象是任务，所有动作都围绕任务展开。
    - 已有任务列表、任务详情、任务状态页。
    - 任务详情里能看到头部、基础时间线、成果区。
    - 可以从近场输入一路进入正式任务链。
    - 可以在工作台查看当前任务与历史任务。
  - 现场演示说法：
    - `我们把用户的现场问题转成一个可追踪任务。`
  - 适合演示的动作：
    - 从悬浮球发起一个请求。
    - 结果会继续进入任务与结果承接链路。
    - 双击打开工作台。
    - 展示该任务已经进入任务列表，详情可追踪。
- 核心信息：
  - 这是一个把问题接成任务、把任务持续推进的产品。
- 建议时长：60-75 秒

## 第 10 页：主场景三，正式结果交付

- 标题建议：`很多 AI 能答，但结果不好接，也不好用`
- 内容：
  - 用户痛点：
    - 我拿到结果后，还得自己复制、保存、找文件。
    - 长结果不知道去哪了。
    - 网页结果、文档结果、文件结果常常混在一起。
  - 当前已有能力：
    - 短结果可以直接在气泡中承接。
    - 长结果已能走正式结果与产物承接链路。
    - 已支持至少一类正式产物的打开与定位。
    - 浏览器结果和网页结果已能转到工作台内结果页承接。
    - 工作台内已有结果页和交付页基础能力。
  - 现场演示说法：
    - `回答之后，结果会进入正式交付出口，可以打开、查看、继续处理。`
  - 适合演示的动作：
    - 发起一个会产生较长结果的请求。
    - 先看气泡中的短反馈。
    - 再进入工作台的任务详情或结果页。
    - 展示正式交付、产物入口和打开动作。
- 核心信息：
  - 结果会成为一个可以被正式承接、继续使用的交付出口。
- 建议时长：60-75 秒

## 第 11 页：增强信任，风险可见、可控

- 标题建议：`Agent 一旦涉及操作，用户最在意的是它会不会乱来`
- 内容：
  - 用户痛点：
    - 它可能会改文件、执行命令、访问网页。
    - 我希望高风险操作先问我，不要自己做。
    - 我也想知道它为什么停住、要我确认什么。
  - 当前已有能力：
    - 已有风险分级与授权链路。
    - 已有待授权、审批、审计、恢复点这些对象基础。
    - 前端已有安全摘要、安全页、待授权承接的基础展示。
    - 工作台中能看到安全相关信息。
  - 现场演示说法：
    - `我们把风险动作纳入授权链。`
  - 适合演示的动作：
    - 发起一个可能触发高风险动作的请求。
    - 展示它会先进入等待授权或安全提示。
    - 在工作台或气泡中展示待确认信息。
  - 备注：
    - 这条链建议作为增强信任的第二段演示，不建议作为第一段主秀。
- 核心信息：
  - 风险动作会进入用户可见、可控、可确认的处理链路。
- 建议时长：45-60 秒

## 第 12 页：产品完整度补充

- 标题建议：`这个产品不只接一次请求，也在承接长期事项与日常使用体验`
- 内容：
  - 长期事项与记忆：
    - 便签页已存在真实便签和事项视图基础。
    - 支持把事项升级为正式任务。
    - 镜像页已有总览、日报和历史概要方向的基础展示。
    - 已有记忆命中和摘要的前端承接骨架。
  - 设置与首次引导：
    - 控制面板已有真实设置读取、保存、回显。
    - 包含通用、悬浮球、记忆、模型等正式设置。
    - 首次引导已具备首次进入与手动重放基础。
  - 现场演示说法：
    - `我们已经把长期事项、核心设置和首次引导做进桌面产品里。`
- 核心信息：
  - 产品在主链路之外，也在逐步具备长期使用所需要的承接能力和产品完成度。
- 建议时长：45-60 秒

## 第 13 页：产品价值

- 标题建议：`为什么用户会选择 CialloClaw`
- 内容：
  - 它让 Agent 直接进入桌面现场，接住用户正在处理的问题。
  - 它把一次请求接成任务，并持续推进到结果。
  - 它把产品重点从回答问题推进到交付结果。
  - 它把风险动作放进确认和授权链路，让协作过程更可控。
  - 它把桌面协作、任务推进和结果交付连成一条完整体验。
- 收束句建议：
  - `我们希望把 Agent 从“能完成任务”推进到“能在桌面现场协作，并真正交付结果”。`
- 核心信息：
  - 这个产品的价值，在于让 Agent 在真实工作里更贴近现场、更能推进任务、更能交付结果。
- 建议时长：45-60 秒

## 第 14 页：结尾 / 问答

- 标题建议：`让 Agent 进入桌面现场，真正交付结果`
- 内容：
  - 一句话总结

  - 问答
  - 可选：项目链接或二维码
- 核心信息：
  - 用一个清晰记忆点收尾，不再引入新的信息。
- 建议时长：30 秒

## 视觉方向

- 整体保持视觉化、产品化，避免做成文字很多的说明文档。
- 使用大标题、短文案、清晰的卡片式布局。
- 每页只传达一个最核心的信息。
- 推荐页面类型：
  - 背景 / 问题：标题 + 三张卡片
  - 用户 / 定位：少字 + 关键词
  - 产品形态：大幅界面图
  - 核心流程：简洁流程图
  - 能力 / 场景 / 价值：卡片式信息图
