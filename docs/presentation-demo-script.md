# 路演演示脚本

## 目的

- 这份文档用于记录现场演示要做什么、按什么顺序做、每一步想让观众看到什么。
- 它和 `docs/presentation.md` 分开使用：
  - `presentation.md` 负责 PPT 叙事。
  - `presentation-demo-script.md` 负责现场操作脚本。

## 演示主线

这次演示建议分成两层：

- 基础能力：证明这是一个已经能跑通主链路的桌面 Agent 产品。
- 产品特色：证明它能在桌面现场协作、推进任务、交付结果。

建议现场顺序：

1. 先演基础能力
2. 再演特色能力
3. 最后用工作台和镜像页收尾

## 一、基础能力演示

### 1. 文字 / 语音对话入口

- 演示目标：
  - 让观众先看到最基础的入口已经可用。
- 现场操作：
  - 展示悬浮球常驻。
  - 悬停后输入一句简单需求。
  - 再补一个长按语音输入的动作。
- 建议讲法：
  - `这是最基础的一层，用户可以直接在桌面现场通过文字或语音发起需求。`
- 观众应当看到：
  - 悬浮球可以作为常驻入口。
  - 文字和语音都能发起请求。

### 2. 打开工作台查看详情

- 演示目标：
  - 让观众理解这套产品已经有完整承接界面。
- 现场操作：
  - 双击打开工作台。
  - 依次展示：
    - 首页信息球
    - 任务页
    - 便签页
    - 镜像页
    - 安全页
- 建议讲法：
  - `请求发出之后，用户还可以回到工作台查看任务、结果和长期状态。`
- 观众应当看到：
  - 工作台是完整承接界面。
  - 系统已经有任务、便签、镜像、安全这些基础页面。

## 二、产品特色演示

### 3. 屏幕感知与就地发起

- 演示目标：
  - 体现产品直接进入桌面现场协作，减少用户切换到聊天页重新描述上下文的成本。
- 现场操作：
  - 在网页或文档里选中一段文字。
  - 直接通过悬浮球发起：总结 / 翻译 / 解释。
  - 展示结果先在气泡中承接。
- 建议讲法：
  - `用户看到哪里、卡在哪里，就可以直接围绕当前内容发起任务。`
  - `这里的重点是 Agent 已经进入了用户当前的工作现场。`
- 观众应当看到：
  - 选中文本后可以就地发起任务。
  - 用户不需要重新复制整理上下文。

### 4. 任务推进与任务页承接

- 演示目标：
  - 体现请求会进入任务链路，并继续往下推进。
- 现场操作：
  - 从悬浮球发起一个请求。
  - 进入工作台任务页。
  - 展示任务详情中的头部、时间线、成果区。
  - 可以补一句：在任务页里还能看到和该任务相关的信息。
- 建议讲法：
  - `我们希望把现场问题接成一个任务，让它能继续推进、能回看、能追踪。`
- 观众应当看到：
  - 请求已经进入任务列表。
  - 任务详情可以承接过程和结果。

### 5. 主动协助

- 演示目标：
  - 体现 Agent 除了接住请求，也会在合适的时机给出帮助。
- 现场操作：
  - 展示一段屏幕感知到的场景。
  - 演示系统主动给出帮助机会。
  - 可以配合便签事项或当前任务上下文来展示。
- 建议讲法：
  - `除了接住用户主动发出的请求，系统也会在合适的时机给出协助机会。`
- 观众应当看到：
  - 产品具备主动协助的方向。
  - 协助是围绕当前桌面现场产生的。

### 6. Agent 主动寻找资料

- 演示目标：
  - 体现 Agent 可以继续推进任务，并在需要时主动进行操作。
- 现场操作：
  - 发起一个需要查找资料的请求。
  - 让 Agent 继续往下做，尝试主动寻找相关资料。
  - 如果触发权限或风险确认，顺势进入下一段安全演示。
- 建议讲法：
  - `当任务需要继续往前走时，Agent 会继续把需要的信息补齐。`
- 观众应当看到：
  - Agent 会沿着任务继续推进。
  - 主动动作和后续风险控制是连在一起的。

### 7. 风险提示与权限触发

- 演示目标：
  - 体现产品在执行动作时是可见、可控的。
- 现场操作：
  - 使用上一段主动寻找资料或其他可能触发风险的请求。
  - 展示系统进入待授权或安全提示。
  - 在工作台或气泡中展示确认信息。
- 建议讲法：
  - `我们把风险动作纳入确认、授权和审计链路。`
- 观众应当看到：
  - 风险动作不会直接越过用户。
  - 安全机制是产品主链路的一部分。

## 三、收尾展示

### 8. 镜像页与用户画像

- 演示目标：
  - 让观众感受到系统除了接住当前请求，也在逐步承接长期工作背景。
- 现场操作：
  - 打开镜像页。
  - 展示总览、日报或历史概要。
  - 如果现场稳定，可以提一下用户画像或长期记忆方向。
- 建议讲法：
  - `系统除了接住这一次请求，也在逐步理解用户长期的事项和工作背景。`
- 观众应当看到：
  - 产品有长期承接能力。
  - 它可以继续服务长期事项和后续工作。

## 推荐演示顺序

如果时间完整，推荐按下面顺序演：

1. 悬浮球文字输入
2. 悬浮球语音输入
3. 打开工作台
4. 选中文本后发起总结 / 翻译 / 解释
5. 进入任务页看详情
6. 展示主动协助
7. 展示 Agent 主动寻找资料
8. 展示待授权 / 安全提示
9. 打开镜像页收尾

## 压缩版顺序

如果现场时间只有 3-5 分钟，建议压缩成下面顺序：

1. 选中文本后就地发起
2. 气泡返回短结果
3. 双击打开工作台
4. 进入任务详情页
5. 展示一次正式结果承接
6. 视稳定性决定是否补安全提示

## 每段对应的产品能力

- 悬浮球文字 / 语音输入：就地发起
- 选中文本后发起任务：桌面现场协作
- 任务页承接：围绕任务推进
- 正式结果页 / 产物入口：正式交付结果
- 待授权 / 安全提示：风险可见、可控
- 镜像页：长期承接能力

## 现场注意事项

- 第一段主秀优先选择最稳的链路：选中文本 -> 发起任务 -> 气泡结果 -> 工作台任务详情。
- 风险触发链路适合作为增强项，不建议一上来就演。
- 主动协助如果现场不稳定，可以只讲概念，不强行演。
- 镜像页适合放在最后收尾，用来补产品完整度和长期价值。

## 建议准备

- 预先准备一段适合演示的网页文本。
- 预先准备一个本地文档或文件。
- 预先准备一个适合触发“主动找资料”或权限确认的问题。
- 确认工作台页面数据是可展示状态。
- 确认镜像页、安全页、任务页都有可讲内容。

## 一句话总结

- 这场演示的重点，不是把每个功能都点一遍。
- 这场演示的重点，是让观众看到一条完整链路：
  - Agent 进入桌面现场
  - 接住用户问题
  - 把请求推进成任务
  - 在需要时触发确认
  - 最终形成正式结果交付
