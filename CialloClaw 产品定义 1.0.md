# 1. 背景

当前最容易试用 Agent 的人群主要有两类：一类是**信息和文档密集型岗位**，例如咨询、研究、内容、产品；另一类是**流程和交付密集型岗位**，例如运营、销售支持、客服、行政和数据运营。

## 1.1 核心目标用户定义

本产品的核心目标用户，是一类**以任务完成和效率提升为导向的用户群体**。他们关注的重点不是 AI 的技术概念本身，也不是模型能力的上限展示，而是 AI 是否能够在真实场景中**稳定、低门槛、低风险地帮助自己完成任务**。

这类用户通常因“AI 能替我做事”而产生兴趣，但决定是否持续使用的关键，在于产品是否真正具备可落地的执行价值，包括是否足够易用、是否安全可控、是否成本透明，以及是否能够持续嵌入日常工作流。

从本质上看，他们不是为“聊天”而来，而是为“省事”而来；不是追求炫技体验，而是追求明确、持续的效率收益。



虽然产品面向的是同一类“任务驱动的效率用户”，但在使用深度上可进一步划分为两层：

* 轻量效率用户

这类用户的核心诉求是“尽快省事”。他们有明确任务，但不愿投入过多时间学习复杂功能，也不希望经历繁琐配置。他们更偏好**简单、直接、可立即见效**的产品体验，可通过**主动协助、快捷任务入口、默认模板能力**等方式满足需求。

* 进阶效率用户

这类用户不仅希望 AI 帮助完成单点任务，还希望将 AI 纳入自己的长期工作流。他们愿意配置技能和插件，希望产品支持更灵活的任务编排和跨应用协作。这部分用户是产品形成长期粘性和使用深度的重要人群。

## **1.2 用户痛点**

* **上手门槛高**

  许多现有的桌面 Agent 产品在使用前需要用户进行复杂的安装、配置和学习较多的操作步骤和概念。例如，需要配置工作区、连接各类服务、理解插件机制等。对于普通用户来说，这些步骤过于繁琐，造成了极高的**入门门槛**，导致用户难以快速上手并投入实际使用，尤其是对于缺乏技术背景的用户。&#x20;

* **使用习惯门槛高**

  即便产品能够被成功安装并配置完成，用户仍然面临着**使用习惯的障碍**。很多用户并不习惯在日常工作或生活中主动调用 Agent 来协助解决问题。用户倾向于依赖传统的操作方式，例如直接搜索、手动处理信息、拖放文件等，难以将 Agent 集成到他们的日常流程中。这就导致了用户在需要时可能会忘记使用 Agent，从而影响产品的使用频率和效果。&#x20;

* **触发不合理**

  目前大多数 Agent 产品的交互触发方式存在问题。要么过度依赖用户主动提问和发起任务，使得 Agent 处于“被动等待”状态，难以在用户需要时及时出现；要么当 Agent 主动介入时，常常在不恰当的时机出现，导致打扰用户的正常操作。例如，某些产品会在用户正在处理复杂任务时突然弹出对话框或通知，反而增加了用户的负担，而没有解决实际问题。这种不合理的触发方式，往往让用户感到烦躁，降低了产品的可用性和满意度。&#x20;

* **重复性操作和上下文切换成本高**

  在日常工作中，用户通常会遇到大量重复性操作，例如复制粘贴信息、整理文件、在不同网页和应用之间来回切换、反复确认下一步该做什么等。每当需要从一个工具切换到另一个工具，或在不同任务之间往返时，都需要重新整理上下文、回忆操作步骤并衔接当前进度，这会持续消耗时间和注意力。用户希望在当前任务现场获得更及时的提示、内容整理、下一步建议和相关资料快速打开能力，从而减少机械性重复劳动和频繁切换带来的负担。

* **执行可控性与安全感不足**

  随着 Agent 参与更多实际执行，用户会进一步关注文件落点是否清晰、危险操作是否隔离、不同风险动作是否有明确确认机制，以及执行失败后是否能够恢复。若缺少清晰的写入边界、权限分级和回滚兜底，用户通常只愿意把 Agent 当作辅助工具，而不会放心将关键任务交由其协助完成。

## 1.3 解决方案

1. **轻量化设计：降低安装、配置和理解成本，让普通用户快速上手**

   我们的产品通过简化安装和配置过程，最大限度地减少用户在启动和使用过程中的复杂步骤。与传统的桌面 Agent 产品相比，CialloClaw **无需复杂的权限设置、插件安装或工作区配置**，普通用户只需简单几步就可以完成产品的初始化和配置，立刻进入可用状态。我们还设计了直观的用户界面和操作流程，用户可以在没有技术背景的情况下，轻松理解并开始使用。这样，产品不仅降低了**上手门槛**，还让用户能迅速体验到其核心功能，避免因繁琐操作而产生的**学习成本**。&#x20;

2. **低负担交互：减少用户把需求说清楚并交给 Agent 执行的成本**

   传统的桌面 Agent 产品往往要求用户明确地表达需求，并通过复杂的交互操作将任务传递给 Agent。CialloClaw 通过 **上下文感知** 和 **自然语言处理技术**，让用户不再需要明确指定每个操作步骤或任务要求。系统会主动捕捉用户当前的任务场景（如正在处理的文件、网页或邮件），并根据上下文自动推测用户的需求，主动提出帮助，而不是让用户反复描述需求。通过这种方式，用户仅需点击几下，即可将任务交给 Agent 处理，**大大减少了需求表达的成本**，提升了操作效率。&#x20;

3. **低打扰主动协助：基于上下文在合适时机提供帮助，而不是一味等待提问或频繁打扰**

   现有的 Agent 产品往往在用户遇到问题时被动等待，而在主动介入时又容易打扰用户。CialloClaw 打破这一局限，采用 **低打扰的主动协助机制**。系统会根据用户当前的上下文（例如正在浏览的页面、编辑的文件或正在进行的任务）主动感知帮助机会，并在合适时机给出提示或建议，而不是频繁弹窗或强行打断用户。**这种设计让用户在无需主动提问的情况下，能够在真正需要帮助时得到及时的支持**，避免了用户的反感，也提升了产品的实用性和满意度。&#x20;

4. **可控执行与分级安全：降低关键任务交付顾虑**

   针对 Agent 参与执行后带来的可控性和安全顾虑，CialloClaw 通过清晰工作区边界、执行面沙盒化、风险分级授权以及快照回滚机制，降低文件误写、环境污染和高风险误操作带来的使用阻力。在此基础上，产品既保持桌面常驻和低打扰体验，又能让 Agent 在更可控的前提下参与实际任务推进。



# 2. 市场调研与竞品分析

## 2.1 市场现状：当前 Agent 产品的主流解法

近几年，Agent 产品的重点已从单纯对话转向任务执行、流程编排与跨应用协作。当前市场主流解法主要包括：

* **计算机使用型 Agent**
  &#x20;以浏览器或桌面 GUI 为操作对象，通过界面理解与鼠标键盘控制完成多步任务，强调自动执行与流程外包。&#x20;

* **工作区 / 平台式 Agent**
  &#x20;以工作区、任务面板、技能系统、定时任务和 workflow 编排为核心，强调“规划—执行—交付”的任务闭环。&#x20;

* **轻量引擎 / 多入口接入型 Agent**
  &#x20;强调低资源占用、跨平台部署和多入口接入，通常通过 CLI、WebUI 或消息入口接收任务，更接近 Agent 引擎能力而非桌面原生协作产品。

整体来看，市场已经验证了执行型 Agent 的需求成立，但不同路线的重点主要集中在执行能力、平台组织能力和引擎接入能力，产品形态分化明显。

## 2.2 市场空缺与机会判断

尽管当前 Agent 市场已经具备任务执行、流程编排和跨应用协作能力，但市场仍缺少一种能够以**低门槛、低打扰、低切换成本**进入真实任务现场，并在长期使用中形成稳定协作关系的产品形态。与此同时，随着 Agent 逐步从“提供建议”走向“参与执行”，市场对于执行边界清晰、风险分级明确、结果可恢复的产品能力也提出了更高要求。&#x20;

当前市场的主要机会体现在以下几个方面：

* **缺少桌面现场承接型 Agent**
  &#x20;现有产品大量依赖消息入口、工作区、浏览器侧边栏或专门平台承接任务，而不是直接在正在发生的桌面任务环境中自然介入。围绕桌面现场承接协作的产品形态仍然相对不足。

* **缺少低打扰主动协助的统一体验**
  &#x20;市场需要的不是更强烈的打断式提醒，而是在合适时机、以合适强度提出建议，再由后续确认承接执行。现有市场虽已有主动协作探索，但“轻提示、低打扰、确认后执行”的统一体验仍未成为主流。

* **缺少从理解、提醒到轻执行的连续链路**
  &#x20;很多产品要么停留在信息辅助层，要么直接强调复杂自动化执行，中间缺少一条从上下文理解、任务提醒、下一步建议到轻执行支持的连续产品路径。

* **缺少以长期协作为目标的 Agent 形态**
  &#x20;主流产品普遍更强调单次任务闭环或执行能力，而长期习惯学习、流程沉淀和“越用越顺手”的协作关系仍未成为统一的主流方向。

* **缺少兼顾轻量用户与进阶用户的成长路径**
  &#x20;一部分产品偏重平台和配置，更适合进阶用户；另一部分产品偏重引擎能力和接入效率，更适合技术用户。市场仍存在一种既能让轻量用户快速获得帮助，又能让进阶用户逐步沉淀流程和协作习惯的产品机会。

* **缺少可控执行的桌面 Agent 体验**
  &#x20;随着 Agent 参与更多实际操作，市场不仅需要“能执行”的产品，也需要“执行边界清晰”的产品。用户会越来越关注文件落点是否明确、危险操作是否与宿主环境隔离、不同风险动作是否有分级确认，以及执行失败后是否具备恢复能力。当前补充的工作区边界、执行面沙盒化、策略引擎和快照回滚能力，正说明这一方向已经成为产品竞争力的重要组成部分。

概括而言，当前市场并不缺少“能做事”的 Agent，真正缺少的是一种能够**在桌面环境中自然出现、持续协作，并在可控前提下参与执行**的产品形态。这构成了后续关键竞品对比和产品切入点判断的基础。&#x20;

## 2.4 CialloClaw 的切入点与独特价值

基于前述市场空缺与竞品对比，CialloClaw 的切入点并不在于继续叠加更多工具调用能力，而在于重构 Agent 进入任务流的方式：以桌面常驻作为主入口，以低打扰主动协助作为主交互，以长期习惯学习作为主增益机制，形成区别于引擎型、工作区型和平台型 Agent 的产品路径。&#x20;

CialloClaw 的独特价值主要体现在以下几个方面：

* **桌面原生的协作入口**
  &#x20;产品以常驻桌面的悬浮球、对话悬浮窗和气泡提示作为核心 UI 形态，不要求用户长期停留在独立平台或工作区中，而是在原有桌面环境中承接协作。

* **低打扰的主动协助机制**
  &#x20;产品通过持续感知当前页面、文件、选中内容、复制行为、任务状态和系统状态，识别帮助机会后先以轻提示提出协助，再由后续确认承接执行，核心逻辑是“先提示，再确认，后执行”。&#x20;

* **围绕真实任务现场提供帮助**
  &#x20;产品并非等待任务被完整描述后再执行，而是围绕当前任务环境直接提供提醒、总结、解释、下一步建议、资料打开和轻量执行支持，减少需求表达成本与上下文切换成本。&#x20;

* **从单次任务走向长期协作**
  &#x20;产品通过“另世我 / 镜子”机制，对任务路径、操作习惯和处理方式进行持续沉淀，在后续相似场景中提供更贴近个人方式的建议、提示或半自动工作流，使 Agent 从单次响应逐步过渡到长期协助。&#x20;

* **同时覆盖轻量使用与进阶扩展两类需求**
  &#x20;一方面，产品通过轻量化入口、低负担交互和低存在感常驻降低首次使用门槛；另一方面，又通过任务巡检、流程沉淀和后续可扩展的工作流能力支撑更深度的长期使用。&#x20;

综合来看，CialloClaw 的差异化不在于单项能力的绝对领先，而在于以**桌面常驻 + 主动协助 + 长期协作**的统一形态，降低 Agent 进入真实任务流的成本，并提升其持续使用的可能性。相较于 PicoClaw 的引擎化路径、QoderWork / DuMate 的工作区路径以及 Claude Cowork 的平台化协作路径，CialloClaw 更接近“桌面任务现场中的长期协作 Agent”。&#x20;



# 3. 产品定义

## 3.1 产品形态与交互入口

CialloClaw 是一款**常驻桌面的 Agent 产品**。
它不是一个需要用户长期停留其中操作的网页平台，也不是以命令行或聊天窗口作为主要入口的工具，而是一个以**悬浮球**为默认入口、以**轻提示卡片**为主动提醒方式、以**对话弹出层**为主要承接界面的桌面助手。

从用户视角看，CialloClaw 主要由四个可感知的产品部件组成：

* **悬浮球**
  &#x20;作为默认常驻入口，停留在桌面边缘，以低存在感方式持续显示当前状态，并支持点击、悬停、拖拽和右键操作。&#x20;

* **任务环**
  &#x20;当用户悬停或触发悬浮球时，周围展开常用任务入口，用于快速发起总结、翻译、提问、提醒等高频操作。&#x20;

* **轻提示卡片**
  &#x20;当系统识别到明确帮助机会时，不直接打断用户，而是以非模态、低侵入的方式给出提示，用户可选择忽略、稍后处理或点击进入进一步操作。&#x20;

* **对话弹出层**
  &#x20;用于承接具体任务、展示结果、确认执行和进行多轮交互，是产品的主要任务处理界面。&#x20;

整体交互遵循“**常驻可见、轻量触达、先提醒后展开**”的原则：
&#x20;用户在不离开当前应用的前提下，即可快速唤起帮助；当系统主动介入时，也优先以轻提示而非强打断方式出现，只有在用户确认后才进入更深层的任务处理。

## 3.2 核心能力

CialloClaw 的核心功能围绕“陪伴 + 助手”展开，不以单一对话能力为中心，而是通过场景理解、任务推进、长期记忆与可控执行，形成一套面向桌面任务环境的连续协作能力。结合当前产品定义，核心功能可概括为以下四项：

### 3.2.1 场景助手

场景助手负责围绕当前任务环境提供即时帮助，是产品最直接面向高频桌面任务的核心功能。其重点不在于让用户先补充完整上下文，再交由系统处理，而是在页面、选中文本、复制内容、上传文件、程序状态、错误界面或停留行为中识别明确线索，并在合适时机给出总结、解释、翻译、提炼与下一步建议。

依托场景助手，产品能够支持以下典型任务：

* &#x20;对网页、文档、代码片段和选中文本进行总结、翻译、解释与重点提炼；&#x20;

* &#x20;对错误信息、终端报错和异常界面提供原因分析与处理建议；&#x20;

* &#x20;从文本或页面中提取关键信息和结构化要点；&#x20;

* &#x20;在明确场景下生成回复草稿、会议前清单、下一步动作建议等可直接使用的结果。

### 3.2.2 待办管家

待办管家负责承接任务文件、待办清单和周期性巡检，是产品从“提醒”走向“推进任务”的核心功能。其重点不在于单纯记录待办，而在于帮助识别优先级、提示临近事项，并围绕当前任务提供下一步协助，使任务从“被记下”进一步转向“被推进”。

待办管家主要支持以下任务：

* &#x20;定时读取和解析指定的 `.md` 任务文件；&#x20;

* &#x20;识别任务项、日期、状态、优先级和标签；&#x20;

* &#x20;生成今日任务摘要、临期提醒和优先级建议；&#x20;

* &#x20;打开相关文件、网页、目录，或生成草稿和执行建议；&#x20;

* &#x20;围绕重复出现的任务情境，提供更稳定的推进方式与协作支持。

### 3.2.3 镜子记忆

镜子记忆不仅保存交互历史，更承担产品的长期认知能力。其目标不是做对话缓存，而是通过结构化沉淀、周期性总结和智能召回，逐步形成对用户偏好、任务状态和工作方式的持续理解，从而支持个性化理解、连续任务执行、主动服务和长期陪伴。

镜子记忆主要支撑以下能力：

* &#x20;沉淀短期记忆和长期记忆，用于连续理解当前任务；&#x20;

* &#x20;生成阶段性总结和用户画像，形成更稳定的协作偏好；&#x20;

* &#x20;从重复行为中识别工作模式和高频路径；&#x20;

* &#x20;在后续相似场景中提供更贴近个人习惯的建议、提示与半自动协助。

### 3.2.4 安全执行模式

安全执行模式负责为 Agent 的实际操作建立基本边界和兜底能力，使产品在参与执行时不仅“能做”，也“可控”。该功能并不将产品定义为重型安全平台，而是在桌面协作基础上，为文件写入、命令执行、网页自动化和系统动作提供必要的风险分级、执行隔离与恢复机制。

安全执行模式主要支持以下能力：

* &#x20;将用户成果文件与应用运行数据分层存储，并通过工作区管理保证文件落点清晰可见；&#x20;

* &#x20;通过执行面沙盒化隔离高风险动作，避免直接污染宿主环境；&#x20;

* &#x20;通过红黄绿灯授权系统对不同风险等级的操作进行静默执行、授权确认或强制拦截；&#x20;

* &#x20;在高风险动作前创建快照，并在失败或异常时支持恢复与回滚。

总体而言，CialloClaw 的核心功能并不是若干离散工具的简单堆叠，而是围绕“理解当前场景、推进当前任务、积累长期习惯、保障执行边界”构建的一套桌面协作能力。这些功能共同定义了产品的核心使用价值：在不脱离当前工作环境的前提下，帮助用户更自然地完成日常任务，并在长期使用中形成持续协作关系。

## 3.3 核心任务场景

CialloClaw 的核心任务场景并不围绕单一对话展开，而是围绕用户已经处于工作流中的桌面环境展开。产品最适合在用户**正在阅读、正在处理任务、正在排查问题、正在切换应用或正在等待结果**的过程中介入，通过低打扰方式提供理解、提醒、建议和轻量承接。整体上，核心任务场景可归纳为以下四类。

### 3.3.1 内容理解与即时处理

当用户处于阅读网页、浏览文档、查看技术资料、选中文本或复制片段内容的场景时，CialloClaw 提供即时的内容理解与处理支持。该场景的核心价值在于减少重复性的上下文补充和内容搬运，让总结、翻译、解释、提炼和草稿生成能够在当前页面或当前应用附近直接完成，而不必先离开现场进入独立工具。&#x20;

这一场景主要覆盖：

* &#x20;网页、文档、代码片段的总结、改写、翻译与解释；&#x20;

* &#x20;从选中文本中提取关键信息和结构化要点；&#x20;

* &#x20;围绕当前内容生成回复草稿、会议清单或下一步建议。

### 3.3.2 任务巡检与任务推进

当用户通过 Markdown 任务文件、待办清单或阶段性工作安排管理事务时，CialloClaw 适合作为持续巡检与任务推进工具使用。其核心不只是“提醒还有哪些任务”，而是围绕任务状态给出更可行动的承接，例如生成摘要、识别临期事项、打开相关资料、提供下一步建议，并帮助任务从记录状态走向推进状态。

这一场景主要覆盖：

* &#x20;开工前、会前、截止临近时的任务简报与提醒；&#x20;

* &#x20;对任务文件的定时读取、结构识别和状态判断；&#x20;

* &#x20;围绕当前任务打开文件、网页、目录或生成草稿内容。

### 3.3.3 异常发现与问题排查

当用户遇到错误弹窗、终端报错、页面异常、长时间停留或明显重复失败时，CialloClaw 适合作为问题排查与下一步协助工具使用。该场景的重点不是自动接管故障处理，而是在错误发生或卡住时及时提供解释、原因分析、排查建议和上下文整理，降低用户在问题定位过程中的中断成本和切换成本。&#x20;

这一场景主要覆盖：

* &#x20;对报错信息、异常界面和失败结果进行解释与初步分析；&#x20;

* &#x20;围绕当前问题整理上下文、归纳尝试记录并给出下一步建议；&#x20;

* &#x20;在涉及系统动作或高风险操作时，仅提供建议或进入确认流，不直接默认执行。

### 3.3.4 长期习惯学习与持续协作

当产品在用户的长期使用中持续积累任务线索、交互记录和处理方式后，CialloClaw 适合作为个性化协作助手使用。该场景的重点不在于单次任务完成，而在于通过周期性总结、习惯提炼和相似场景召回，使产品逐步理解用户常见的任务路径、工作方式和决策顺序，从而形成“越用越顺手”的长期协作关系。

这一场景主要覆盖：

* &#x20;周期性生成总结报告和用户画像；&#x20;

* &#x20;从重复行为中归纳高频路径、工作模式和偏好；&#x20;

* &#x20;在相似任务再次出现时，基于历史习惯提供更贴近个人方式的建议与准备支持。&#x20;

总体而言，CialloClaw 的核心任务场景并不是“代替用户完成所有操作”，而是**在桌面工作流中承接理解、推进、提醒和协作**。其最核心的产品价值，在于让帮助发生在任务现场、发生在合适时机，并在长期使用中逐步演变为稳定的桌面协作关系。&#x20;

## 3.4 典型用户故事

## 故事 1：我想要一个装好就能用、顺手就能帮上忙的工具

我平时工作很杂，每天都要在聊天软件、表格、文档、网页后台和文件夹之间来回切。

&#x20;一会儿找资料，一会儿改内容，一会儿对数据，一会儿催进度。很多事情其实不难，但特别碎，光是把东西找齐、把前因后果接上，就已经很费劲了。

我不是不能接受新工具，但我不想一上来先学半天。

&#x20;我不想先研究怎么设置，也不想先搞懂一堆规则。我更希望它装好就能用，我直接告诉它“帮我把这个整理一下”“帮我找一下那个文件”“把这段话改顺一点”，它就能马上开始帮我。

对我来说，最有用的不是它功能有多全，而是我正忙着的时候，它能不能顺手帮我省一点事。

&#x20;比如我正在看文档，它可以顺手帮我总结重点；我在整理资料，它可以帮我归一下类；我快要写一份东西了，它能先帮我搭个草稿。这样我就不用为了一个小动作，又切去另一个地方。

所以我最容易有的感觉是：
**“这个工具不用我先学会怎么用，而是我一打开，它就能直接帮我干活。”**

***

## 故事 2：我希望它越来越懂我，但别太烦，也别乱动我的东西

我当然希望这个工具能越用越懂我。

&#x20;比如我平时做一类事情，习惯先看什么、再做什么，它要是能慢慢记住，下次就不用我每次都从头讲一遍。这样用久了，它才真的像个帮手，而不是每次都要重新带一遍的新人。

但我也很在意分寸。

&#x20;我不希望它老是跳出来提醒我，弄得我更烦；也不希望它自己动我的文件、帮我覆盖内容、乱发东西，最后出了问题还不知道怎么找回来。

我更能接受的是，它该提醒的时候提醒一下，而且让我知道为什么提醒；我不想被打扰的时候，也能关掉或者少一点。

&#x20;碰到重要的事情，它不要自己做决定，先问我一句；做完了，也最好让我知道它动了什么，这样我心里才踏实。

所以对我来说，真正好的感觉是：
**“它会慢慢懂我，能帮我做事，但不会一直打扰我，也不会在关键地方自己乱来。”**

## 3.5 产品边界

CialloClaw 的定位，不是命令行驱动的 Agent 引擎，也不是以工作区、任务面板和复杂编排为中心的重型协作平台。产品强调在原有桌面环境中自然承接协作，而不是要求用户先进入一个独立系统，再将任务显式交付给 Agent。

同时，CialloClaw 也不以高侵入、强接管式的自动执行为核心目标。其重点在于通过轻提示、任务建议、内容处理、任务推进和有限度的执行支持，帮助用户更顺畅地完成任务；对于高风险操作，则通过分级授权、执行隔离和恢复机制建立基本边界。

因此，CialloClaw 的产品边界可以概括为：**以桌面常驻为入口，以低打扰协作为主要交互方式，以任务现场承接和长期协作为核心目标，并在可控前提下参与有限执行的桌面 Agent 产品。**



# 4. 产品细化

## 4.1 设计原则与整体交互逻辑

CialloClaw 的产品细化建立在“**低存在感常驻、任务现场承接、先提示后展开、可控前提下执行**”这一整体原则之上。产品的目标不是替代用户原有的工作环境，也不是通过频繁弹窗或强制接管来体现智能，而是在用户已经处于工作流中的情况下，以更低打扰、更低切换成本的方式提供持续协助。

在设计原则上，产品遵循以下四点：

* **默认常驻，按需展开**
  &#x20;产品以悬浮球作为默认常驻入口，平时保持低存在感，仅通过状态变化传递必要信息；只有在用户主动点击、悬停或系统识别到明确帮助机会时，才展开更进一步的交互。

* **先提示，再确认，后执行**
  &#x20;对于大多数场景，系统优先通过轻提示、任务建议或状态提醒表达协助意图，而不是直接打断当前任务或默认执行动作。只有在用户确认后，才进入更深层的对话、任务承接或执行流程。

* **优先在当前任务现场承接协作**
  &#x20;产品尽量围绕当前页面、选中内容、任务文件、错误状态、程序状态和系统状态提供帮助，减少用户离开当前应用、重复描述上下文或切换到独立平台的成本。其核心不是把任务“拉进系统”，而是让系统“进入任务现场”。

* **高风险动作必须可控**
  &#x20;当任务涉及文件修改、命令执行、网页自动化或系统级操作时，产品以清晰工作区边界、分级授权、执行隔离和恢复机制作为基本约束，保证 Agent 的执行建立在可理解、可确认、可恢复的前提下。

基于上述原则，CialloClaw 的整体交互逻辑可以概括为：

> **常驻入口 → 场景感知 → 识别机会 → 轻提示 → 用户确认 → 对话承接 / 任务推进 / 有限执行 → 结果反馈与记忆沉淀**。

具体来说，产品在日常状态下以悬浮球形式停留在桌面边缘，通过场景感知持续关注当前任务环境；当识别到明确帮助机会时，优先使用轻提示或状态变化向用户表达建议；如果用户选择继续，则进入对话弹出层或相应任务入口，由系统承接后续内容处理、任务推进或有限执行；当交互完成后，系统再将其中有价值的信息沉淀为短期上下文或长期记忆，为后续相似场景提供支持。

因此，CialloClaw 的整体交互逻辑并非“用户每次主动发问，系统被动响应”的传统助手模式，也不是“系统持续强介入、默认接管”的高侵入模式，而是一种以桌面常驻为入口、以轻提示为桥梁、以任务推进为目标、以可控执行为边界的协作逻辑。

## 4.2 桌面入口细化

### 4.2.1 悬浮球

悬浮球是 CialloClaw 的默认常驻入口，也是产品最核心的外在形态。其定位不是传统桌面宠物，也不是单纯的快捷图标，而是一种兼具**状态表达、快速唤起和轻量陪伴**能力的桌面入口。产品希望通过悬浮球在用户视觉边缘持续存在，但不与当前任务争夺注意力，从而形成“低存在感常驻、需要时一秒触达”的交互体验。

在日常状态下，悬浮球主要承担三类作用：

* &#x20;通过角色表情、状态微指示器和细微动画表达当前状态；&#x20;

* &#x20;作为用户主动进入产品的第一入口；&#x20;

* &#x20;作为任务环、轻提示和对话弹出层的锚点。

从交互上看，悬浮球支持悬停、点击、右键和拖拽等基础操作，并通过贴边、自动隐藏和低存在感模式减少遮挡与干扰。其设计目标不是让用户频繁“打开一个应用”，而是让产品像系统级入口一样自然存在于桌面环境中。

### 4.2.2 任务环与轻提示

任务环和轻提示共同构成 CialloClaw 的一级任务承接层。前者负责承接高频、短链路、可快速触发的常用动作，后者负责在系统识别到明确帮助机会时，以低侵入方式提出建议。两者的共同目标，是在不打断当前工作流的前提下，让用户更容易进入协作。

任务环围绕悬浮球展开，适合承接总结、提问、翻译、搜索、提醒等高频动作。其价值不在于展示全部能力，而在于将少量最常用、最容易即时完成的小任务放在一级入口，降低用户进入成本。任务环中的能力既包括固定槽位中的常用任务，也包括根据当前上下文动态变化的任务入口。

轻提示则是产品主动协助的主要外显方式。与弹窗式打断不同，轻提示以简短文字、轻量卡片或状态变化提示为主，只在判断“帮助价值较高、打扰成本可接受”时出现，并允许用户选择展开、稍后处理或直接忽略。这样，系统的主动性被控制在一个更可接受的范围内，避免主动协助演变为通知噪声。

### 4.2.3 对话弹出层

对话弹出层是 CialloClaw 的主要任务承接界面，用于处理需要进一步交流、展示结果、确认动作或进入多轮交互的场景。其定位不是一个完整独立工作区，而是一个围绕悬浮球展开的轻量任务面板，用于在当前工作现场附近承接更深层的协作。

在结构上，对话弹出层通常包含输入区域、结果展示区域、快捷动作区域和任务状态区域。用户可以在其中发起问题、查看系统生成的总结或草稿、确认下一步动作，也可以在需要时查看当前任务状态和执行进度。对于场景助手、待办管家、镜子记忆和安全执行模式等核心功能，对话弹出层都是承接后续操作的主要界面。

这一设计的重点在于：用户不必跳转到独立网页或完整平台，只需围绕当前桌面环境展开一个轻量窗口，即可进入更深层的任务协作流程。对话只是形式之一，其本质仍是任务承接，而不是单纯聊天。

### 4.2.4 右键菜单与控制面板

右键菜单和控制面板共同承担产品的低频管理与系统级控制能力。前者强调快捷、克制和工具感，后者强调规则管理、能力开关和状态查看，两者共同构成产品的控制入口，但不参与日常高频协作。

右键菜单主要用于承接暂停 Agent、请勿打扰、打开聊天、打开控制面板、隐藏悬浮球和退出程序等系统式操作。其定位更接近桌面工具的上下文菜单，而不是任务处理界面，因此要求结构简洁、可快速扫读，并尽量避免复杂层级。

控制面板则作为低频但必要的完整管理界面，主要承接任务配置、提示频率、隐私与上下文来源、记忆管理、安全设置、日志与历史等内容。它的作用不是成为新的主工作区，而是在用户需要时提供规则调整、状态查看和能力管理入口，从而支撑产品的长期可控使用。

整体来看，桌面入口层的设计目标并不是增加交互复杂度，而是通过悬浮球、任务环、轻提示、对话弹出层和控制入口之间的分工，形成一套**由浅入深、从轻到重、从陪伴到协作**的桌面交互结构。

## 4.3 核心功能细化

### 4.3.1 场景助手

#### **功能定位**

场景助手负责围绕用户当前所处的任务环境，持续识别帮助机会，并以低打扰方式提供总结、解释、提醒、建议和轻量承接支持。其核心目标不是让用户每次都先打开对话框、补充上下文再发起请求，而是让系统在用户已经暴露出明确任务线索时，直接理解当前场景并进入协作。&#x20;

从产品逻辑上看，场景助手不是一个单点工具，而是一层**事件驱动的上下文感知与协作承接层**。它将“用户正在看什么、做什么、卡在什么地方、当前系统处于什么状态”转化为可被 Agent 理解的信号，再根据时机和价值决定是否进行提示、建议或进一步承接。

#### **核心输入 / 触发来源**

场景助手的输入主要来自三类感知能力：程序感知、屏幕感知和硬件感知。三者共同构成系统对“当前场景”的判断基础。

**1. 程序感知**

程序感知主要围绕当前活跃页面、窗口和程序状态展开，用于识别用户正在处理的任务类型及其上下文线索。

&#x20;典型输入包括：

* &#x20;当前活跃窗口或页面内容；&#x20;

* &#x20;用户上传的文件及文件类型；&#x20;

* &#x20;短时上下文信息，用于连续理解当前任务，而不是只做单点识别；&#x20;

* &#x20;程序运行状态、错误弹窗、终端报错和异常界面。

**2. 屏幕感知**

屏幕感知主要围绕用户在当前界面上的行为展开，用于识别“此刻是否值得介入”。

&#x20;典型输入包括：

* &#x20;定时全屏检测与页面状态监测；&#x20;

* &#x20;在某页面长时间停留、反复切换或长时间卡住；&#x20;

* &#x20;复制、选中文本、选中图片区域等强意图动作；&#x20;

* &#x20;浏览网页、阅读文档、查看报错信息等高频内容处理场景。

**3. 硬件感知**

硬件感知主要围绕系统本身的状态变化展开，其作用不是做传统意义上的监控面板，而是把底层系统状态转化为可感知、可互动、可行动的 Agent 输入。

&#x20;典型输入包括：

* &#x20;CPU、内存、磁盘、IO、网络、电池、温度等系统状态；&#x20;

* &#x20;网络连通性、延迟、抖动；&#x20;

* &#x20;前台应用类别、系统空闲 / 忙碌状态；&#x20;

* &#x20;输入节奏、停顿频率等反映当前工作状态的信号。

为了降低耦合和风险，硬件感知并不直接把原始采样暴露给执行层，而是由主进程将其转换为语义事件，例如 `memory_pressure_high`、`network_unstable`、`disk_space_low` 等，再交由 Agent 决策层使用。

#### 核心输出 / 用户可完成的任务

基于上述感知输入，场景助手主要输出以下几类协作结果：

* **内容理解类输出**
  &#x20;对网页、文档、代码片段、选中文本和错误信息进行总结、翻译、解释、提炼和术语说明。&#x20;

* **问题排查类输出**
  &#x20;对报错、异常页面、终端失败和长时间停滞场景提供原因分析、排查建议和下一步动作提示。

* **任务推进类输出**
  &#x20;从当前内容中提取待办事项、生成草稿、整理上下文、打开相关资料或给出下一步行动建议。&#x20;

* **状态提醒类输出**
  &#x20;当系统资源异常、网络波动、电量不足或设备负载过高时，以更可理解的方式给出提醒，并在必要时提出建议或准备动作。

换言之，场景助手不只是“告诉用户发生了什么”，而是尽量把输出推进到“现在可以做什么”这一层。

#### 交互方式

场景助手遵循“低侵入、先提示、后展开”的交互原则。

&#x20;其典型承接方式包括：

* &#x20;通过悬浮球状态变化表达当前场景中存在帮助机会；&#x20;

* &#x20;通过轻提示卡片给出简短建议或待确认动作；&#x20;

* &#x20;通过任务环承接高频、短链路操作；&#x20;

* &#x20;在用户确认后进入对话弹出层，进行更深入的解释、草稿生成或任务承接。&#x20;

在提示策略上，系统优先使用**强意图触发**，如选中、复制、上传、打开任务文件、出现错误弹窗、截止临近等；而对于长时间停留、频繁切换窗口等弱信号，更适合用于静默建议，而不适合直接做强提示。

#### 控制机制与边界

场景助手虽然强调主动协作，但主动性并不等于无边界。其控制原则包括：

* &#x20;用户可暂停 Agent、关闭监测、关闭主动建议或管理触发规则；&#x20;

* &#x20;系统需提供状态可见、行为可查、规则可配、执行可追踪的控制能力；&#x20;

* &#x20;默认以建议态和轻提示为主，不频繁弹窗、不抢焦点、不打断当前任务流；&#x20;

* &#x20;涉及发消息、改文件、执行系统动作等外部操作时，必须进入确认流，并由安全执行模式承接。&#x20;

因此，场景助手的边界可以概括为：**负责发现机会、理解场景、提出协助和承接轻量任务，但不以高风险自动执行为默认目标。**

#### 典型价值

从产品价值上看，场景助手解决的不是“系统能不能回答问题”，而是“当用户已经在做事时，系统能不能在合适时机顺手帮上一把”。它所降低的，主要是以下几类成本：

* &#x20;用户手动补充上下文的成本；&#x20;

* &#x20;在多个工具之间切换的成本；&#x20;

* &#x20;卡住后重新整理问题的成本；&#x20;

* &#x20;因系统状态异常而产生的额外判断成本。

### 4.3.2 待办管家

#### **功能定位**

待办管家负责围绕用户已经记录下来的待办事项、阶段性安排和任务文件，提供持续巡检、状态识别、提醒与下一步协助。其目标不是替代用户原有的任务记录方式，而是在用户已经有任务记录基础的前提下，帮助任务从“被写下来”进一步走向“被持续推进”。

从产品逻辑上看，待办管家不是一个静态待办列表，也不是单纯的提醒器，而是一层**任务巡检与推进层**。它持续关注用户任务文件中的变化、任务状态与时间线索，并将这些信息转化为更可行动的输出，例如摘要、优先级建议、资料入口和下一步动作提示。

#### **核心输入 / 触发来源**

待办管家的输入主要来自任务文件、任务状态和时间规则三类来源，共同构成系统对“当前该推进什么”的判断基础。

1. **任务文件输入**

待办管家支持读取用户指定的 `.md` 任务文件，并从中提取可用于巡检和推进的结构化信息。

典型输入包括：

* Markdown 中的任务列表，如 `- [ ]`、`- [x]`；

* 带日期、优先级、标签或上下文说明的任务项；

* 同一任务文件中的标题、分组和上下文描述。

- **任务状态输入**

在任务文件被解析后，系统会进一步判断任务当前所处的状态，用于识别哪些任务值得提醒或推进。

典型输入包括：

* 今日到期或即将超时的任务；

* 长时间未处理的任务；

* 已完成但未归档的任务；

* 需要依赖其他条件才能开始的任务。

- **时间与巡检规则输入**

待办管家并不只在用户打开文件时工作，而是基于定时巡检和文件变更触发来持续关注任务状态。

典型输入包括：

* 固定频率巡检，如每 10 分钟、每小时、每天早上；

* 文件变更后的即时重新解析；

* 开工前、会前、截止临近等关键时间节点。

#### **核心输出 / 用户可完成的任务**

基于上述输入，待办管家主要输出以下几类协作结果：

* **任务摘要类输出**
  对当前待办进行汇总，生成今日任务摘要、未完成事项列表和临期提醒。

* **优先级判断类输出**
  根据时间、状态和上下文信息，对任务进行优先级提示和排序建议，帮助用户判断“现在先做什么”。

* **下一步推进类输出**
  围绕具体任务给出下一步动作建议，例如打开资料、打开网页、打开目录，或生成消息草稿、邮件草稿、会议前清单等。

* **任务协助类输出**
  对于部分可承接的轻量任务，系统可在提醒之外进一步提供辅助支持，使任务不只停留在“知道有这件事”，而是更容易进入“开始处理这件事”的状态。

换言之，待办管家不只是“告诉用户还有什么没做”，而是尽量把输出推进到“下一步可以怎么做”。

#### **交互方式**

待办管家遵循“持续巡检、低打扰提醒、按需展开承接”的交互原则。

其典型承接方式包括：

* 通过悬浮球状态变化或轻提示卡片提示任务临近、任务未完成或今日摘要；

* 通过任务环或快捷入口快速进入任务相关操作；

* 在用户确认后进入对话弹出层，查看更完整的任务摘要、优先级建议和下一步动作；

* 在需要时直接打开相关资料、网页、目录，或生成草稿内容。

在提醒策略上，待办管家优先围绕明确时间线索和状态变化进行触发，例如“今天到期”“即将超时”“长时间未处理”等，而不是对所有任务一视同仁地频繁提醒。其目标是让提醒与推进尽可能贴近真实工作节奏，而不是制造新的通知负担。

#### **控制机制与边界**

待办管家虽然承担任务推进功能，但其边界并不在于全自动接管任务流。其控制原则包括：

* 用户需主动指定任务文件或任务来源，系统不默认接管所有个人事务；

* 系统以识别、提醒、建议和轻量承接为主，不默认对任务进行高风险自动执行；

* 涉及发消息、改文件、系统操作等外部动作时，必须进入确认流，并由安全执行模式承接；

* 产品重点在于帮助用户推进任务，而不是替代用户完成整套复杂流程。

因此，待办管家的边界可以概括为：**负责识别任务、判断状态、提出提醒和承接下一步支持，但不以全自动流程接管为默认目标。**

#### **典型价值**

从产品价值上看，待办管家解决的不是“用户有没有把任务记下来”，而是“记下来的任务为什么没有真正形成行动”。它所降低的，主要是以下几类成本：

* 反复打开任务文件检查状态的成本；

* 手动筛选临期任务和重要任务的成本；

* 在任务开始前重新整理资料和上下文的成本；

* 任务系统与真实工作环境脱节所带来的推进成本。

因此，待办管家真正补足的，不只是提醒能力，而是从任务记录到任务推进之间的那一段空白。

### 4.3.3 镜子记忆

#### **功能定位**

镜子记忆是 CialloClaw 面向长期协作的核心功能，负责将用户在持续使用过程中产生的偏好、任务状态、工作习惯和阶段性结论进行结构化沉淀，并在后续相似场景中智能召回。其目标不是简单保存聊天记录，也不是把所有历史内容原样堆积，而是让 Agent 从“即时响应工具”逐步演变为“持续认知用户的系统”。

从产品逻辑上看，镜子记忆由两部分共同构成：
一部分是**记忆系统**，负责对交互信息进行提取、分类、压缩、存储与召回；
另一部分是**镜子机制**，负责通过周期性总结和习惯提炼，把离散行为转化为对用户工作方式的长期理解。前者是认知基础设施，后者是可被用户感知的长期协作能力。

#### **核心输入 / 触发来源**

镜子记忆的输入主要来自用户交互、任务推进和周期性总结三类来源，共同构成系统对“这个用户一贯如何做事”的理解基础。

1. **用户交互输入**

镜子记忆会从用户与 Agent 的交互中提取可复用信息，而不是直接保存原始文本。

典型输入包括：

* 用户的表达偏好、回答风格偏好和常见语气；

* 用户的能力特征、知识背景和工具使用习惯；

* 在多轮协作中反复出现的约束条件和工作偏好。

- **任务状态输入**

镜子记忆不仅记录“聊过什么”，更关注“当前任务推进到了哪里”。

典型输入包括：

* 当前任务目标和阶段性进展；

* 未完成事项、下一步动作和中间结论；

* 某类任务中反复出现的处理路径和决策顺序。

- **周期性总结输入**

系统会基于短期记忆与长期记忆进行阶段性压缩和总结，以形成更稳定的用户画像与工作方式抽象。

典型输入包括：

* 周期性总结报告中的任务类型、常见路径和高频步骤；

* 用户在一段时间内形成的新习惯和偏好变化；

* 从重复行为中归纳出的工作模式和处理框架。

#### **核心机制**

镜子记忆的核心链路可以概括为：**信息提取 → 分类存储 → 记忆压缩 → 检索召回**。这一链路决定了记忆模块不是简单的历史堆积，而是一个面向长期协作的认知系统。

1. **信息提取**

信息提取的目标，是从对话和任务上下文中抽取“可复用信息”，而不是原始文本本身。系统更关注用户的偏好、技能、目标、任务状态和稳定事实，而不是逐句保存聊天内容。为保证后续可检索和可管理，提取结果应以结构化格式表达，而不是无约束的自然语言片段。

* **分类存储**

镜子记忆的核心思想之一是：不同类型的记忆具有不同生命周期。

例如，用户偏好和用户能力适合长期保留；短期目标和会话状态则更适合中短期存在；任务状态和事实信息则根据场景决定保留时长。这样的分类体系可以避免“垃圾记忆污染”，也避免无意义持久化。

在产品层面，可将主要记忆类型理解为以下几类：

* **用户偏好**：如喜欢结构化回答、偏好某种输出风格；

* **用户能力**：如熟悉 Python、偏好官方文档；

* **短期目标**：如正在准备某次汇报、正在写某份材料；

* **任务状态**：如当前任务进展、下一步动作；

* **稳定事实**：如项目名称、长期工作对象或固定背景信息。

- **记忆压缩**

为了避免上下文无限增长，镜子记忆不会把历史内容一直原样带入后续任务，而是采用滚动摘要与阶段性压缩机制。其目标是在保留当前任务所需上下文的同时，将更早的交互内容压缩为摘要记忆，用于后续召回。这样既能控制 Token 成本，也能避免长链路协作中上下文越来越臃肿。

* **检索召回**

镜子记忆的最终价值体现在“合适的时候，用合适的记忆”。

当用户提出新问题、进行长对话、继续未完成任务或进入相似场景时，系统可基于语义检索与分类过滤，从中召回最相关的偏好、任务状态和历史经验，并将其注入当前上下文。这样做的重点不是召回越多越好，而是通过相关性筛选与 Top-K 控制，减少记忆污染。

#### **存储分层与记忆模式**

镜子记忆在产品上体现为“短期记忆 + 长期记忆 + 镜子总结”三层机制。

1. **短期记忆**

短期记忆主要用于支持当前任务和近距离上下文连续性，典型内容包括：

* 当前会话上下文；

* 程序感知和硬件感知所产生的即时状态；

* 当前任务推进中的中间状态和待确认动作。

- **长期记忆**

长期记忆主要用于沉淀用户画像与长期协作基础，典型内容包括：

* 用户长期偏好；

* 稳定能力特征；

* 反复出现的工作模式；

* 经压缩保留的总结性结论。

- **镜子总结**

镜子总结是用户最容易感知到的记忆外显形式。系统会根据短期记忆与长期记忆进行周期性回顾和总结，并将阶段性结论沉淀为总结报告与用户画像，用于后续更贴近个人方式的协作。文档中给出的两个典型载体是：

* `summary.md`：用于保存阶段性总结，在达到特定字数或时间周期后输出总结结果；

* `USER.md`：用于沉淀用户画像与长期偏好，使系统更稳定地理解用户。

#### **用户可感知的输出与能力**

基于上述机制，镜子记忆主要输出以下几类用户可感知结果：

* **个性化理解类输出**
  在后续任务处理中，系统更倾向于使用贴近用户习惯的表达方式、结果结构和建议风格。

* **连续任务承接类输出**
  在长任务、多轮任务和中断恢复场景中，系统能够记住当前目标、已完成进展和下一步动作，减少重复解释成本。

* **习惯学习类输出**
  从重复行为中提炼工作路径，在相似场景出现时主动给出更贴近个人方式的建议、提示或准备动作。

* **阶段总结类输出**
  定期输出阶段性总结，帮助用户理解自己最近常做的任务、最耗时的重复步骤、可能适合优化的环节以及新形成的习惯。

换言之，镜子记忆不只是“记住过去发生了什么”，而是尽量把记忆转化为“以后如何更懂用户、如何更顺手地协助用户”。

#### **交互方式**

镜子记忆默认在后台运行，不以高频显式交互为主，其价值更多通过长期体验体现出来。主要交互方式包括：

* 在日常使用中，自动提取并沉淀可复用信息，而不要求用户频繁手动维护；

* 在相似任务再次出现时，通过建议、提示和任务承接体现记忆效果；

* 在周期性总结场景中，以总结报告或偏好概览让用户感知长期价值；

* 在控制面板中提供记忆查看、记忆管理和偏好设置入口。

这种设计意味着，镜子记忆不是一个“用户每次都要点开”的显式功能，而是一种在持续使用后逐渐体现价值的底层协作能力。

#### **控制机制与边界**

镜子记忆虽然承担长期认知能力，但其使用边界必须清晰。其控制原则包括：

* 记忆内容并非全部持久化，而是按类型区分生命周期，避免无意义信息长期保留；

* 系统以结构化信息提取和滚动摘要为主，而不是无限累积原始历史内容；

* 用户应能够查看、修改、删除或关闭记忆能力，确保记忆可控、可理解；

* 记忆引入置信度、TTL 和覆盖更新机制，降低错误信息长期污染系统的风险；

* 检索时需控制召回范围和相关性，避免无关记忆污染当前任务；

* 记忆的重点是辅助理解和协作，而不是替代用户决策。

因此，镜子记忆的边界可以概括为：**负责沉淀可复用信息、支撑长期协作和个性化理解，但不以无限存储、无条件记住一切为目标。**

#### **典型价值**

从产品价值上看，镜子记忆解决的不是“系统能不能保存聊天历史”，而是“为什么每次重新开始时，系统都像第一次认识用户”。它所降低的，主要是以下几类成本：

* 在长任务和多轮任务中反复补充上下文的成本；

* 每次重新说明偏好、风格和约束条件的成本；

* 相似任务中从头整理思路和路径的成本；

* Agent 无法形成长期陪伴感和协作积累的成本。

因此，镜子记忆真正补足的，不只是“记住信息”的能力，而是从单次响应走向长期协作的那一段能力缺口。

### 4.3.4 安全执行模式

#### **功能定位**

安全执行模式是 CialloClaw 面向实际操作场景的核心保障功能，负责在 Agent 参与文件处理、命令执行、网页自动化和系统动作时，建立清晰的执行边界、风险分级和恢复机制。其目标不是把产品做成重型安全平台，而是在保持桌面常驻、低打扰协作体验的前提下，让 Agent 的执行建立在“可理解、可确认、可恢复”的基础上。

从产品逻辑上看，安全执行模式不是一个独立的单点工具，而是一层贯穿任务执行全过程的安全控制层。它将“文件写到哪里、动作是否高风险、是否需要确认、执行失败后如何恢复”这些原本对用户不可见的问题，转化为可管理、可查看、可干预的产品能力。

#### **核心输入 / 触发来源**

安全执行模式主要围绕工作区边界、执行动作类型和风险判断三类输入展开。

**1. 工作区与文件边界输入**

系统首先需要明确哪些内容属于用户真正需要保留的成果，哪些内容属于应用运行时使用的数据。用户生成的文档、导出结果和项目成果会进入可见、可管理的工作区；配置、缓存、日志和索引等运行数据则分开存放。这一分层是安全执行模式的第一道边界，也是文件落点透明化的基础。

**2. 执行动作输入**

当 Agent 准备进行实际操作时，系统会识别动作类型与影响范围。典型动作包括：

* 文件读取、写入、删除和覆盖；

* shell / PowerShell / bash 命令执行；

* 依赖安装、编译、测试和 Git 操作；

* 无头浏览器自动化、表单填写和网页交互；

* 系统级配置改动、联网下载和外部发送动作。

**3. 风险评估输入**

系统会结合多个维度对动作进行风险判断，包括动作类型、目标范围、是否跨越沙盒边界、是否可逆、是否涉及凭据或金钱、是否会持久化修改环境，以及是否来源于不可信代码或网站。基于这些信息，系统再决定后续是静默执行、弹窗确认、强制人工确认，还是直接拒绝。

#### **核心输出 / 用户可完成的任务**

基于上述输入，安全执行模式主要输出以下几类结果：

* **安全工作区输出**
  将用户成果文件统一写入可见工作区，避免重要内容进入不可见目录或与缓存数据混杂，降低误删和丢失风险。

* **风险分级输出**
  对不同动作打上绿灯、黄灯、红灯标签，并明确告诉用户哪些操作可以静默执行，哪些需要授权，哪些必须人工确认。

* **隔离执行输出**
  将高风险操作放入执行层沙盒处理，使主程序保持轻量桌面体验，同时降低污染宿主环境的风险。

* **恢复与追踪输出**
  在高风险动作前创建快照或检查点，在执行失败、异常中断或结果不符合预期时，提供恢复、回滚、日志和记录能力。

换言之，安全执行模式不只是“阻止危险动作”，而是尽量把执行过程推进到“可以安全地继续做、可以看懂发生了什么、出了问题还能收回来”这一层。

#### **交互方式**

安全执行模式在大多数情况下并不单独出现，而是嵌入到任务执行流程中，通过轻量但明确的方式让用户感知安全边界。

其典型承接方式包括：

* 在执行前，通过轻提示或确认卡片说明当前动作、影响范围和风险等级；

* 在执行中，通过仪表盘、状态条或任务流展示当前正在进行的操作、终端输出、文件变化或浏览器行为；

* 在执行完成后，展示结果、变更、日志和可恢复入口；

* 在高风险场景下，通过更明确的确认流拦截操作，而不是默认继续执行。

整体交互遵循“低频可见、关键时刻明确介入”的原则。对于低风险动作，系统尽量不增加额外操作负担；对于高风险动作，则优先保证用户可理解、可确认、可中断。

#### **控制机制与边界**

安全执行模式的核心控制机制包括以下几个方面：

* **工作区边界控制**
  所有导出和生成内容默认只写入已设定的工作区，避免越界写入系统目录或散落在用户难以找到的位置。

* **执行面沙盒化**
  主程序运行在宿主机，危险执行放入独立执行层处理。日常任务优先走轻沙盒，高风险任务再进入重沙盒或更强隔离环境。

* **分级授权机制**
  通过红黄绿灯策略引擎，对静默执行、请求授权、强制确认和拒绝进行统一管理。

* **快照与恢复机制**
  在高风险动作前建立快照、检查点或补丁基础，一旦执行失败或结果异常，可进行恢复与回滚。

* **日志与可见性机制**
  将执行动作、终端输出、文件变更、浏览器行为和授权记录纳入可查看的日志与仪表盘，保证执行过程不是黑箱。

同时，安全执行模式也有明确边界。其目标是为 Agent 的有限执行提供可控基础，而不是默认接管全部复杂流程；对于涉及支付、不可逆外部提交、系统级持久修改等高风险动作，系统应以更严格的确认和拒绝策略为主，而不以自动化为默认方向。

#### **典型价值**

从产品价值上看，安全执行模式解决的不是“系统会不会执行”，而是“系统执行时，用户是否敢放心继续交给它”。它主要降低的是以下几类成本：

* 对文件误写、覆盖和越界存储的顾虑；

* 对命令执行、依赖安装和网页自动化污染宿主环境的顾虑；

* 对高风险动作发生时缺少确认、日志和恢复能力的顾虑；

* 对 Agent 执行过程不可见、不可追踪、不可干预的顾虑。

因此，安全执行模式真正补足的，不只是“增加安全功能”，而是让 CialloClaw 在参与实际任务操作时，从“能执行”进一步走向“可控地执行”。
