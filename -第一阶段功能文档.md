## 2. 第一阶段产品定位

第一阶段的 CialloClaw 是一个：

**Windows 本地桌面 Agent**。

它不是重平台，不要求先锁死技术方案，不以重工作区和复杂工作流编排为前提；它以桌面常驻入口为核心，通过低打扰主动协助、任务巡检、长期记忆与可控执行，为用户提供可持续使用的桌面协作能力。

第一阶段强调：

- 本地运行
- Windows 优先
- 桌面常驻入口
- 先提示，再确认，后执行
- 低打扰协作
- 可暂停、可恢复、可终止、可审计
- 执行能力受工作区和轻沙盒约束

第一阶段不锁定最终桌面壳层技术，不在本阶段文档中要求必须采用某一 UI 技术栈。

## 3. 已确认的关键取舍

以下取舍已经明确，后续开发默认按此执行：

### 3.1 技术路线

- 先按功能文档写，不锁技术栈
- 桌面壳层技术方案暂不在本阶段定案
- 只要求整体架构支持后续替换 UI 壳层与 IPC 实现

### 3.2 全局内容触发

第一阶段必须支持：

- **复制内容触发**

第一阶段暂不要求：

- 全局选中文本可靠获取

也就是说：

- `clipboard` 是第一阶段强依赖输入源
- `selection` 可以保留接口和事件，但作为后续增强能力

### 3.3 后台任务追踪展示

第一阶段控制面板中的“后台任务追踪”要求至少支持：

- 任务时间线
- 当前步骤
- 工具调用记录
- 结果摘要
- 审批入口
- 即时暂停
- 恢复
- 终止

第一阶段不要求：

- 展示原始 CoT
- 暴露模型原始思维链

### 3.4 Markdown 任务文件语法

第一阶段只要求识别：

- `- [ ]`
- `- [x]`

第一阶段暂不强制支持：

- 截止时间
- 优先级
- 标签
- 扩展元数据 DSL

这些元数据能力留给后续阶段。

### 3.5 视频总结

第一阶段视频总结只要求支持：

- **在线视频链接输入**

并且加入一条明确交互规则：

- 当用户复制到剪贴板的链接被识别为视频链接时，系统可以主动提示是否生成视频总结
- 只有在用户确认后，才真正启动视频总结任务

第一阶段不要求：

- 本地视频文件导入
- 本地字幕文件导入
- 批量视频处理

### 3.6 沙盒能力

第一阶段只做：

- **轻沙盒**

第一阶段不做：

- 重沙盒
- Hyper-V / Windows Sandbox / WSL2 / Docker 的高危隔离后端

重沙盒后端属于后续阶段。

## 4. 第一阶段功能总览

第一阶段功能收敛为 8 大模块：

1. 桌面入口与交互外壳
2. 场景助手
3. 待办管家 / 任务巡检
4. 镜子记忆 / 长期协作

下面逐项展开。

### 关于“暂停 Agent”的明确要求

暂停 Agent 不是单纯 UI 状态切换，而是：

1. 暂停当前用户指定任务
2. 中断 Agent 执行流程中可中断部分
3. 等待底层执行确认暂停成功
4. 将暂停结果反馈给用户

## 5.2.3 视频总结（第一阶段重要功能）

### 功能目标

围绕在线视频链接，提供视频内容结构化总结。

### 第一阶段必须支持

- 输入源：在线视频链接
- 复制到剪贴板的链接如果被识别为视频链接，系统应主动提示是否生成视频总结
- 用户确认后，启动视频总结任务

### 第一阶段建议处理链路

1. 剪贴板捕获链接
2. URL 识别
3. 视频链接识别
4. 主动提示
5. 用户确认
6. 视频内容采集 / 转写 / 切块 / 总结
7. 输出结构化结果

### 输出结果

建议固定为结构化输出：

- 视频摘要
- 关键点
- 分章节总结
- 可选问答摘要
- 来源链接

### 第一阶段不要求

- 本地视频文件处理
- 批量视频任务
- 自动无提示执行

### 对应架构模块

- `perception/clipboard`

- `cognition/planner`
- `cognition/agents/researcher`
- `action/tools/browser`
- `action/tools/llm`
- `action/tools/video`（建议新增）
- `ui/notification`
- `ui/chat_popup`

### 关键事件

- `clipboard.url.captured`
- `video.link.detected`

# 5.3 待办管家 / 任务巡检

## 5.3.1 任务文件输入

### 功能目标

支持用户通过 Markdown 任务文件进行异步协作。

### 第一阶段必须支持

- 读取指定目录下的 `.md` 任务文件
- 支持一个或多个任务文件/文件夹作为巡检目标
- 支持固定频率巡检
- 支持启动时巡检
- 支持文件变更后立即巡检

### 第一阶段输入语法

只识别：

- `- [ ]`
- `- [x]`

例如：

```md
- [ ] 提交周报
- [x] 整理上周会议纪要
```

### 第一阶段不要求

- 截止时间语法
- 标签语法
- 优先级 DSL
- 多层复杂任务结构

### 对应架构模块

- `perception/files`
- `runtime/scheduler`
- `action/tools/filesystem`
- `cognition/planner`
- `state/task`
- `state/log`

### 关键事件

- `todo.scan.started`
- `todo.file.detected`
- `todo.file.changed`
- `todo.tasks.parsed`

# 5.5 安全卫士 / 执行与恢复

## 5.5.1 工作区边界

### 第一阶段必须支持

- 工作区位置设置
- 打开工作区文件夹
- 重要成果与配置/缓存/日志分开存储
- 导出内容默认进入可见工作区

### 第一阶段明确规则

所有导出、生成、落盘动作必须先经过 `WorkspaceManager`，不能让功能模块任意写盘。

### 对应架构模块

- `action/tools/filesystem`
- `state/config`
- `state/log`
- `action/safety/workspace`（建议新增）

---

## 5.5.2 红黄绿授权系统

### 第一阶段必须支持

- 绿灯：静默执行，仅记录
- 黄灯：执行前询问
- 红灯：强制人工确认

### 风险判断维度

至少考虑：

- 动作类型
- 目标范围
- 是否跨边界
- 是否可逆
- 是否涉及凭据 / 支付 / 身份

### 第一阶段审批入口

审批应可在：

- 轻提示
- 聊天弹层
- 控制面板 / 仪表盘

中完成。

### 对应架构模块

- `action/approval`
- `ui/interactive`
- `state/approval`
- `state/log`

### 关键事件

## 7. 与已有架构文档的对接方式

第一阶段开发时，建议按以下方式对齐既有架构：

### 7.1 状态模型对齐

所有功能都尽量落到以下核心状态对象：

- `Session`
- `Task`
- `Loop`
- `Approval`
- `WorkingMemory`
- `EpisodicMemory`
- `LogEntry`

规则：

- 用户一次实际协作目标，映射为一个 `Task`
- 长任务可带 `Loop`
- 需要确认的动作必须生成 `Approval`
- 过程记录必须进入 `LogEntry`

### 7.2 事件总线对齐

新功能默认通过：

- 新增感知事件
- 新增任务事件
- 新增审批事件
- 新增执行结果事件

来接入，而不是跨层直接调用。

### 7.3 Repository 对齐

第一阶段所有新功能的数据持久化都必须走 repository 抽象层，不允许 UI 或工具层直接操作底层数据库。

### 7.4 扩展接口对齐

未来新增功能，优先采用：

- 新增 Perception 扩展
- 新增 Tool 扩展
- 新增 Agent 扩展
- 新增 UI 扩展

而不是在主干里硬编码。
