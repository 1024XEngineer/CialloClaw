## 1. 产品概述

### 1.1 产品名称

CialloClaw

### 1.2 一句话定义

CialloClaw 是一个**常驻桌面的协作型 Agent**：它不要求用户先进入聊天窗口或工作区，而是直接在当前桌面任务现场中，以低打扰方式识别帮助机会，承接理解、提醒、推进与有限执行。

### 1.3 产品愿景

把 Agent 从“一个你要专门去用的工具”，变成“一个在你当前桌面工作流中顺手出现、顺手帮忙、但不会乱来的协作对象”。真正解决的不是 Agent 能不能做事，而是 Agent 能不能**长期、自然、放心地被使用**。

### 1.4 产品定位

CialloClaw 不是：

- 以聊天窗口为主入口的 AI 助手
- 以 workflow 编排为核心的重平台
- 以强接管和高侵入自动化为卖点的系统

CialloClaw 是：

- 以悬浮球为桌面常驻入口
- 以通话、轻量输入、文本选中、文件拖拽为主要发起方式
- 以气泡和仪表盘承接不同层级协作
- 以长期记忆和可控执行建立信任的桌面协作 Agent

------

## 2. 背景与机会

### 2.1 行业背景

当前 Agent 产品已经证明“从会说到会做”有明确需求，但大量产品仍停留在两类路线：

- 要么是面向执行的底层引擎，能力强但普通用户门槛高
- 要么是围绕工作区展开的平台形态，适合管理任务，但不适合直接嵌入桌面真实工作流

### 2.2 当前问题

用户在真实桌面使用 Agent 时，主要遇到以下问题：

1. 安装和配置门槛高
2. 难形成使用习惯
3. 触发方式不合理，要么过于被动，要么过于打扰
4. 上下文切换成本高
5. 缺乏执行安全感，不敢真正授权

### 2.3 市场机会

当前市场真正缺的是一种产品形态：

- 能直接在桌面任务现场承接协作
- 以轻提示和确认后执行为主
- 从理解、提醒、建议逐步走向轻执行
- 能建立长期习惯学习和协作关系
- 能同时兼顾轻量用户与进阶用户
- 在风险边界、恢复机制、成本治理上清晰可见

------

## 3. 目标用户

### 3.1 核心用户定义

核心用户是一类**任务完成与效率提升导向**的人。他们不关心 AI 技术名词，而关心 AI 能不能在真实场景中稳定、低门槛、低风险地帮自己省事。

### 3.2 用户分层

#### 3.2.1 轻量效率用户

特征：

- 有明确任务
- 不想学习复杂功能
- 希望尽快见效

诉求：

- 开箱即用
- 少打字
- 少配置
- 当前页面附近快速得到帮助

#### 3.2.2 进阶效率用户

特征：

- 希望将 AI 纳入长期工作流
- 愿意持续使用
- 关注流程沉淀与长期习惯

诉求：

- 任务巡检
- 习惯学习
- 长期记忆
- 更深层的协作能力

#### 3.2.3 谨慎型效率用户

特征：

- 对权限、成本、本地性、恢复能力敏感
- 愿意尝试，但不愿承担出错代价

诉求：

- 工作区边界清晰
- 高风险动作可确认
- 失败可恢复
- 成本可见可控

------

## 4. 用户痛点

### 4.1 上手门槛高

现有产品常需要用户先安装、配置、理解插件、理解工作区或工具接入逻辑，普通用户很难快速进入可用状态。

### 4.2 难形成使用习惯

即便已经装好，用户在真实工作中也未必会主动想起 Agent，因为当前主流使用路径仍然是“先想到它，再进入它”。

### 4.3 触发不合理

纯被动等待无法在合适时机出现；强提醒又容易打断用户。用户需要的是“低打扰的主动协助”，而不是持续冒泡。

### 4.4 上下文切换成本高

用户在网页、文档、聊天软件、文件夹、代码界面之间反复来回，很多时间浪费在复制、整理、解释上下文上。

### 4.5 执行可控性不足

用户担心 Agent 改错文件、越权执行、写入位置不明确、失败无法恢复，导致只敢把 Agent 当建议工具，而不敢把它当协作者。

------

## 5. 产品目标

### 5.1 总目标

打造一个可以长期常驻在桌面上、低打扰地介入任务现场，并在可控边界内承接理解、提醒、推进和有限执行的协作型 Agent。

### 5.2 用户价值目标

1. 减少用户表达成本
2. 减少上下文补充成本
3. 减少工具切换成本
4. 提升任务推进效率
5. 降低授权和执行顾虑

### 5.3 产品能力目标

1. 建立悬浮球常驻入口
2. 建立通话优先、文字补充的轻交互体系
3. 建立围绕任务现场的场景助手能力
4. 建立任务巡检与待办推进能力
5. 建立镜子记忆能力
6. 建立安全卫士与治理能力

### 5.4 差异化能力目标

除基础协作能力外，CialloClaw 还应建立以下差异化能力目标：

1. **建立多生态插件与多模型接入能力**
   通过插件化感知包和多模型配置，使系统既能服务开发场景，也能服务日常办公、信息处理与娱乐环境。
2. **建立自动化任务与定时任务能力**
   支持 cron 定时任务与 heartbeat 周期巡检，使系统不仅能够响应用户即时输入，也能主动执行例行任务、提醒和汇总任务。
3. **建立屏幕感知与硬件感知能力**
   让系统不只理解文本输入，也能围绕页面停留、选中文本、异常界面、硬件资源变化等真实桌面信号识别帮助机会。
4. **建立执行面沙盒化能力**
   保证产品开箱即用的同时，把高风险执行隔离到受控环境中完成。
5. **建立中断、恢复、回滚与成本熔断能力**
   使用户在授权系统执行时，具备暂停、恢复、回滚、查看影响范围和控制成本的能力，从而建立长期信任。

## 6. 产品原则

### 6.1 默认常驻，按需展开

产品应始终可触达，但不应长期抢占用户注意力。

### 6.2 通话优先，文字补充

通话是降低输入门槛的主方式，文字输入保留为补充能力，用于纠偏、补充和安静场景。

### 6.3 上下文优先，聊天后置

优先围绕当前页面、选中内容、文件、错误信息、任务状态等上下文承接需求，而不是要求用户先进入聊天窗口把需求讲完整。

### 6.4 先提示，再确认，后执行

系统先识别帮助机会，再以轻量方式提示；用户确认后才进入更深层协作或执行。

### 6.5 高风险动作必须可控

所有涉及高风险执行的行为都必须具备边界说明、授权确认、日志记录和恢复能力。

### 6.6 开箱即用优先

CialloClaw 的基础能力必须做到安装后即可运行，不应要求用户在首次使用前先安装 Docker、配置虚拟化环境或完成复杂依赖准备。

产品应采用“两阶段能力启用”策略：

1. **基础能力安装即用**
   包括桌面入口、UI、场景理解、文件阅读、网页搜索、轻执行、基础日志与治理能力。
2. **高风险能力按需初始化**
   当用户首次触发高风险任务时，再检测并初始化更强隔离能力。

这一原则的目标，是确保产品“先可用，再增强”，而不是“先配置，后尝试”。

## 7. 产品形态与整体交互

### 7.1 主要组成

CialloClaw 采用“**桌面常驻入口 + 轻量承接层 + 仪表盘工作台 + 后台能力系统**”的产品形态。它不是一个要求用户长期停留其中操作的独立平台，而是一套嵌入现有桌面工作流中的协作系统。产品层面主要由以下四部分组成：

#### 7.1.1 悬浮球

悬浮球是默认常驻入口，也是用户感知 Agent 存在感的核心锚点。其设计目标不是“显眼”，而是“始终可触达、低摩擦、不妨碍工作”。

悬浮球承担的角色包括：

- 用户发起协作的第一入口
- 系统表达当前是否可协作的状态锚点
- 当前任务对象已识别时的承接入口
- 通话模式的唤起点
- 悬停层与轻量输入的挂载点
- 文件拖拽与文本选中后的目标对象

悬浮球应具备以下特性：

1. **低存在感待机**
   默认以弱提醒形态停留，不持续吸引注意力，不做高频动画。
2. **状态可见但不抢焦点**
   系统若识别到帮助机会，可通过形态微变化、亮度变化、轻微呼吸感等方式提示，但不得频繁弹出独立强提醒。
3. **始终可被接近**
   用户无需切换应用或打开窗口，只需单击、双击、长按、悬停、拖拽即可开始交互。
4. **可承接明确任务对象**
   当用户已经选中文本或拖入文件时，悬浮球从“通用入口”切换为“当前任务对象入口”。

悬浮球在产品中的意义，不是一个图标，而是“**让用户在当前桌面任务流中随手把任务交给系统**”的最短路径。

#### 7.1.2 语音

语音是 CialloClaw 的主要表达入口，用于降低用户输入门槛，减少“组织 prompt”的成本。

在产品策略上，语音优先于长文本输入，但并不替代文字输入；它更适合作为高频、轻负担、任务现场式的表达方式。

语音的主要职责包括：

- 承接即时需求表达
- 在用户不想打字或不方便打字时替代文字输入
- 与当前屏幕上下文共同构成完整任务描述
- 支持短时表达和持续表达两种模式
- 支持用户在系统回应过程中打断与补充

语音在体验上应满足以下原则：

1. **唤起简单**
   通过长按悬浮球唤起，不额外增加按钮层级。
2. **表达直接**
   用户不需要先进入会话页，按住就可以说。
3. **状态清楚**
   用户必须能明确知道当前是“正在听”“已锁定通话”“已取消”“正在处理”。
4. **与上下文共用一条链路**
   语音不是单独通道，而是与鼠标指向、当前页面、文本选中、拖入文件一起构成任务输入。

#### 7.1.3 气泡

气泡是悬浮球附近的轻量承接区。它不等于聊天窗口，也不承担长期历史记录功能，而是负责**状态反馈、意图确认、短结果返回和下一步建议**。

气泡主要承接以下内容：

- “我理解你可能想做什么”的判断
- “当前正在做什么”的状态反馈
- 适合当前任务现场直接消费的短结果
- 后续建议动作，例如“继续提炼”“写入文档”“查看详情”

气泡应遵循以下产品原则：

1. **默认轻量，拒绝堆叠成连续内容**
   气泡首先服务于当前任务对象，而不是变成完整聊天线程或独立对话框。
2. **以当前任务为中心**
   一个气泡对应一轮明确的任务承接，而不是无限串联所有历史内容。
3. **结果超出气泡承载范围时及时分流**
   长文本结果进入文档，结构化结果进入结果页，连续任务进入任务详情。
4. **生命周期可控**
   气泡要支持自动淡出、隐藏、恢复、置顶和删除，既不永久占屏，也不应突然消失。

#### 7.1.4 仪表盘

仪表盘是低频但完整的工作台。

它不是默认主入口，而是在以下场景中承担系统级承接：

- 用户希望查看当前任务状态
- 用户希望查看今日待办与巡检结果
- 用户希望回看记忆、日报、画像
- 用户希望查看风险、授权、恢复与费用
- 用户需要在一个集中区域理解“系统现在在关注什么”

仪表盘不是一个重平台，其职责不是“把用户拉进来长期停留”，而是：

- 在需要时给出完整上下文
- 在低频场景下集中查看全局状态
- 为持续任务和治理信息提供承接界面
- 为后续更复杂的协作提供收口点

### 7.2 核心入口方式

#### 7.2.1 左键单击

左键单击是悬浮球最轻量的接近动作。

它表达的不是“立即进入深层协作”，而是“我现在可能需要帮助”。

左键单击在不同状态下有不同含义：

**A. 默认待机状态下**

- 表达用户接近悬浮球
- 可触发轻量承接或可操作提示态
- 不直接等同于打开重界面
- 在无明确任务对象、无待处理事项、无当前事件触发时，单击可先触发轻量接近反馈，而不直接进入深层协作
- 该反馈可包含轻微状态变化、微表情或短暂视觉回应，用于表达系统已感知到用户接近并准备承接

**B. 当前任务对象已明确时**

- 承接已选中的文本
- 承接已拖入的文件
- 进入意图确认流
- 在必要时触发气泡层而不是立即打开仪表盘

**C. 与双击的区分**

- 单击：轻量接近 / 当前任务承接
- 双击：明确打开仪表盘

因此，左键单击的产品角色是“**低成本发起协作**”，不是“打开应用”。

#### 7.2.2 左键双击

左键双击是明确进入完整工作台的动作，语义必须稳定。它的含义固定为：**打开仪表盘**。

双击的产品目标有两个：

1. 给用户一个稳定的“查看全局状态”的入口
2. 与单击的轻量接近动作形成明确分工

双击进入仪表盘后，用户应能够立即看到：

- 当前任务状态
- 近期待办与巡检结果
- 镜子记忆概要
- 安全与治理状态
- 快捷键帮助

双击不应触发以下行为：

- 直接进入聊天会话
- 直接执行某个任务
- 打开独立重窗口并抢占系统焦点

双击的核心价值是：**让用户在需要完整上下文时，始终知道去哪里看。**

#### 7.2.3 左键长按

左键长按是语音主入口。这是当前产品里最重要的“低负担表达动作”。

完整规则如下：

**A. 唤起**

- 用户长按悬浮球后，系统进入收音态
- 悬浮球需给出明确状态变化，表示“正在听”

**B. 表达**

- 用户持续按住即可进行短时表达
- 适合一句话或一段简短口述需求

**C. 松开**

- 松开即结束本轮输入
- 系统将语音内容与当前上下文一并送入后续处理链路

**D. 上滑锁定**

- 长按过程中上滑，进入持续通话模式
- 适用于连续补充背景、不断修正需求、与系统来回沟通的场景

**E. 下滑取消**

- 长按过程中下滑，取消本次语音输入
- 取消后不进入处理链路，不保留本次表达结果

**F. 交互价值**

长按不是单纯“打开麦克风”，而是让用户在桌面当前任务现场中，以最低负担说出需求，不必先思考输入方式。

#### 7.2.4 鼠标悬停

悬停是“轻量发起协作”的核心入口。

它用于承接两种需求：

1. 用户主动补充一句话
2. 系统基于当前上下文给出可能问题推荐

悬停层展示内容包括：

- 悬浮球下方轻量输入框
- 3 个基于当前上下文的推荐问题

悬停层的设计原则如下：

**A. 只承接轻量表达**
 输入框用于“一句话输入”，而不是长段文字对话。

**B. 主动推荐必须克制**
 推荐问题只在用户悬停于悬浮球且近期活跃时触发，不允许在页面任意位置静止停留时全局冒出。

**C. 输入与推荐并存**
 推荐问题不是替代输入框，而是与输入框同时存在，服务不同用户习惯。

**D. 不应演化为聊天弹窗**
 悬停层必须保持轻量，目的是让用户“顺手开始”，不是让用户“重新进入一个聊天产品”。

悬停在产品架构中的意义是：**降低发起门槛，同时建立适度存在感。**

#### 7.2.5 文件拖拽

文件拖拽是“围绕明确文件对象发起协作”的入口。

当用户把文件拖到悬浮球时，系统默认理解为：用户希望围绕这个文件做某种处理。

可承接的典型任务包括：

- 分析
- 总结
- 提炼
- 改写
- 解释
- 生成摘要
- 生成草稿
- 识别文件中的待办或问题

文件拖拽入口的产品价值在于：

1. 让文件直接成为任务对象
2. 省去“先描述这是什么文件”的上下文补充
3. 使任务从“说出来”变成“拖进去”
4. 提升真实桌面工作流中的协作自然度

文件拖拽后，系统默认不应直接执行，而应：

- 先识别文件类型
- 再判断可能意图
- 再进入确认与承接流程

#### 7.2.6 文本选中

文本选中是局部任务对象识别入口。

它的价值在于：让用户不必复制一段内容再粘贴进聊天框，而是直接围绕当前选中内容开始协作。

适用场景包括：

- 解释
- 翻译
- 总结
- 润色
- 延展
- 提问
- 提炼重点

文本选中后的产品行为原则：

**A. 先进入“可操作提示态”**
 系统应让用户知道：当前选中内容已被识别，可以继续围绕其发起协作。

**B. 不应直接打断**
 仅仅选中文本不应立即弹出重界面或大弹窗。

**C. 后续通过其他动作继续承接**
 例如：

- 再点悬浮球
- 拖动选中文本到悬浮球
- 长按悬浮球后结合语音表达

文本选中入口的本质，是让 CialloClaw 真正进入“当前任务现场”，而不是让用户把内容搬到系统里。

### 7.3 协作闭环

CialloClaw 的整体交互不是单一步骤，而是两条核心闭环。

#### 7.3.1 闭环一：主动输入闭环

**常驻入口 → 用户主动输入 → Agent 输出 → 记忆沉淀**

适用场景：

- 用户已经有明确任务
- 用户主动想要一个结果
- 用户愿意明确发起一次协作

这条闭环的特点是：

- 用户主导开始
- 系统快速承接
- 任务边界清楚
- 结果容易消费和沉淀

典型路径：

- 长按悬浮球说“帮我总结这段内容”
- 悬停输入“写成更正式的邮件”
- 文件拖拽后说“帮我提炼重点”

#### 7.3.2 闭环二：主动协助闭环

**常驻入口 → 场景感知 → 识别帮助机会 → 轻提示 → 用户确认 → 对话承接 / 任务推进 / 有限执行 → 结果反馈与记忆沉淀**

适用场景：

- 用户未主动完整表达需求
- 当前上下文中存在明显帮助机会
- 用户需要提醒、建议、下一步动作，而不是一次性长结果

这条闭环的特点是：

- 系统先发现机会
- 系统不直接抢占
- 用户有确认权
- 最终由更完整的承接层完成结果交付

典型路径：

- 识别到用户停留在一段错误信息上，悬停推荐“解释这个错误”
- 识别到任务文件中有临期事项，任务巡检里提醒“今天要优先处理”
- 识别到系统资源异常，提示“当前 CPU 占用过高，是否查看原因”

#### 7.3.3 两条闭环的分工关系

闭环一强调“**用户主动发起**”，闭环二强调“**系统低打扰协助**”。

二者的关系不是替代，而是互补：

- 用户明确知道自己要什么时，走主动输入闭环
- 用户并不想完整输入，但当前场景值得被帮助时，走主动协助闭环
- 两条闭环最终都要进入统一承接体系：气泡、仪表盘、文档输出、结果页、任务详情

因此，CialloClaw 的产品不是“问答型产品”，也不是“纯提醒型产品”，而是一套能够在**主动输入**与**主动协助**之间平滑切换的桌面协作系统。

## 8. 核心使用场景

### 8.1 内容理解与即时处理

当用户在阅读网页、查看文档、浏览代码、遇到报错或选中一段文字时，系统提供总结、翻译、提炼、解释、下一步建议等帮助。价值在于让处理直接发生在现场，而不是先复制到别的工具里。

### 8.2 任务巡检与任务推进

当用户用 markdown 待办、任务文件或阶段性清单管理事务时，系统帮助识别临期事项、优先级和下一步动作，并可打开资料或生成草稿。

### 8.3 异常发现与问题排查

当用户遇到错误信息、异常界面、停留过久或重复失败时，系统提供原因分析和下一步建议。

### 8.4 视频理解与长内容总结

当用户提供视频链接，或在本地 / 浏览器中观看较长视频内容后，希望快速获得重点摘要、结构化笔记和下一步建议时，系统应提供视频理解与总结能力。

#### 8.4.1 适用场景

- 教程视频
- 产品演示视频
- 软件操作教学
- PPT 讲解
- 代码教学
- 带图表或演示画面的分享内容

#### 8.4.2 用户价值

1. 降低长视频内容消费成本
2. 将“看完再整理”变成“边看边可沉淀”
3. 支持从视频直接生成摘要、清单、任务或草稿
4. 让视频内容进入镜子记忆与长期协作体系

#### 8.4.3 输入来源

- 用户提供的视频链接
- 视频平台原始字幕或自动字幕
- 视频音频流
- 视频画面关键帧
- 标题、简介、封面、时长、上传者等元数据
- 用户在浏览器实时观看时的录屏 / 音频输入

#### 8.4.4 能力要求

1. 优先利用视频平台已有字幕，降低转写成本。
2. 无字幕时，对视频音频做转写。
3. 对教程、演示、PPT 类视频支持关键帧抽取与视觉补充理解。
4. 长视频支持切片、分段处理与阶段性总结。
5. 输出应支持短摘要、结构化重点、时间线重点、可编辑文档草稿等多种形态。

#### 8.4.5 输出形式

**A. 气泡短结果**

适用于：

- 一句话总结
- 三到五条重点
- 当前视频主题判断
- 下一步建议

**B. 文档输出**

适用于：

- 完整视频摘要
- 学习笔记
- 会议纪要草稿
- 可继续编辑的长文本结果

**C. 结果页输出**

适用于：

- 按时间轴整理的重点内容
- 关键片段索引
- 结构化知识点列表

#### 8.4.6 边界说明

当前阶段的视频总结目标是“降低用户理解和整理成本”，而不是完整的视频制作或深度视频编辑能力。

## 9. 核心功能模块

### 9.1 场景助手

#### 9.1.1 功能定义

场景助手是 CialloClaw 最核心、最前台的能力模块。

它负责围绕当前任务环境提供即时帮助，让系统能够在“用户正在做事”的过程中承接理解、处理和建议。

它不要求用户先完整补充上下文，而是优先基于当前任务现场识别有意义的输入线索，包括：

- 当前页面内容
- 选中的文本
- 拖入的文件
- 程序状态
- 错误弹窗
- 终端报错
- 系统资源状态
- 网络状态
- 当前停留内容

场景助手的本质，不是“回答问题”，而是：**围绕当前任务环境，降低用户理解、处理和推进任务的成本。**

#### 9.1.2 典型能力

**A. 内容理解类**

- 对网页文章、文档内容、代码片段进行总结
- 对一段内容做重点提炼
- 对专业表达做通俗解释
- 对英文或其他语言内容做翻译与解释
- 对长段内容生成简明版摘要

**B. 问题排查类**

- 对错误信息、异常界面、终端报错做原因分析
- 对系统资源异常（高 CPU、高内存、低电量）给出提示和建议
- 对重复失败场景提供可能原因和排查方向

**C. 任务推进类**

- 结合当前页面内容给出下一步建议
- 根据文档内容生成待办草稿
- 根据会议资料生成“会前检查清单”
- 根据已有上下文生成回复草稿、汇报草稿、说明草稿

**D. 状态提醒类**

- 系统资源异常提醒
- 设备状态提醒
- 网络波动提醒
- 当前任务长时间无进展时的提示

#### 9.1.3 用户价值

场景助手给用户创造的价值主要有四点：

1. **减少复制粘贴和切换工具的成本**
2. **减少为了让 AI 理解上下文而做额外解释的成本**
3. **提升从“看懂”到“下一步该做什么”的速度**
4. **让帮助发生在当前任务现场，而不是任务现场之外**

#### 9.1.4 输入来源

场景助手可使用的输入来源包括：

**A. 显式输入**

- 用户语音输入
- 悬停输入框中的短句输入
- 用户点击的推荐问题

**B. 明确任务对象**

- 文本选中内容
- 文件拖拽内容
- 用户当前打开页面或当前窗口的主要可视内容

**C. 环境与状态输入**

- 错误信息
- 程序异常状态
- 硬件 / 系统状态
- 长时间停留或重复失败等行为信号

#### 9.1.5 补充：场景感知信号

除显式输入和明确任务对象外，场景助手还可基于以下信号识别帮助机会：

- 用户复制或选中的文本内容
- 用户选中的图片区域或片段信息
- 用户在某页面长时间停留
- 用户在异常界面、错误弹窗或重复失败场景中反复停留
- 剪贴板内容变化
- 当前前台应用类别
- 系统空闲 / 忙碌状态
- 当前浏览内容与近期任务之间的潜在关联

这些信号的价值，不在于替代用户表达，而在于帮助系统在真实桌面任务现场更早识别“现在可能值得帮助”。

#### 9.1.6 输出形式

场景助手的输出应按结果类型自动分层：

**A. 气泡短结果**

适用于：

- 一句话解释
- 简短翻译
- 三点摘要
- 明确结论
- 下一步建议

**B. 文档输出**

适用于：

- 较长总结
- 回复草稿
- 会议准备清单
- 方案说明
- 可继续编辑内容

**C. 结果页 / 文件输出**

适用于：

- 多条可浏览结果
- 结构化信息集合
- 文件类产物

**D. 仪表盘承接**

适用于：

- 需要持续查看状态的任务
- 需要和任务状态、巡检、风险信息一起理解的任务

#### 9.1.7 交互要求

场景助手必须遵循以下交互顺序：

1. **先识别任务对象**
2. **再判断可能意图**
3. **必要时通过气泡进行确认**
4. **确认后返回结果或继续展开**
5. **结果超过轻量范围时进入正式交付**

产品上不允许直接跳到“重聊天窗口”，而应优先通过悬浮球附近的轻量交互完成承接。

#### 9.1.8 主动协助触发约束

场景助手的主动协助必须遵循以下规则：

1. **优先使用强意图信号**
   例如：选中、复制、上传、打开任务文件、错误弹窗出现、截止临近。
2. **弱信号只用于静默建议**
   例如：长时间停留、频繁切换窗口、页面来回切换。这类信号只适合作为轻提示依据，不应触发强提醒。
3. **默认提示形态轻量**
   主动协助默认通过悬浮球状态变化、轻提示卡片、托盘提示、右键扩展入口表达，不使用高侵入弹窗打断用户任务流。
4. **执行前必须进入确认流**
   如涉及改文件、发消息、执行命令、系统动作等，必须经过授权确认，不可因主动协助而自动越权执行。

#### 9.1.9 边界说明

场景助手当前阶段不负责：

- 长链路任务编排
- 高风险自动执行
- 团队级任务协同
- 复杂多步骤自动化闭环

它负责的是：**高频、即时、围绕现场的理解与推进。**

#### 9.1.10 验收标准

1. 用户能围绕页面、文本、文件快速发起请求
2. 用户无需手动补充完整上下文也能得到可用结果
3. 短结果能在气泡中直接消费
4. 长结果能自动分流到正式交付层
5. 用户能理解它是在“围绕当前任务帮忙”，而不是在“进行普通聊天”

### 9.2 待办管家

#### 9.2.1 功能定义

待办管家负责承接任务文件、待办清单、时间线索与周期性巡检。

它的目标不是简单“记录任务”，而是让任务从“被记下”走向“被推进”。

待办管家的本质能力包括：

- 识别任务结构
- 判断任务优先级
- 提示临近事项
- 给出下一步动作建议
- 在需要时打开资料、网页、目录或生成草稿

它是 CialloClaw 从“看懂当前内容”走向“推动当前事务”的关键模块。

#### 9.2.2 异步协作定位

待办管家不仅是任务巡检模块，也是 Agent 工作流产品化的核心入口之一。用户不一定每次都要通过 chat 明确发起需求，也可以通过 `.md` 任务文件、便签、周期巡检和规则触发实现异步协作。

这意味着待办管家不仅承担“记录任务”的职责，更承担：

- 承接文件型任务输入
- 从静态任务清单中提取下一步动作
- 把重复事务转化为可提醒、可推进、可模板化的流程
- 在用户尚未主动发起时，以低打扰方式帮助推进事务

#### 9.2.3 典型能力

**A. 任务文件解析**

- 读取指定 `.md` 任务文件
- 识别任务项
- 识别日期、优先级、状态、标签
- 识别重复事项和时间线索

**B. 今日任务摘要**

- 汇总当天应处理事项
- 展示当前最值得先做的事项
- 区分“快到时间”和“尚未到时间”

**C. 临期提醒与排序**

- 提醒快到期任务
- 区分逾期、即将到来、后续安排
- 识别哪些事项应该优先推进

**D. 下一步动作承接**

- 打开相关文件、网页、目录
- 生成下一步草稿
- 提示当前阻塞点或前置条件

#### 9.2.4 自动协助能力

在任务巡检基础上，待办管家还可提供：

- 根据任务内容打开对应文件、目录、网页或资料
- 根据任务关键词触发固定动作或打开固定资源
- 为任务生成邮件草稿、消息草稿、会议前清单
- 对连续重复任务提示是否适合沉淀为模板流程或自动流程

#### 9.2.5 用户价值

待办管家的核心价值不是“告诉用户还有哪些任务”，而是：

1. 帮用户缩短“从知道有任务”到“开始推进任务”的路径
2. 帮用户识别真正该优先做的事
3. 降低事项散落在文件、目录、时间线中的管理成本
4. 让任务管理从静态清单变成动态推进

#### 9.2.6 输入来源

- 指定任务文件
- 用户主动录入的时间信息
- 任务状态变更
- 周期性巡检规则
- 用户对事项的完成 / 取消 / 跳过动作

#### 9.2.7 输出形式

**A. 任务状态模块**
 展示当前正在进行或已结束的任务：

- 名称
- 状态
- 当前进展
- 安全摘要
- 产出文件

**B. 任务巡检模块**
 展示：

- 近期要做
- 后续安排
- 重复事项
- 已结束事项

#### 9.2.8 仪表盘承接逻辑

**任务状态** 更偏“执行中 / 刚完成”的当前任务视角。
 **任务巡检** 更偏“待推进 / 未来安排”的任务管理视角。

这两个模块必须严格分工：

- 不把所有内容都堆到一个列表里
- 不让用户混淆“当前任务”和“未来事项”

#### 9.2.9 规则说明

1. 已结束任务展示近 3 天 / 7 天 / 更多
2. 未完成任务默认展示全部
3. 重复事项展示规则本身，不与单次实例混淆
4. 当重复事项已进入当天场景时，应在“近期要做”中作为具体实例展示
5. 任务状态与巡检提醒都应遵循低打扰原则，不高频打断

#### 9.2.10 边界说明

待办管家当前不负责：

- 项目管理级复杂流程编排
- 团队多人同步
- 自动完成整条执行链路

它主要负责“任务可见、优先级清楚、下一步明确”。

#### 9.2.11 验收标准

1. 能从指定任务文件中正确抽取任务项与时间信息
2. 用户能一眼区分近期事项、后续事项、重复事项和已结束事项
3. 用户能从任务摘要中直接知道下一步怎么做
4. 用户能通过按钮快速完成“完成 / 跳过 / 编辑 / 打开资料”等动作
5. 用户感受到的是“被推进”，而不是“多了一个任务清单”

### 9.3 镜子记忆

#### 9.3.1 功能定义

镜子记忆是 CialloClaw 面向长期协作的核心能力。

它不是简单的聊天记录模块，而是系统对用户偏好、任务状态和工作方式的持续理解与沉淀机制。

镜子记忆承担三层职责：

1. **短期记忆**：支撑连续任务理解
2. **长期记忆**：沉淀偏好、习惯、阶段性信息
3. **镜子总结**：让长期价值被用户显性看见

#### 9.3.2 记忆系统定位

镜子记忆不是聊天记录缓存，而是 Agent 的长期认知系统。其目标不是“记住更多原始内容”，而是对用户交互、任务状态、偏好与行为信号进行结构化沉淀、压缩存储与智能召回，以支撑连续协作、个性化理解、主动服务和长期陪伴。

#### 9.3.3 主要作用

**A. 多轮上下文承接**
 用户不需要手动指定要带哪些历史信息，系统自动检索与当前任务相关的过往记录。

**B. 历史对话概要查看**
 用户可以在仪表盘中看到以往协作的摘要，而不是一长串原始聊天记录。

**C. 日报生成**
 系统记录用户当天做了什么、完成了哪些内容、涉及了哪些权限或动作，用于复盘和查看。

**D. 用户画像沉淀**
 逐步形成用户偏好，例如：

- 更喜欢什么表达风格
- 更习惯怎样的输出形式
- 常处理什么类型任务
- 更能接受怎样的提示频率

#### 9.3.4 用户价值

镜子记忆的价值不在“记住很多”，而在：

1. 让用户少重复解释
2. 让相似任务越来越顺手
3. 让系统逐渐贴近用户偏好
4. 让长期协作的价值可见，而不是发生在黑箱里

#### 9.3.5 输入来源

- 用户主动输入与系统对话
- 任务状态变化
- 周期性总结结果
- 用户对结果的偏好反馈
- 用户对主动提示的接受 / 忽略 / 调整行为

#### 9.3.6 记忆分层

镜子记忆应至少分为两层：

**A. 短期记忆**

用于承接当前任务和近期任务，包括：

- 最近对话上下文
- 当前任务目标
- 最近几轮操作结果
- 程序状态与硬件感知事件
- 当前阶段未解决问题

**B. 长期记忆**

用于沉淀长期可复用信息，包括：

- 用户偏好
- 表达风格偏好
- 常见任务类型
- 提示频率偏好
- 周期任务规律
- 阶段性用户画像

#### 9.3.7 输出形式

**A. 仪表盘镜子模块**
 展示：

- 历史对话概要
- 日报
- 用户画像

**B. 个性化承接**
 在后续任务中体现为：

- 更准确的意图猜测
- 更贴近偏好的输出风格
- 更合适的提醒节奏

**C. 总结输出**
 在阶段性节点向用户展示“最近你做了什么”“你有哪些习惯被学到”

#### 9.3.8 产品要求

1. 默认后台运行，不高频显式打扰
2. 对用户可见，而不是完全黑箱
3. 支持查看、管理和必要时跳转到本地存储路径
4. 不把镜子做成普通聊天历史页，而要强调“摘要、阶段总结、偏好沉淀”

#### 9.3.9 记忆系统规则

1. 记忆默认以本地存储为前提，不以上云为默认条件。
2. 用户应可一键开启或关闭长期记忆。
3. 记忆保存的重点应是**结构化信息**，而不是完整原始文本堆积。
4. 长期记忆应具备生命周期、权重、置信度与覆盖更新机制，避免记忆污染。
5. 记忆召回应基于当前任务相关性，而不是机械带入历史内容。
6. 用户应能查看、删除、修改或清空已沉淀的长期记忆。

#### 9.3.10 任务级记忆

除用户偏好记忆外，系统还应具备任务级记忆，用于记录：

- 当前任务目标
- 已完成步骤
- 当前进度
- 关键约束
- 下一步动作
- 尚未解决的问题

任务级记忆的目标是让系统“记住任务做到哪里”，而不是简单“记住之前说过什么”。

#### 9.3.11 周期性总结机制

系统应根据短期记忆与长期记忆，在以下条件下触发阶段性总结：

- 上下文达到一定规模
- 到达固定时间段
- 任务阶段自然结束
- 周期性日报 / 周报生成时

总结产物可包括：

- 日报
- 周报
- 历史协作摘要
- 用户画像更新
- 阶段性习惯总结

这些总结应对用户可见、可验证、可管理，而不是完全黑箱发生。

#### 9.3.12 边界说明

镜子记忆当前阶段不负责：

- 自动替用户长期写作
- 完整人格系统
- 复杂知识库管理
- 团队共享记忆

它首先服务于“**连续任务承接**”和“**长期协作习惯形成**”。

#### 9.3.13 验收标准

1. 用户能理解镜子不是聊天记录堆叠，而是长期协作入口
2. 用户能在新任务中感受到记忆效果
3. 用户能在仪表盘中清楚查看历史概要、日报、画像
4. 用户不会因为镜子不可见而觉得系统在做黑箱推断

### 9.4 安全卫士

#### 9.4.1 功能定义

安全卫士是 CialloClaw 的治理核心。

它负责在 Agent 参与文件处理、命令执行、网页自动化和系统动作时，建立：

- 清晰边界
- 风险分级
- 授权确认
- 影响可见
- 恢复能力
- 成本治理

安全卫士的目标，不是让系统“更保守”，而是让用户敢于把有限执行权交出来。

#### 9.4.2 核心能力

**A. 工作区边界控制**

- 明确可写入范围
- 明确是否允许工作区外写入
- 明确影响的是哪些文件、网页、应用

**B. 执行面沙盒化**

- 高风险动作不直接污染宿主环境
- 在隔离环境中先处理或预演

**C. 红黄绿灯授权系统**

- 绿灯：静默执行，仅记录
- 黄灯：执行前询问
- 红灯：强制人工确认

**D. 一键中断与恢复**

- 任务可暂停 / 取消
- 失败后可回退
- 关键动作前建立恢复点

**E. Token 与费用治理**

- 展示当前任务输入 / 输出 Token
- 展示当前任务费用
- 展示今日累计 Token 与费用
- 展示单次任务和每日上限
- 展示是否触发超限提醒和降级

**F. 审计与日志**

- 记录文件操作
- 记录网页操作
- 记录命令操作
- 记录系统动作
- 记录执行结果和影响范围

#### 9.4.3 执行面沙盒化补充

CialloClaw 采用“主程序跑在宿主机，危险执行进入沙盒”的执行面沙盒化策略，而不是整应用完全沙盒化。

#### 分层如下

**A. 轻沙盒**

用于大多数日常任务，例如：

- shell / PowerShell / bash 命令
- 编译 / 测试
- Git 操作
- 依赖安装
- 文件改动预演
- 本地网页抓取
- 无头浏览器自动化

**B. 重沙盒**

用于高风险或不可信任务，例如：

- 运行不明来源代码
- 下载并执行第三方脚本
- 安装系统级工具
- 浏览器自动登录 / 自动表单 / 支付前流程
- 陌生仓库 setup 脚本执行
- 可能污染宿主机环境的依赖安装

这一策略的目标是：既不牺牲“开箱即用”，也不把高风险执行直接暴露给宿主环境。

#### 9.4.4 默认工作区与写入边界

安全卫士应建立明确的 Workspace 写入边界：

1. 用户正式成果文件与应用运行数据必须分层存储。
2. 导出的文档、结果文件、生成产物默认写入用户可见的工作区。
3. 配置、缓存、日志等运行数据写入系统标准配置 / 缓存目录。
4. 所有写入操作必须先经过 Workspace Manager 做路径规范化与边界校验。
5. 若发生路径穿越、越界写入或非授权目录写入，应直接拒绝执行。
6. 写入应优先采用临时文件 + 原子替换，避免半写失败和损坏文件。

#### 9.4.5 授权策略引擎

风险等级不应只按动作类型静态定义，而应结合以下维度综合判断：

- 动作类型：读 / 写 / 删 / 执行 / 网络 / 外设
- 目标范围：任务目录 / 项目目录 / 用户目录 / 系统目录
- 是否跨边界：沙盒内 / 沙盒外 / 宿主机全局
- 是否可逆：可回滚 / 不可回滚
- 是否涉及金钱、身份或凭据
- 是否会持久化修改环境
- 是否来源于不可信代码或网站
- 是否需要联网下载执行

最终输出红 / 黄 / 绿灯风险等级。

#### 9.4.6 一键中断机制

安全卫士必须支持一键中断能力，使用户在系统执行过程中可随时叫停。

一键中断包括三层能力：

**A. 立即停**

- 阻止新工具调用继续发出
- 当前任务进入中断态
- UI 立刻反馈“已停止接收新动作”

**B. 优雅关闭**

根据工具类型分别中断：

- 浏览器自动化
- Shell / 命令执行
- 文件写入
- GUI 自动化
- 网络外发行为

不允许简单粗暴 kill 导致状态不一致或文件损坏。

**C. 可恢复 / 可回滚**

中断后用户应能看到：

- 停在第几步
- 哪些动作已完成
- 哪些动作未开始
- 哪些动作可以恢复继续
- 哪些动作需要回滚

#### 9.4.7 三层回滚机制

1. **任务工作区级回滚**
   所有高风险改动优先发生在任务工作区中，最终通过 patch / diff / sync plan 回写。
2. **项目级快照回滚**
   对项目目录建立 checkpoint，用于恢复 AI 修改前状态。
3. **系统或平台级恢复能力**
   在必要场景下支持更高层级的恢复与审计入口。

#### 9.4.8 用户价值

安全卫士给用户的价值主要有五点：

1. 知道系统在做什么
2. 知道为什么需要确认
3. 知道影响到哪里
4. 出错后知道怎么恢复
5. 知道这件事花了多少成本

#### 9.4.9 仪表盘展示

安全卫士在仪表盘中至少包括两层信息：

**A. 预览态**

- 当前安全状态
- 待确认操作数
- 最近异常
- 最近恢复点
- 当前任务 Token / 金额
- 今日累计 Token / 金额

**B. 详细态**

- 待确认 / 被拦截操作
- 授权记录
- 审计记录
- 影响范围
- 恢复与回滚
- 规则与权限管理
- Token 与费用
- 执行日志

#### 9.4.10 规则说明

1. 所有高风险动作默认不能静默通过
2. 授权必须能区分本次允许与规则级允许
3. 恢复点必须能看到其关联任务和影响对象
4. 安全卫士必须和任务状态联动，当前任务页中可查看安全摘要
5. 成本治理不只做统计，还要能明确表达是否接近上限

#### 9.4.11 边界说明

安全卫士当前不等于完整企业级安全系统。其第一阶段目标是：**把边界、确认、日志、恢复和成本做成用户可理解的治理入口。**

#### 9.4.12 验收标准

1. 用户能明确区分正常、待确认、异常、可恢复等状态
2. 用户能查看当前任务的风险、影响和消耗
3. 用户能理解绿黄红三级动作差异
4. 用户能找到恢复入口
5. 用户对“是否可控”的主观感知明显提高

### 9.5 平台扩展与自动化引擎

#### 9.5.1 功能定义

平台扩展与自动化引擎负责为 CialloClaw 提供可扩展能力边界和轻量自动化能力。它使产品不只停留在“理解与建议”，还能够在可控边界内承接桌面办公、信息处理和轻开发场景中的例行任务与多生态协作。

#### 9.5.2 核心能力

**A. 多生态插件能力**

- 支持不同感知包扩展不同场景能力
- 支持围绕开发、办公、日常内容消费等场景扩展能力
- 支持插件启停与权限管理

**B. 多模型配置能力**

- 支持多模型接入
- 支持按任务场景做模型选择与路由
- 支持成本与性能平衡策略

**C. Skills 生态兼容能力**

- 支持兼容社区 Skills 生态
- 支持从外部来源安装技能
- 支持查看和管理已安装技能

**D. 自动化与例行任务能力**

- 支持 cron 定时任务
- 支持 heartbeat 周期巡检
- 支持周期性汇总、提醒和检查任务
- 支持围绕文件、任务、网页与信息流做例行处理

#### 9.5.3 用户价值

1. 同一产品能覆盖更多高频桌面场景
2. 用户不必每次都手动发起任务
3. 例行事务可被系统持续承接
4. 产品从“即时助手”延展为“长期协作 Agent”

#### 9.5.4 边界说明

本模块当前阶段不追求重型 workflow 平台，而强调：

- 轻量
- 可理解
- 可配置
- 可追踪
- 可控边界内的自动化

## 10. 仪表盘设计

### 10.1 仪表盘定位

仪表盘是 CialloClaw 的低频完整工作台。

它不是默认主入口，不承载“所有事情都在里面做完”的重平台职责，而是承担以下角色：

1. **当前状态的总览入口**
2. **持续任务的查看入口**
3. **任务巡检与推进入口**
4. **长期记忆的可见入口**
5. **治理与信任的集中入口**
6. **快捷键和系统使用帮助入口**

因此，仪表盘的定位可以概括为：

> 一个在必要时打开，用来理解“系统现在在关注什么、做到了哪里、还有什么需要我决策”的集中工作台。

### 10.2 仪表盘模块

仪表盘包含 5 个一级模块：

1. 任务状态
2. 任务巡检
3. 镜子功能
4. 安全卫士
5. 快捷键

#### 10.2.1 任务状态

**模块定位**
 承接“当前正在发生的任务”和“刚刚完成的任务”。

**核心目标**
 帮助用户快速看清：

- 当前有哪些任务仍在进行
- 当前任务执行到哪一步
- 哪些任务已结束
- 哪些任务有产出
- 当前任务是否伴随风险或待授权事项

**结构设计**
 分为两个主区：

**A. 未完成**
 展示范围：当前全部未完成任务
 展示内容：

- 任务名称
- 当前状态：暂停 / 等待授权 / 正在进行 / 失败
- 当前进展：具体执行到哪一步
- 记忆与上下文应用情况
- 当前安全摘要
- 操作按钮：暂停 / 继续、取消、修改、查看安全详情

**B. 已结束**
 展示范围：近 3 天 / 7 天 / 更多
 展示内容：

- 任务名称
- 最终状态：已完成 / 已取消
- 关键操作摘要
- 产出物入口（点击打开文件或结果）
- 操作按钮：重启、查看安全详情

**设计原则**

1. 未完成与已结束必须严格分区
2. 任务状态必须直观，不隐藏当前阶段
3. 每条任务都应能快速理解“下一步可做什么”
4. 安全信息在这里先展示摘要，细节再进入安全卫士

#### 10.2.2 任务巡检

**模块定位**
 承接“尚未完成但需要持续关注的事项”。

**核心目标**
 帮助用户理解：

- 今天和最近该做什么
- 哪些事项暂时不用动
- 哪些事项会重复出现
- 哪些事项已经处理完

**结构设计**
 分为四个视图：

**A. 近期要做**

适用于快到时间、今天要做、最近几天要做的事项。
 包含：

- 事件名称
- 状态：即将到来 / 已逾期
- 时间信息：开始 / 截止 / 剩余时间
- 事件类型：单次 / 重复实例
- 备注或 Agent 建议
- 操作按钮：标记完成、取消 / 跳过本次、编辑、打开相关资料

**B. 后续安排**

适用于已安排但暂时无需处理的事项。
 包含：

- 事件名称
- 状态：未到时间
- 时间信息
- 前置条件
- 操作按钮：提前到近期要做、编辑、取消、打开相关资料

**C. 重复事项**

适用于每天 / 每周 / 每月等循环事项。
 包含：

- 事件名称
- 重复规则
- 下次发生时间
- 最近一次状态
- 生效范围
- 相关资料
- 操作按钮：暂停 / 开启重复、修改、取消整个重复事项、打开资料

**D. 已结束**

适用于已完成、已取消事项。
 包含：

- 事件名称
- 最终状态
- 原定时间
- 结束时间
- 事件类型
- 操作按钮：恢复为未完成、删除记录

**设计原则**

1. “近期要做”与“后续安排”必须清晰区分
2. “重复事项”展示的是规则本身，不是某次实例
3. 已结束事项只承接回看与恢复，不继续参与今日判断
4. 模块目标是推进，不是堆砌任务列表

#### 10.2.3 镜子功能

**模块定位**
 承接长期协作价值的显性展示。

**核心目标**
 帮助用户看见：

- 最近自己做过什么
- 系统记住了什么
- 系统对自己有哪些理解
- 长期协作是否在产生价值

**结构设计**
 分为三个主区域：

**A. 历史对话记录概要**

- 展示重要协作历史摘要
- 支持跳转到本地路径查看完整记录

**B. 日报**

- 展示今日主要做过的事
- 展示完成内容、涉及对象、可能的权限与系统动作

**C. 用户画像**

- 展示偏好、习惯、常见任务类型、输出偏好、提醒接受度等

**设计原则**

1. 不做成普通聊天记录页
2. 强调“摘要、阶段总结、长期价值”
3. 强调系统对用户的理解是可见的
4. 允许用户查看但不要求频繁操作

#### 10.2.4 安全卫士

**模块定位**
 承接风险、恢复、授权、审计和成本治理。

**核心目标**
 帮助用户理解：

- 当前有没有待确认事项
- 最近有没有异常
- 执行影响到哪里
- 任务是否可恢复
- 当前和今日累计成本是多少

**结构设计**

**A. 状态总览**

- 当前安全状态
- 待确认操作数
- 最近异常
- 最近恢复点
- 当前任务 Token / 金额
- 今日累计 Token / 金额

**B. 待确认 / 被拦截操作**

- 操作名称
- 风险等级
- 目标对象
- 触发原因
- 操作按钮：允许本次、拒绝本次、查看详情

**C. 授权记录**

- 操作名称
- 风险等级
- 授权结果
- 是否记住规则
- 时间

**D. 审计记录**

- 文件操作记录
- 网页操作记录
- 命令操作记录
- 系统动作记录

**E. 影响范围**

- 影响文件
- 影响网页
- 影响应用
- 是否越出工作区
- 是否涉及覆盖 / 删除

**F. 恢复与回滚**

- 最近恢复点列表
- 恢复点时间
- 关联任务
- 涉及对象
- 操作按钮：恢复到该节点、查看影响范围

**G. 规则与权限管理**

- 绿灯动作
- 黄灯动作
- 红灯动作
- 工作区范围
- 命令执行权限
- 网页自动化权限
- 系统动作权限
- 定时截屏等配置

**H. Token 与费用**

- 当前任务输入 / 输出 / 累计 Token
- 当前任务金额
- 今日累计输入 / 输出 / 累计 Token
- 今日累计金额
- 上限配置
- 是否触发超限提醒 / 降级 / 暂停

**I. 执行日志**

- 查看执行记录
- 查看错误日志
- 查看 Token 日志
- 查看费用日志

**补充：成本与中断可见性**

安全卫士模块中，仪表盘应额外支持以下可见信息：

- 当前任务成本预算使用情况
- 是否接近单任务预算上限
- 今日累计 Token / 金额消耗
- 最近一次熔断或自动降级记录
- 当前任务是否可暂停 / 已暂停 / 可恢复
- 最近中断点与最近恢复点

**设计原则**

1. 先让用户“看懂”，再让用户“做决策”
2. 安全模块要偏系统化表达，不应娱乐化
3. 信息必须能支持回看与追责
4. 安全模块与任务状态模块要联动，但层级分明

#### 10.2.5 快捷键

**模块定位**
 承接固定快捷键说明，帮助高频用户形成效率使用路径。

**展示内容**

- 仪表盘内部固定快捷键
- 打开 / 关闭仪表盘
- 模块切换快捷键
- 常见操作快捷键
- 通话相关提示
- 悬停 / 轻量承接说明

**设计原则**

1. 不支持用户自定义
2. 说明应短、明确、可记忆
3. 优先呈现最常用动作
4. 是帮助入口，不是系统设置页

### 10.3 仪表盘首页信息架构建议

仪表盘首页建议按以下顺序组织：

**第一层：当前最重要信息**

- 当前任务状态摘要
- 最近待确认事项
- 今日最优先任务

**第二层：模块入口**

- 任务状态
- 任务巡检
- 镜子功能
- 安全卫士
- 快捷键

**第三层：辅助信息**

- 今日 Token / 费用摘要
- 最近恢复点
- 最新日报简版
- 最近一条任务建议

这样的组织方式可以确保用户打开仪表盘后：

1. 第一眼知道“现在最重要的是什么”
2. 第二步知道“要去哪个模块看”
3. 第三步知道“系统整体运行得怎么样”

### 10.4 仪表盘与悬浮球的关系

仪表盘与悬浮球不是两个独立产品，而是两层不同深度的承接。

> 仪表盘是悬浮球协作能力的完整展开态，不是另一个独立产品。

这一定义用于明确二者关系：

- 悬浮球不是仪表盘的简化版入口
- 仪表盘也不是脱离悬浮球体系的独立工作区
- 二者属于同一套桌面协作系统的不同承接层级

**悬浮球负责**

- 快速接近
- 轻量输入
- 通话发起
- 当前任务现场承接
- 短结果返回

**仪表盘负责**

- 全局状态查看
- 长任务持续承接
- 任务推进管理
- 长期记忆查看
- 风险治理与恢复

用户的认知应始终是：

> 悬浮球是“顺手开始”，仪表盘是“完整看清”。

### 10.5 仪表盘设计原则总结

1. **不是默认主入口**
2. **不是重工作区平台**
3. **是完整但低频的工作台**
4. **要让状态、任务、记忆、治理信息都能被一眼理解**
5. **必须服务于任务推进和信任建立，而不是服务于信息堆叠**
6. **与悬浮球分工明确：前者轻，后者全**

### 10.6 验收标准

**任务状态**

- 用户能快速理解当前任务进行到哪一步
- 能看到任务安全摘要
- 能打开产出物

**任务巡检**

- 用户能区分近期事项、后续事项、重复事项和已结束事项
- 能快速执行完成 / 跳过 / 编辑 / 打开资料等动作

**镜子功能**

- 用户能看见历史概要、日报、画像
- 用户不会把镜子理解成普通聊天记录列表

**安全卫士**

- 用户能理解当前风险、待确认、恢复、日志和成本
- 用户能找到恢复入口和授权入口

**仪表盘整体**

- 用户能理解它是工作台，而不是重平台或聊天窗口替代
- 用户能在打开后 10 秒内定位到当前最关心的信息

## 11. 操作面板

### 11.1 模块定位

操作面板是 CialloClaw 的**系统设置与偏好配置中心**。它不承担任务承接、结果查看和协作推进职责，而是承接用户对产品行为、外观、记忆、巡检、数据与日志等系统级设置。

操作面板属于**隐形功能**，默认不通过悬浮球或仪表盘主路径暴露，而是通过：

> **桌面托盘图标右键 → 打开操作面板**

进入。

这样设计的原因是：

1. 操作面板属于低频设置入口，不应干扰主协作路径
2. 设置项不应与任务状态、记忆查看、安全治理混在同一个工作台中
3. 用户在日常使用中不需要频繁进入设置，但在需要时必须能稳定找到
4. 通过托盘右键进入，符合桌面常驻产品对“系统级设置入口”的用户预期

### 11.2 与仪表盘的分工关系

操作面板与仪表盘必须严格分工：

#### 仪表盘负责

- 当前任务状态
- 任务巡检
- 镜子记忆查看
- 安全卫士查看
- 快捷键帮助
- 当前协作和治理信息的查看与承接

#### 操作面板负责

- 全局设置
- 默认行为配置
- 外观偏好
- 桌面入口行为
- 记忆策略配置
- 巡检规则配置
- 成本与日志设置
- 关于与反馈

用户心智应始终保持：

> **仪表盘是“工作台”，操作面板是“设置中心”。**

### 11.3 入口方式

操作面板的主入口为：

- **桌面托盘右键**
- 在托盘菜单中点击“操作面板”或“设置”

不建议在 MVP 阶段通过以下路径作为主入口：

- 悬浮球直接打开
- 仪表盘首页高频入口
- 气泡中的快捷跳转

原因是操作面板属于低频管理入口，若过于前置，会让产品心智从“协作型 Agent”偏向“设置型桌面工具”。

### 11.4 页面目标

操作面板的页面目标主要有四个：

1. **让用户能配置系统级行为**
2. **让用户能调整默认体验偏好**
3. **让用户能查看和管理数据、日志、成本相关设置**
4. **让用户能在不进入主工作流的情况下完成必要维护**

### 11.5 信息架构

操作面板建议包含以下一级分组：

1. 通用设置
2. 外观与桌面入口
3. 记忆

5. 数据与日志

3. 设置项分类清楚，不会出现明显混淆
4. 常用低频设置可以在 10 秒内找到
5. 删除数据、恢复默认等高风险操作有明确确认
6. 语言、外观、记忆、巡检、成本、关于等核心配置都能被用户理解其作用

## 12. 结果交付策略

### 12.1 设计目标

结果交付策略解决的核心问题，不是“系统能不能生成结果”，而是“结果应该以什么形式、在什么位置、以什么节奏交给用户，才能真正被消费和继续使用”。

CialloClaw 不采用“所有结果都塞进聊天窗口”的方式，而是根据结果长度、结构、可编辑性、是否需要继续推进、是否需要长期保留，做分层交付。

### 12.2 结果交付分层模型

CialloClaw 的结果交付分为两层：

#### 第一层：轻量交付层

用于承接即时反馈、短结果和前置说明。主要载体是：

- 气泡
- 状态文案
- 轻量建议动作

#### 第二层：正式交付层

用于承接可继续使用、可继续编辑、可回看和可追踪的结果。主要载体是：

- 工作区文档
- 浏览器 / 结果页
- 生成文件
- 文件夹定位
- 任务详情 / 历史任务

### 12.3 轻量交付

#### 12.3.1 适用场景

轻量交付适用于以下结果：

- 一句解释
- 一小段翻译
- 三点摘要
- 简短改写
- 明确结论
- 下一步建议
- 状态反馈
- 意图确认

#### 12.3.2 交付载体

轻量交付的默认载体为气泡。

气泡中应优先包含：

1. 当前最重要的结论
2. 一小段必要说明
3. 一个主动作和少量次动作

#### 12.3.3 设计要求

1. 气泡必须在 3 秒内让用户看懂重点
2. 结果不宜超过用户一屏阅读负担
3. 文案必须短、直接、任务导向
4. 若结果已不适合当前载体，系统必须自动分流，而不是强行塞入气泡

#### 12.3.4 边界

轻量交付不适用于：

- 长草稿
- 完整方案
- 结构化搜索结果集合
- 多文件产物
- 可追溯执行任务
- 复杂审计信息

### 12.4 正式交付

正式交付适用于需要继续使用、继续编辑、继续查看或长期保留的结果。

#### 12.4.1 文档交付

适用于：

- 总结稿
- 回复草稿
- 说明文档
- 会议纪要草稿
- 方案草稿
- 可继续编辑的内容

#### 12.4.2 浏览器 / 结果页交付

适用于：

- 搜索结果
- 多条可浏览信息
- 可跳转的结构化结果
- 网页任务结果

#### 12.4.3 文件交付

适用于：

- 图片
- 表格
- 压缩包
- 导出文件
- 报告类文件
- 单一目标文件

#### 12.4.4 任务详情交付

适用于：

- 持续任务
- 失败后需回看原因的任务
- 有恢复点的任务
- 需要再次推进的任务
- 需要查看执行过程和审计信息的任务

### 12.5 自动分流规则

#### 12.5.1 优先留在气泡中的结果

- 少于 120 字的解释或结论
- 1 到 5 条短要点
- 当前任务的状态反馈
- 简单的意图确认
- 下一步建议

#### 12.5.2 自动进入文档的结果

- 需要完整阅读的长文本
- 草稿类结果
- 需要继续修改的内容
- 层次清晰、适合编辑的输出

#### 12.5.3 自动进入结果页的结果

- 多条信息集合
- 搜索类输出
- 结果之间存在跳转关系
- 用户可能继续浏览的内容

#### 12.5.4 自动进入任务详情的结果

- 连续任务
- 有状态变化的任务
- 含风险 / 恢复 / 审计信息的任务
- 用户后续还会继续处理的任务

#### 12.5.5 自动进入文件交付的结果

- 文件产物型输出
- 需要明确保存位置的输出
- 需要拖拽、上传、发送的输出

### 12.5 补充：任务与自动化结果的交付规则

对于自动化、巡检和周期任务的结果，不应简单等同于普通问答结果，需按以下方式交付：

1. **例行提醒类结果**
   默认以悬浮球提示、托盘提示或气泡短结果交付。
2. **周期汇总类结果**
   默认进入文档或仪表盘摘要模块，例如日报、周报、今日任务汇总。
3. **执行记录类结果**
   默认进入任务详情或安全卫士模块，以便查看影响范围、授权与日志。
4. **文件类产物结果**
   默认写入 Workspace，并在气泡中说明保存位置。

### 12.6 结果交付时的用户感知要求

无论交付形式如何，用户都必须清楚感知到三件事：

1. **系统已经完成了什么**
2. **结果现在在哪里**
3. **下一步我可以做什么**

### 12.7 失败与中断时的交付策略

当任务失败或中断时，系统不能只给“失败了”这样的模糊结果，而必须交付以下信息：

1. 当前失败在哪一步
2. 失败原因是否明确
3. 是否有可恢复点
4. 是否可重试
5. 是否需要用户授权 / 修改输入 / 更换方式

### 12.8 验收标准

1. 用户能清楚理解结果被交付到了哪里
2. 不同类型结果能自动分流到合理位置
3. 气泡不会承担超出其能力范围的长结果
4. 长结果、结构化结果和可追踪任务均能获得对应正式交付
5. 用户拿到结果后能自然继续下一步动作，而不是重新寻找入口

## 13. 关键状态与交互规则

### 13.1 设计目标

关键状态与交互规则的目的，是保证 CialloClaw 在长期桌面常驻中行为稳定、可预期、易理解。

### 13.2 悬浮球状态

#### 13.2.1 待机形态

表示系统当前处于默认可用状态，没有正在处理中的任务，也没有必须提醒用户的事项。

特征：

- 低存在感
- 不主动打扰
- 随时可接近
- 在被轻度接近时可给出短暂、克制的反馈

#### 13.2.2 可唤起形态

表示系统当前可被轻量承接。

#### 13.2.3 承接中形态

表示系统已经开始接收输入，但尚未进入处理阶段。

#### 13.2.4 意图确认中形态

表示系统已识别任务对象，但仍需确认用户想做什么。

#### 13.2.5 处理中形态

表示系统已进入解析、理解、生成或任务推进阶段。

#### 13.2.6 等待确认形态

表示系统当前任务不能继续推进，正在等待用户做决策。

#### 13.2.7 完成形态

表示系统已完成当前任务，结果已可消费。

#### 13.2.8 异常形态

表示任务执行失败、上下文无法识别、能力当前不可用或流程被中断。

#### 13.2.9 暂停形态

表示系统当前被用户主动暂停，不再继续承担主动协助、巡检提醒与新的自动执行任务。

特征：

- 状态表达清晰
- 保持可恢复入口
- 不继续主动推荐
- 不继续发起新动作
- 用户可随时恢复

### 13.3 承接状态

- 悬停承接
- 文本选中承接
- 文字拖拽承接
- 文件拖拽承接
- 语音承接
- 推荐承接
- 结果承接

### 13.4 状态组合原则

1. 同一时刻只能有一个主状态清晰可感知
2. 承接状态只做补充说明，不应盖过主状态
3. 状态切换必须与用户动作或系统阶段对应
4. 从强状态回落到弱状态应自然，不频闪

### 13.5 通用交互优先级规则

#### 13.5.1 单击优先级

- 默认解释为轻量接近或当前任务承接
- 不默认解释为打开主页面

#### 13.5.2 双击优先级

- 始终解释为打开仪表盘
- 不与当前轻量输入冲突

#### 13.5.3 长按优先级

- 一旦进入长按阈值，即优先解释为语音
- 不再解释为单击 / 双击 / 普通拖动

#### 13.5.4 拖拽优先级

- 未满足长按阈值时，悬浮球位移可解释为移动
- 文件或文本对象拖入悬浮球时，优先解释为任务对象输入

#### 13.5.5 异常回退规则

- 任何手势识别失败时，应回退到最近稳定状态
- 不允许用异常弹窗打断用户主流程

### 13.6 主动推荐规则

#### 13.6.1 触发条件

主动推荐必须同时满足以下条件：

1. 当前处于非通话模式
2. 用户鼠标悬停在悬浮球上
3. 用户近期处于活跃状态
4. 当前桌面内容具备可感知信息
5. 距离上次推荐已超过冷却时间

#### 13.6.2 不触发条件

以下情况必须禁止主动推荐：

1. 用户长期静止无操作
2. 当前桌面内容长时间无变化
3. 用户已明显离开设备
4. 刚刚已经触发过相同推荐
5. 当前已经在一个高占用任务处理中

#### 13.6.3 呈现规则

- 默认最多展示 3 个推荐问题
- 推荐语句必须短、明确、可点击
- 不应在推荐中出现长解释
- 推荐不应遮挡主要工作区域

#### 13.6.4 自适应规则

随着长期使用，主动推荐需要根据镜子记忆中的用户反馈做调节。

### 13.7 通话规则

#### 13.7.1 唤起规则

- 长按悬浮球进入收音态
- 状态必须明确

#### 13.7.2 持续表达规则

- 按住为短时表达
- 上滑后进入持续表达
- 系统应允许用户在持续表达中补充背景和修正需求

#### 13.7.3 取消规则

- 下滑取消后不进入后续处理链路
- 本次输入不应被误记为正式需求

#### 13.7.4 打断规则

- 系统正在回应时，用户仍可再次通过语音打断和补充
- 打断后应合并为同一任务语义链，而不是新建完全独立任务

### 13.8 气泡规则

#### 13.8.1 数量规则

- 同时最多保留 3 个气泡
- 超出后优先淘汰最早且未置顶气泡

#### 13.8.2 生命周期规则

- 显现
- 透明化
- 隐藏
- 消散
- 置顶气泡不自动消散

#### 13.8.3 操作规则

- 支持置顶
- 支持删除
- 支持轻量动作按钮
- 支持隐藏后恢复显示

#### 13.8.4 分流规则

- 当内容超出气泡承载范围时，必须自动分流
- 气泡负责“告知”和“短结果”，不负责所有正式交付

### 13.9 仪表盘规则

#### 13.9.1 打开规则

- 双击悬浮球打开
- 打开后不应抢占全部注意力
- 默认展示总览信息，而不是直接深入某个复杂流程
- 打开仪表盘后，悬浮球继续保留为系统锚点，但退出主交互角色，进入静默挂起态
- 仪表盘打开期间，悬浮球不继续承担普通轻承接入口职责

#### 13.9.2 模块规则

- 五个模块职责必须稳定
- 同一信息不得在多个模块中高冗余重复展示
- 摘要信息与详细信息分层呈现

#### 13.9.3 返回规则

- 用户离开仪表盘后，应自然回到悬浮球主体系
- 仪表盘不是“进入后很难退出”的重层

### 13.10 验收标准

1. 用户能理解并区分主要状态
2. 交互优先级一致，不会产生明显冲突
3. 主动推荐不会在错误时机打扰用户
4. 语音、悬停、拖拽、选中都能进入统一承接逻辑
5. 用户能形成稳定的产品心智

## 14. MVP 范围

### 14.1 本期目标

MVP 阶段的目标不是一次性做完整桌面 Agent，而是先验证以下四件事：

1. 用户是否愿意接受一个长期常驻但低打扰的桌面入口
2. 用户是否能通过通话、悬停、拖拽、选中等轻动作自然发起协作
3. 仪表盘是否能承担低频完整工作台角色
4. 用户是否会因为可见的记忆和治理能力而更愿意长期使用

### 14.2 必做

#### 14.2.1 桌面入口相关

1. 悬浮球常驻
2. 左键单击轻量接近 / 当前任务承接
3. 左键双击打开仪表盘
4. 左键长按唤起通话
5. 上滑锁定 / 下滑取消
6. 鼠标悬停显示轻量输入框与推荐问题
7. 文件拖拽到悬浮球
8. 文本选中作为后续协作上下文

#### 14.2.2 场景助手基础能力

1. 总结
2. 翻译
3. 解释
4. 提炼重点
5. 错误解释与简单建议
6. 简单下一步建议

#### 14.2.3 仪表盘基础结构

1. 任务状态模块
2. 任务巡检模块
3. 镜子模块
4. 安全卫士模块
5. 快捷键模块

#### 14.2.4 镜子基础能力

1. 历史对话概要
2. 日报
3. 用户画像基础展示

#### 14.2.5 安全卫士基础能力

1. 风险状态
2. 待确认项数量
3. 最近恢复点
4. Token / 金额总览
5. 基础日志入口

#### 14.2.6 操作面板基础能力

1. 托盘右键打开操作面板
2. 通用设置基础项
3. 外观与桌面入口基础项
4. 镜子记忆启用与生命周期设置
5. 巡检规则基础项
6. 数据与日志基础设置
7. 关于与检查更新入口

#### 14.2.7 平台与治理基础能力

1. 多模型配置基础能力
2. 插件 / 感知包基础框架
3. cron / heartbeat 基础定时能力
4. 屏幕感知基础版：
   - 文本选中
   - 复制感知
   - 页面异常停留识别
5. 硬件感知基础版：
   - CPU
   - 内存
   - 磁盘
   - 网络
   - 电池语义事件
6. 一键暂停 / 中断 Agent
7. Workspace 默认路径与安全写入边界
8. Token / 金额基础展示与预算提醒

### 14.3 暂不纳入

#### 14.3.1 执行闭环类

1. 真实高风险执行闭环
2. 完整审批链路
3. 自动回滚技术实现
4. 企业级安全隔离能力

#### 14.3.2 长期能力类

1. 完整长期记忆压缩与召回
2. 复杂用户画像建模
3. 高级提醒频率自适应
4. 团队共享记忆

#### 14.3.3 工作区与产物类

1. 完整工作区文件写入体系
2. 文件导出与版本管理
3. 大规模结果页体系
4. 多任务并行工作台

#### 14.3.4 协同与平台类

1. 团队协作
2. 多人任务同步
3. workflow 编排平台能力
4. 外部系统深度联动

### 14.4 本期为什么不做

MVP 阶段不做这些内容，原因不是它们不重要，而是因为当前阶段需要先验证以下优先级：

1. **入口成立**：用户愿不愿意让它常驻
2. **发起成立**：用户能不能用轻动作顺手开始
3. **承接成立**：气泡和仪表盘是否够用
4. **信任成立**：用户是否能理解边界与治理入口

### 14.5 本期范围边界判断标准

#### 纳入标准

- 能降低用户发起协作的门槛
- 能提升任务现场理解效率
- 能帮助建立产品使用习惯
- 能提升“可控、可信”的感知
- 不显著增加产品复杂度

#### 不纳入标准

- 需要大规模技术闭环才能成立
- 会明显把产品拉向重平台
- 会让用户认知负担大幅增加
- 不利于先验证核心体验

### 14.6 后续迭代方向

#### 阶段 1：前台协作体验成立

- 悬浮球、通话、悬停、拖拽、选中、气泡、仪表盘基础闭环成立

#### 阶段 2：结果与状态承接增强

- 更丰富的文档交付
- 更完整的任务详情
- 更细粒度的状态和推荐策略

#### 阶段 3：长期协作增强

- 更强的镜子记忆
- 更强的日报与用户画像
- 更准确的推荐与偏好适配

#### 阶段 4：可控执行增强

- 工作区真实写入
- 高风险确认流
- 恢复与回滚
- 成本治理与自动降级

### 14.7 MVP 验收标准

1. 用户能在首次体验中理解悬浮球、通话、悬停三种主要入口
2. 用户能通过文件拖拽和文本选中完成任务发起
3. 场景助手可围绕真实任务对象提供可用结果
4. 仪表盘五个模块的职责清晰，用户能快速定位信息
5. 镜子和安全卫士虽为基础版，但用户能感知到“长期协作”和“治理边界”的存在
6. 操作面板作为系统设置中心，入口稳定、分类清楚
7. 用户主观上认为产品“基本不打扰，但确实有帮助”

## 15. 成功指标

### 15.1 指标设计原则

CialloClaw 的成功不能只看“用了多少次”，还要看：

1. 用户是否形成了新习惯
2. 系统是否真的进入了任务现场
3. 轻量承接是否减少了表达和切换成本
4. 用户是否更愿意相信系统、继续使用系统

因此，指标要分为四层：

- 入口指标
- 使用指标
- 体验指标
- 信任指标

### 15.2 入口指标

#### 15.2.1 悬浮球使用率

定义：活跃用户中，有多少人实际与悬浮球发生交互。

#### 15.2.2 单击触发率

定义：悬浮球曝光后，被单击的比例。

#### 15.2.3 双击打开仪表盘率

定义：悬浮球交互中，双击打开仪表盘的比例。

#### 15.2.4 长按通话率

定义：活跃用户中，发起长按通话的比例。

#### 15.2.5 悬停推荐点击率

定义：推荐问题展示后，被点击的比例。

#### 15.2.6 文件拖拽发起率

定义：有文件协作需求的用户中，通过拖拽发起的比例。

#### 15.2.7 文本选中后继续协作率

定义：文本选中后，用户继续通过点击 / 语音 / 拖拽围绕其发起协作的比例。

#### 15.2.8 托盘右键打开操作面板率

定义：活跃用户中，通过托盘右键进入操作面板的比例。

### 15.3 产品指标

用于衡量“系统是否真的帮用户完成任务”。

#### 15.3.1 任务发起率

定义：活跃用户中，实际进入一轮协作任务的人数占比。

#### 15.3.2 任务完成率

定义：已发起任务中，用户认为结果可用并完成本轮协作的比例。

#### 15.3.3 任务推进率

定义：获得结果后，用户继续执行下一步动作的比例。

### 15.4 体验指标

#### 15.4.1 上下文补充减少感

定义：用户主观认为“少解释了很多”的比例。

#### 15.4.2 打扰感评分

定义：用户对悬浮球、气泡、主动推荐总体打扰程度的评分。

#### 15.4.3 可理解性评分

定义：用户是否理解悬浮球、气泡、仪表盘、操作面板各自作用。

### 15.5 信任指标

#### 15.5.1 风险可控感

定义：用户是否认为高风险动作边界清晰、确认合理、恢复可用。

#### 15.5.2 长期使用意愿

定义：用户是否愿意将产品常驻保留并持续使用。

### 15.6 长期指标

- 周留存
- 主动使用频率
- 镜子记忆回访率
- 仪表盘回看率

### 15.7 平台与信任指标补充

#### 15.7.1 自动化指标

- 定时任务创建率
- 周期巡检触发率
- 周期任务有效完成率

#### 15.7.2 主动协助指标

- 主动提示接受率
- 主动提示忽略率
- 主动提示后继续协作率

#### 15.7.3 安全与信任指标

- 高风险操作确认通过率
- 高风险操作拒绝率
- 中断后恢复率
- 回滚触发率
- 越界写入拦截率

#### 15.7.4 成本治理指标

- 单任务平均 Token 消耗
- 单任务平均成本
- 成本熔断触发率
- 自动降级触发率

#### 15.7.5 内容理解扩展指标

- 视频总结使用率
- 视频总结后二次编辑率
- 屏幕感知触发后使用率

### 15.8 阶段性目标判断方式

MVP 阶段可按三类目标判断：

#### 第一类：成立性目标

- 是否被理解
- 是否能稳定触发
- 是否有正向反馈

#### 第二类：理解性目标

- 用户是否理解入口
- 是否理解仪表盘
- 是否理解气泡与正式交付分工
- 是否理解操作面板是设置中心

#### 第三类：信任性目标

- 用户是否愿意继续使用
- 是否认为系统不太打扰
- 是否认为系统边界清晰

### 15.9 验收标准

1. 入口指标能证明悬浮球、悬停、通话、拖拽、选中确实被使用
2. 产品指标能证明任务不仅被发起，也能被完成
3. 体验指标能证明系统没有明显打扰且易于理解
4. 信任指标能证明用户对风险、恢复和成本有基本感知
5. 长期指标能证明 CialloClaw 有机会从“试一次”变成“持续用”

## 16. 产品边界

### 16.1 设计目标

产品边界的作用，不只是“列出不做什么”，而是帮助团队在产品演进过程中始终保持方向稳定。

### 16.2 不做什么

#### 16.2.1 不做命令行驱动的 Agent 引擎

CialloClaw 不以命令行、脚本编排或 Agent 调度台作为主入口。

#### 16.2.2 不做重工作区平台

CialloClaw 不应演化为一个要求用户长期停留、在其中管理所有任务和流程的重平台。

#### 16.2.3 不做高侵入自动化系统

CialloClaw 不以“默认接管”“强提醒”“未经确认直接执行”为卖点。

#### 16.2.4 不做纯聊天产品

CialloClaw 的主价值不是聊天窗口中的对话体验，而是围绕桌面任务现场进行承接、推进与治理。

### 16.3 要做什么

CialloClaw 要做的是：

- 成为桌面真实任务现场中的协作者
- 让用户以更低门槛发起协作
- 让系统围绕上下文理解并推进任务
- 让长期记忆和可控执行逐步成立
- 让风险等级、影响范围、日志和恢复点都可见

因此必须做到：

- 风险等级明确
- 为什么要确认说得清楚
- 影响范围可见
- 日志可查
- 恢复点可用
- 成本透明

### 16.4 边界判断原则

未来遇到新需求时，团队可用以下 5 个问题来判断是否应纳入 CialloClaw：

#### 16.4.1 它是否服务于“桌面任务现场承接”？

如果一个需求必须把用户带离当前工作流，进入一个重系统才能体现价值，那它大概率不属于当前主线。

#### 16.4.2 它是否降低了协作发起成本？

如果一个需求让用户必须学更多概念、点更多层级、输入更多说明，它大概率偏离了核心方向。

#### 16.4.3 它是否帮助任务继续推进？

如果一个需求只是增加展示、增加趣味、增加复杂度，而不帮助理解、提醒、推进或治理，优先级应降低。

#### 16.4.4 它是否强化了长期协作关系？

如果一个需求让系统更懂用户、更会承接相似任务，它更值得纳入；反之，如果只是一次性炫技能力，应谨慎评估。

#### 16.4.5 它是否破坏了“低打扰、可控”的产品原则？

任何会显著提高打扰感、侵入感、不可控感的需求，都应被严格约束。

### 16.5 当前阶段的边界结论

在当前 PRD 阶段，CialloClaw 的边界可以总结为：

> CialloClaw 是一个以悬浮球为入口、以任务现场承接为核心、以仪表盘为完整查看与治理中枢、以长期记忆和可控执行为延展方向的桌面协作 Agent。

它不追求成为最强自动化系统，也不追求成为最复杂工作台，而是追求成为最自然进入用户桌面工作流的协作者。
